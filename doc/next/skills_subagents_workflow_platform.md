## TL;DR

Генераторы промптов — почти идеальные кандидаты на Claude Code skills, потому что
skill — это ровно то, чем генератор уже является: триггер-описание + процедурная
инструкция + опциональный скрипт. Но сильный ход — **не** «конвертировать все 8
генераторов в 8 skills», а сделать **2–4 тонких skill-адаптера поверх уже
существующих скриптов**. Промпты ролей (`developer.md`, `tester.md`, …) в skills
**не конвертировать** — их будущее в субагентах (`.claude/agents/`), это отдельный
пакет (см. § Роли → субагенты).

## Контекст и проблема

Проект уже имеет **три представления** одной workflow-логики:

1. **md-документы** (`doc/team_workflow/generate_*.md`, `workflow_router.md`) —
   человекочитаемый закон процесса;
2. **Python-скрипты** (`scripts/workflow.py`, `scripts/generate_orchestration_prompt.py`) —
   детерминированный движок (Phases 1–5 оркестрации уже автоматизированы);
3. **`scripts/workflow_strings.py`** — SSoT строк промпта агента для вывода роутера.

Самое медленное и ошибкоопасное звено конвейера сегодня — **copy-paste-петля**:
человек запускает скрипт, копирует «Промпт для агента» из консоли, вставляет в чат
агента. Skills замыкают её в один слэш.

Проект дважды прошёл войну за SSoT (`tasklist.md` → derived; `workflow_strings.py` →
единый источник строк). Skills не должны стать третьим фронтом этой войны.

## Принцип тонкого адаптера (ядро решения)

> **Документ — закон, скрипт — движок, skill — руль.**

Целевая слоистая архитектура:

```
Claude Code (человек)          Cursor / Codex (агенты)
        │                              │
        v                              │ напрямую, без skills
.claude/skills — тонкие адаптеры       │
  /workflow · /resume-package          │
  /orchestrate · /audit-chain          │
        │ вызывают                     │
        v                              v
scripts/ — детерминированный движок
  workflow.py · generate_orchestration_prompt.py · workflow_strings.py
        │ читают как закон
        v
doc/team_workflow/ + backlog_registry.yaml — SSoT
  генераторы промптов · промпты ролей · process.md
```

Жёсткие правила слоя skills:

- тело skill'а ≤ ~50 строк; никакой скопированной логики из генераторов —
  только «запусти скрипт X, следуй его выводу»;
- каждая стоп-точка процесса (review принятого контракта между plan-next и
  оркестрацией, WIP=1, RED-вердикт Ops Impact Gate) — **явный STOP** в теле skill'а;
- граница фазы пайплайна = граница сессии/субагента (инвариант token-бюджета 12k,
  см. `CLAUDE.md` § Context Policy);
- путь md+скрипты для Cursor/Codex сохраняется нетронутым (мультиагентность
  `cursor_ai` / `claude_code` / `codex` — не ломать).

## Ранжирование кандидатов

| Кандидат | Пригодность | Почему |
|---|---|---|
| `workflow_router.md` → `workflow.py` | Отличный, **делать первым** | Логика уже в коде; skill `/workflow` убирает copy-paste-петлю — главный ручной шаг каждого цикла |
| `generate_resume_prompt.md` | Отличный | Самый частый бытовой сценарий («продолжи пакет»); срабатывание по естественной фразе |
| `generate_orchestration_prompt.md` | Хороший, с оговоркой | Скрипт уже делает Phases 1–5; skill = обёртка + STEP 1. Оговорка: не проскакивать review контракта |
| Цепочка аудита (`run_audit_chain` + 2 генератора) | Хороший, вторая волна | Многошаговая, периодическая, состояние уже в `check_audit_chain_state.py`; кандидат на расписание |
| `generate_plan_next_prompt.md` | Средний | Фаза мышления, а не механики; ценность — гарантированный preflight `check_readset.py` |

## Фазовый план

1. **Фаза 1:** skill `/workflow` — тонкая обёртка над `scripts/workflow.py`,
   единственная точка входа (как и задумано в `process.md`).
2. **Фаза 2:** `/resume-package` и `/orchestrate` — обёртки над существующими
   скриптами со встроенными стоп-точками.
3. **Фаза 3:** `/audit-chain` — после обкатки первых двух; затем перевод триггера
   «после каждого 3-го закрытого пакета» с человеческой памяти на автоматику
   (hook в `close_package.py` или scheduled task).
4. **Отдельный пакет (не skills):** роли → субагенты — см. ниже.

Перед фазой 1: зафиксировать правило тонкого адаптера в
`doc/team_workflow/process.md` — **до** написания первого skill'а. Иначе через
три эпохи придётся аудировать расхождение skills с генераторами так же, как
сейчас аудируются закрытые пакеты.

## Роли → субагенты, не skills (раскрытие идеи)

Промпты ролей (`product_owner.md`, `analyst.md`, `architect.md`, `designer.md`,
`developer.md`, `tester.md`, ops-роли) в skills конвертировать **нельзя** —
это категориальная ошибка:

- **Skill — это процедура** («как сделать X»). **Субагент — это исполнитель**
  (идентичность + ограниченный toolset + свежий контекст). Промпт роли — ровно
  второе: identity, ответственность, формат handoff-артефакта.
- **Свежий контекст на роль** — субагент стартует с чистым окном. Это *архитектурно*
  решает token-бюджет 12k: граница роли = граница контекста, политика из CLAUDE.md
  становится свойством системы, а не дисциплиной.
- **Ограничение инструментов = enforcement процесса.** Tester-субагент с read-only
  toolset (без Edit/Write) *физически* не может «починить код, чтобы тесты прошли» —
  независимость приёмки перестаёт держаться на честном слове. Developer-субагент
  получает write-доступ только к write-set из контракта.
- **Ops Impact Gate (STEP 3.5) → параллельный fan-out:** RAGOps/LLMOps/MLOps/Performance
  субагенты запускаются параллельно по триггер-таблице из `process.md`, каждый
  возвращает Impact Report; оркестратор агрегирует вердикты GREEN/YELLOW/RED.
  Сейчас это сериализовано или пропускается.
- **Главная сессия = оркестратор:** держит WIP=1, стоп-точки, эскалации
  (Designer→Architect, Tester→PO и т.д. из `process.md` § Эскалация).
- **SSoT не дублируется:** определения `.claude/agents/*` — тонкий frontmatter
  (name, description, tools) + ссылка/включение текста роли из
  `doc/team_workflow/<role>.md`. md-промпт роли остаётся законом и продолжает
  работать для Cursor/Codex.

Это отдельный пакет со своим контрактом — больший риск и больший выигрыш, чем
skills-обёртки; делать после обкатки Фазы 1–2.

## Другие сильные идеи (вторичные, фиксируем)

- **Hooks как механический enforcement token-safety:** PreToolUse-hook, который
  прогоняет `scripts/check_readset.py` перед чтением файлов из Forbidden-списка
  CLAUDE.md — политика из декларативной становится исполняемой.
- **Scheduled audit:** автотриггер аудит-цепочки после каждого 3-го закрытого
  пакета (счётчик в `close_package.py` → уведомление или scheduled task).
- **Eval skills перед раскаткой:** skill-creator умеет evals; минимальный eval
  на триггеринг (`/workflow` срабатывает на «что дальше по пакету?») до коммита.
- **Метрики:** колонка skill-инвокаций в `archive/pipeline_metrics.md` —
  измерить, что copy-paste-петля реально исчезла (время цикла до/после).

## Риски и анти-цели

| Риск | Митигация |
|---|---|
| Дрейф SSoT (4-е представление логики) | Правило тонкого адаптера: ≤50 строк, только вызовы; фиксация в `process.md` до первого skill'а |
| Вендор-лок на Claude Code | Путь md+скрипты для `cursor_ai`/`codex` остаётся нетронутым; skills — опциональный слой |
| Эрозия человеческих гейтов | Каждая стоп-точка — явный STOP в теле skill'а; `/orchestrate` не стартует без review контракта |
| Пробой token-бюджета в одной сессии | Инвариант: граница фазы = граница сессии/субагента; никаких skill'ов «весь STEP 1–8 разом» |
| Стоимость сопровождения | Мало и тонко: 2–4 skill'а, не 8; eval через skill-creator |

**Анти-цели:** не конвертировать промпты ролей в skills; не переносить логику
генераторов в тела skills; не закрывать мультиагентный путь.
