## Волна `wave-konspekt-passport` (P0)

### A1 «Прочитать рубрику» — роль quality_rubric + парсер + паспорт вместо галочки

- **Problem.** Качество конспекта на поверхности продукта = существование файла:
  «✅ Все конспекты готовы» при 100% покрытия. При этом каждый cloud-конспект уже несёт
  «## ✅ Рубрика качества конспекта» — таблицу 10 критериев × оценка 1–5 × комментарий
  (у урока_2 включая честное «Проверка точности 4/5»). Продукт её не читает нигде.
- **Evidence.** `grep -r "Рубрика качества" app/` → 0 (HEAD 208);
  `app/section_index.py:322-356` — в `_ROLE_BY_NORMALIZED_HEADING` 7 ролей, рубрики нет;
  `app/konspekt_discovery.py:57` — конспект = frontmatter `type: konspekt`;
  `:95-98` — `coverage_summary` считает covered/total;
  `app/ui/topics_tab.py:44,50-58` — галочка/счётчик покрытия;
  требование рубрики живёт в другом репо:
  `hometutor-studio/scripts/validate_smart_konspekt.py:124` (обязательный раздел),
  `:100-101` (self_eval в YAML запрещён — «оценки только в рубрике»).
  В живой базе рубрику несут 7 файлов двух курсов.
- **Proposed.**
  1. Роль `quality_rubric` в `_ROLE_BY_NORMALIZED_HEADING`
     («рубрика качества конспекта», «рубрика качества», «quality rubric») — честная
     деградация None сохраняется (фабрики A/B рубрику не пишут).
  2. Детерминированный парсер markdown-таблицы рубрики в `app/konspekt_discovery.py`
     (рядом с `KonspektMeta`): `[(критерий, оценка, макс, комментарий)]` + средняя.
     Без LLM; кривая таблица → None, не исключение.
  3. Поверхности: (а) `topics_tab.py` — вместо «✅ Все конспекты готовы» строка-паспорт
     «N/M с конспектом · рубрика в K · средняя X/5»; (б) `topics_tab_right_column.py` —
     у документа рядом с уже существующим ✅-линком Obsidian (`:410-421`) подпись
     «рубрика X/5»; (в) читалка Живого конспекта — в caption источника
     (`living_konspekt_reader.py:90`) тот же бейдж. Раскрытие полной таблицы —
     expander по клику, без нового экрана.
- **Files.** `app/section_index.py`, `app/konspekt_discovery.py`,
  `app/ui/topics_tab.py`, `app/ui/topics_tab_right_column.py`,
  `app/ui/living_konspekt_reader.py`; тесты: новый узкий тест парсера рубрики
  (fixture-markdown с таблицей урока_2), расширение
  `tests/test_living_konspekt_add_panel_reader.py`.
- **DoD.** На живой базе урок_2 показывает «рубрика 4.9/5 · 10 критериев» на вкладке
  тем и в читалке; конспект без рубрики — честное «рубрики нет»; 0 LLM-вызовов;
  время рендера вкладки не деградирует (парсер за lru_cache по mtime, по образцу
  `_cached_parse_sections`).
- **Doc-sync.** `docs/user_guide.md` (вкладка тем + Живой конспект).
- **Effort.** Часы.

### A2 «Статус знания + мой вопрос» — тройка статусов и open_question в строке корзины

- **Problem.** Из 6 качеств Живого конспекта два — дыры: открытых вопросов студента
  структурно не существует (есть только чужие контрольные вопросы лекции), статус знания
  бинарный («Прочитано» с датой). «Сборка не закончена» на MC говорит о количестве,
  не о понимании.
- **Evidence.** `app/ui/living_konspekt_reader.py:114-138` — только «Моя мысль» +
  «Прочитано»; `app/ui/living_konspekt_state.py:194-227` — `update_section_fields`
  умеет `note`/`read_at`; `app/konspekt_artifact.py:130-131` — note уже выживает в
  артефакте как «### 💬 Моими словами»; `app/ui/mission_control.py:845-863` —
  карточка считает разделы/лекции/концепты.
- **Proposed.**
  1. Два новых поля строки корзины: `knowledge_status`
     (`understood` / `unsure` / `unclear` / None) и `open_question` (str) —
     через тот же `workbench_service.update_section_fields`, персистентность app_kv
     и выживание при пересборке наследуются бесплатно.
  2. Читалка: сегмент из трёх кнопок «Понял / Сомневаюсь / Не понял» (замена или
     соседство «Прочитано» — решить по месту; read_at сохраняется как дата последнего
     статуса) + поле «Мой вопрос» рядом с «Моя мысль».
  3. Артефакт: `open_question` рендерится блоком «### ❓ Мой открытый вопрос»;
     статус — маркер в caption раздела.
  4. MC-карточка: `build_living_konspekt_card_stats` дополняется
     «понято X из N · открытых вопросов Y» (чистая функция, тестируется отдельно).
- **Files.** `app/workbench_service.py`, `app/ui/living_konspekt_state.py`,
  `app/ui/living_konspekt_reader.py`, `app/konspekt_artifact.py`,
  `app/ui/mission_control.py`; тесты `tests/test_living_konspekt_workbench.py`,
  `tests/test_mission_control_living_konspekt_card.py`.
- **DoD.** Статус и вопрос переживают перезапуск приложения и «↩ Пересобрать»;
  карточка MC показывает понимание, а не только количество; старые корзины без новых
  полей работают без миграции (None-дефолты).
- **Doc-sync.** `docs/user_guide.md` (Живой конспект).
- **Effort.** ~День.
- **Dependencies.** нет (A1 и A2 независимы).

## Волна `wave-konspekt-liveliness` (P1)

### B1 «Вопрос → тьютор» — открытый вопрос становится дверью

- **Problem.** Открытый вопрос (A2) без выхода — ещё одна запись «в стол»; сомнение
  должно вести к действию (критерий студента «приводит к действию»).
- **Proposed.** Кнопка «Спросить тьютора» у открытого вопроса: навигация в tutor chat
  с префиллом вопроса + контекст раздела (документ, заголовок, строки — всё уже в row).
  После ответа — предложение «закрыть вопрос» (поле очищается, факт закрытия пишет
  exposure-след слоем из №1). Needs discovery: механика префилла ввода tutor chat.
- **Files.** `app/ui/living_konspekt_reader.py` (кнопка),
  `app/ui/tutor_chat*` (префилл — по месту), след — существующий слой №1.
- **Effort.** Дни. **Dependencies.** A2.

### B2 «Новизна раздела» — концепты × mastery

- **Problem.** «Оценка новизны знаний» — критерий студента с полным нулём в коде:
  конспект не знает, что читатель уже знает.
- **Evidence.** Строки корзины уже несут `concept` (`mission_control.py:848-852`);
  mastery по концептам уже считается для карты (`build_kg_payload`: mastery/decay).
- **Proposed.** Derived-бейдж на раздел/сборку: «нового для тебя ~N%» = доля концептов
  раздела с mastery ниже порога. Только представление, данные не трогаются.
  Needs discovery: дешёвый геттер mastery по списку концептов без сборки полного payload.
- **Effort.** Дни. **Dependencies.** A2 (место в UI), независим от B1.

### B3 «Шум-фильтр корзины» — служебные разделы не предлагаются как содержательные

- **Problem.** Рубрика качества, Flashcards, Quiz, План повторения — служебные разделы
  конспекта — сейчас равноправные «содержательные разделы»: предлагаются в корзину
  и уходят в RAG как контент.
- **Evidence.** `app/ui/living_konspekt_add_panel.py:28` — `_NOISE_HEADINGS` = только
  оглавления; `app/section_index.py:32` — то же в `_SKIP_HEADING_NORMALIZED`.
- **Proposed.** Расширить шумовой фильтр панели добавления ролями-сервисами
  (quality_rubric из A1 + «учебные артефакты для повторения»/Flashcards/Quiz/План
  повторения). RAG-индексацию НЕ трогать в этом ходе (влияние на retrieval — отдельное
  решение с замером, вне плана).
- **Files.** `app/ui/living_konspekt_add_panel.py`, тест
  `tests/test_living_konspekt_add_panel_reader.py`.
- **Effort.** Часы. **Dependencies.** A1 (роль).

## Волна `wave-konspekt-factories` (P2)

### C1 «Грейды фабрик» — словарь ролей как объявленный контракт

- **Problem.** Три планки качества молчаливы: студент не отличает базовый in-app
  конспект (main_idea/terms/summary) от богатого операторского/cloud.
- **Evidence.** Признание в `app/section_index.py:317-321`; фабрика A —
  `OBSIDIAN_EXPORT_COMPOSE_PROMPT` (`prompts/_impl.py:1341-1395`) без вопросов/примеров/
  рубрики.
- **Proposed.** Грейд = derived от фактического состава ролей файла (`sections_by_role`),
  не от frontmatter: «базовый / богатый / богатый + рубрика». Бейдж в тех же местах, что
  паспорт A1. Доращивание COMPOSE-промпта до богатой структуры — отдельное решение
  владельца (меняет длительность in-app генерации).
- **Effort.** Дни. **Dependencies.** A1.

### C2 «Судьба фабрики B» — прописать бездомный smart_konspekt

- **Problem.** `generate_smart_konspekt` — 0 вызывающих в продукте, промпт по умолчанию
  указывает на файл, которого в репо нет → FileNotFoundError в чистом чекауте; модуль
  числится кандидатом в мёртвые.
- **Evidence.** `app/config.py:151` (дефолт `doc/prompts/smart_lecture_konspekt_universal.md`),
  `app/prompts/_impl.py:1431-1436` (резолв от корня hometutor), файл существует только в
  `hometutor-studio/doc/prompts/`; `scripts/check_dead_modules.py:19,37`;
  `.env` без `OBSIDIAN_EXPORT_PROMPT_PATH`.
- **Proposed.** Решение владельца, два честных варианта: (а) промпт переезжает в репо
  hometutor + документированная CLI-дверь (`scripts/generate_konspekt.py` по образцу
  studio-версии) — фабрика становится жилой; (б) модуль переезжает в studio целиком —
  продукт перестаёт возить мёртвый вес. Полурешение «оставить как есть» — хуже обоих.
- **Effort.** Дни (любой вариант).

### C3 «Факты ↔ интерпретации» — принцип разметки

- **Problem.** Качество «разделение фактов и интерпретаций» существует только как
  зачаток (текст лекции vs «💬 Моими словами»).
- **Proposed.** Единый визуальный принцип во всех рендерах артефакта: текст лекции /
  «Проверка точности» (роль из A1-словаря — добавить `accuracy_check`) / мои слова /
  мой вопрос — различимы стилем. Только представление.
- **Effort.** Волна. **Dependencies.** A1, A2.

## Рекомендованный порядок

A1 → A2 → B3 (дёшев, сразу после A1) → B1 → B2 → C1 → C2 → C3.

**North star:** студент отвечает на вопрос «насколько хорош мой конспект и что я в нём
ещё не понял?» с одного экрана, без debug-тира.

**Метрики.** Видимых критериев качества: 0 → 10 (готовая рубрика). Статусов знания:
1 → 3. Открытые вопросы: поля нет → задано/закрыто на карточке MC. «Живость» корзины
(доля разделов с мыслью/статусом/вопросом): не измеряется → видна.
