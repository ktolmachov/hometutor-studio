# Kilo Budget System — Master Reference

**Цель:** Предотвратить overflow Kilo сессий на уровне репозитория.
Защита работает в двух режимах: **проактивный** (gate + simulator, без Kilo) и **ретроспективный** (v1 probe через реальные сессии).

---

## Архитектура

```
scripts/
  _kilo_guard.py              ← pure core (shared by ALL tools)
  kilo_proxy_relay.py         ← runtime relay  (Tier 2 — реальные сессии)
  kilo_budget_probe.py        ← v1 probe harness (Tier 2 — manual)
  kilo_budget_simulate.py     ← static simulator (Tier 1 — fast, offline)
  kilo_budget_gate.py         ← pre-commit gate  (Tier 1 — automatic)
  kilo_injection_calibrate.py ← fixture rebuilder (Tier 1 — offline)
  kilo_budget_daily.py        ← daily health check (Tier 1 — one command)

fixtures/
  kilo_injection_baseline.json   ← committed fixture used by gate (current source)
  kilo_injection_calibrated.json ← latest calibrator output (offline estimate)
  kilo_injection_captured.json   ← authoritative relay capture (target state; optional)

doc/
  kilo_budget_gate.md          ← gate reference
  kilo_proxy_relay.md         ← эксплуатация релея (стриминг, LM Studio, JSONL)
  kilo_budget_system.md        ← этот файл
  team_workflow/
    budget_health_prompt.md    ← daily-use prompt
```

### Single source of truth

Вся guard-логика живёт в `scripts/_kilo_guard.py`.
Relay, simulator, gate и daily check вызывают одну функцию `evaluate_guard()`.
Для identical payload drift между static verdict и runtime verdict структурно невозможен.
Но committed fixture, calibrated estimate и real relay capture — это разные источники baseline.

**Важная оговорка.** Single-source-of-truth гарантирует равенство verdict только для identical payload.
Сейчас committed fixture в репозитории имеет `fixture_kind=calibrated_estimate`, поэтому static verdict
является индикатором, а не заменой runtime probe. Authoritative baseline появляется после реального
capture в `fixtures/kilo_injection_captured.json`.

## Два уровня защиты

### Tier 1 — Static (offline, <300ms, автоматический)

| Инструмент | Когда запускается | Что ловит |
|---|---|---|
| `kilo_budget_gate.py` | pre-commit hook | Регрессию vs HEAD в launcher файлах |
| `kilo_budget_simulate.py` | вручную / CI | Verdict для любого launcher + message-level/section-level attribution |
| `kilo_injection_calibrate.py` | вручную | Пересчёт calibrated estimate из project files |
| `kilo_budget_daily.py` | ежедневно | Read-only health check по committed fixture; опционально calibrated estimate |

### Tier 2 — Runtime (с реальным Kilo, ручной)

| Инструмент | Когда запускается | Что ловит |
|---|---|---|
| `kilo_proxy_relay.py` | постоянно при работе в Kilo | Реальный payload каждого запроса |
| `kilo_budget_probe.py` | периодически (после изменений) | Точный hidden injection, обновляет fixture |

## Injection anatomy (оценочная картина)

| File | Purpose | Status |
|---|---|---|
| `fixtures/kilo_injection_baseline.json` | fixture used by pre-commit gate | present (`calibrated_estimate`) |
| `fixtures/kilo_injection_calibrated.json` | latest calibrator output | regenerated by calibrate script |
| `fixtures/kilo_injection_captured.json` | authoritative runtime capture | not yet created (target state) |

Свежая Kilo сессия инжектирует примерно:

| Источник | Chars | Тип |
|---|---|---|
| Kilo system prompt | ~1,200 | kilo_system (оценка) |
| CLAUDE.md | ~11,600 | claude_md (читается) |
| memory/feedback_... | ~1,100 | memory (читается) |
| memory/MEMORY.md | ~700 | memory (читается) |
| memory/project_flashcards... | ~2,500 | memory (читается) |
| memory/project_token_... | ~4,700 | memory (читается) |
| memory/reference_token_... | ~2,900 | memory (читается) |
| Tool schemas | ~1,600 | tools |
| `<available_skills>` block | ~18,000 | skills (оценка) |
| **TOTAL** | **~44,400** | |

С launcher'ом: total body = **55k–63k** на старте сессии.
Gap до warn (70k): **7k–15k** — это стоимость 2–5 сообщений в истории.

> **Вот почему сессии переполнялись**: synthetic fixture показывал 2.2k chars.
> Calibrated estimate показывает ~44k, но authoritative hidden injection подтверждается
> только через v1 probe + capture.

## Guard thresholds

```
warn_body_chars:            70,000  → level: warn
max_body_chars:             90,000  → level: soft_block
hard_block_body_chars:     110,000  → level: hard_block
max_messages:               15      → level: soft_block  (raised from 8!)
max_largest_message_chars:  24,000  → level: warn
max_tools:                  13      → level: warn
```

`MAX_MESSAGES` был 8 — слишком мало: injection сам занимает 8 system messages.
Поднят до 15 чтобы дать 4–5 turns истории сверх injection overhead.

Env-override: `KILO_RELAY_MAX_MESSAGES=N`, `KILO_RELAY_WARN_BODY_CHARS=N`, и т.д.

## Workflow

### Ежедневно (2 секунды, read-only)

```bash
python scripts/kilo_budget_daily.py
```

Это проверка по committed fixture, который использует реальный pre-commit gate.
Если нужен локальный estimate поверх текущих CLAUDE/memory sources:

```bash
python scripts/kilo_budget_daily.py --use-calibrated-estimate
```

### Перед коммитом (автоматически через pre-commit)

```bash
pip install pre-commit && pre-commit install  # один раз
# Дальше gate запускается сам на каждый commit
```

Или вручную:
```bash
python scripts/kilo_budget_gate.py --dry-run
```

### После изменений в launcher / workflow doc

```bash
# Проверить конкретный launcher с message-level attribution
python scripts/kilo_budget_simulate.py simulate \
  --launcher doc/team_workflow/generate_plan_next_prompt.md \
  --injection fixtures/kilo_injection_baseline.json \
  --attribute --section-attribute
```

### После изменений в CLAUDE.md / memory files

```bash
# Пересчитать локальную estimate-модель injection (memory content redacted by default)
python scripts/kilo_injection_calibrate.py

# Если нужно явно вшить raw memory (privacy-sensitive)
python scripts/kilo_injection_calibrate.py --embed-memory-content
```

### После Kilo update (раз в 1–2 недели)

```bash
# 1. Запустить v1 probe
python scripts/kilo_budget_probe.py prepare --port 8791 --guard-mode block
# 2. Выполнить 3 probe в свежих Kilo сессиях (orch/planning/autonomous)
# 3. Захватить новый captured fixture
python scripts/kilo_budget_simulate.py capture \
  --from-jsonl logs/kilo_relay.jsonl -o fixtures/kilo_injection_captured.json
# 4. Перезапустить daily check
python scripts/kilo_budget_daily.py
```

```

- `ok -> warn` — рабочее дерево хуже HEAD
- `+55,023` — на сколько вырос body

```

- Всё хорошо. Gap до warn: 70k − 62.8k = +7.2k.
- Добавить 7k символов в launcher → warn level.

## FAQ

**Q: Calibrated fixture точный?**
Injection оценивается ±10–20%. Тело сообщений из реальных файлов — точное.
`kilo_local_recall` и available_skills — оценки. Для точности: запусти v1 probe и `capture`.

**Q: Fixture обновляется автоматически?**
Нет, `kilo_budget_daily.py` по умолчанию read-only и считает по committed fixture.
`--use-calibrated-estimate` добавляет локальную estimate-модель без записи в файл.
`--write-fixture` нужно использовать только осознанно, когда действительно хочешь обновить
`fixtures/kilo_injection_baseline.json`. Authoritative runtime snapshot хранится отдельно
в `fixtures/kilo_injection_captured.json` после `capture`.

**Q: У меня `planning` постоянно `warn`. Что делать?**
Запусти attribution: `kilo_budget_simulate.py simulate --attribute --section-attribute`.
Сначала message-level покажет самый тяжёлый message, затем section-level покажет самый дорогой раздел
внутри launcher.

**Q: Kilo обновился, стоит ли обновить fixture?**
Да. Запусти v1 probe, затем `capture`. Если Kilo добавил новые tools или skills — fixture устареет.

**Q: Можно ли поднять пороги?**
Осторожно. Порог в 70k warn отражает реальный опыт — при большем контексте качество Kilo падает.
Если нужно работать с большим контекстом, лучше сократить injection (memory files, launcher brevity).
