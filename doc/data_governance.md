# Управление локальными данными (data governance)

## Область действия (local-first)

Проект **home-rag_v2** рассчитан на работу в первую очередь **на одной машине**: индекс,
состояние обучения и диагностические логи хранятся рядом с репозиторием. Этот документ
описывает, **что считается «локальными данными приложения»**, как их полностью стереть и
что при этом **не затрагивается**.

## Граница с облачным inference

- **Удаление скриптом затрагивает только локальные файлы** (см. ниже). Оно **не**
  отзывает API-ключи, не меняет `.env` и не удаляет учётные записи у провайдера LLM.
- Запросы к облачным моделям — это **отдельный поток**: провайдер может хранить логи по
  своим правилам; этот репозиторий на это не влияет.
- После полного локального сброса при следующем запросе снова пойдут обращения к
  настроенному endpoint, если ключи остались в `.env`.

## Полное удаление локальных рабочих данных

Скрипт: `scripts/delete_all_data.py`.

Обязательное подтверждение (защита от случайного запуска):

```text
.\.venv\Scripts\python.exe scripts\delete_all_data.py --confirm-token DELETE-ALL-LOCAL-HOME-RAG-DATA
```

Проверка без удаления:

```text
.\.venv\Scripts\python.exe scripts\delete_all_data.py --verify-only
```

Перед запуском остановите процессы **FastAPI**, **Streamlit** и **telegram-бота**,
использующие те же пути БД и Chroma.

### Что удаляется

По умолчанию (пути из `app/config.py` / `get_settings()`):

1. Каталог векторного индекса **Chroma** (`chroma_db/`).
2. Файлы реестра поколений индекса: `index_registry.json`, `index_registry.json.lock`,
   `index_meta.json`, а также состояние активного индекса внутри `chroma_db/`.
3. База пользовательского состояния **`user_state.db`** (путь задаётся настройкой
   `user_state_db`, по умолчанию `data/user_state.db`).
4. Каталог **`data/cache/`** (кеш course/workspace рядом с БД).
5. **`data/concept_graph.json`** и каталог **`data/graph_generations/`** (производные
   графовые артефакты).
6. Телеметрия и история в **`logs/`** (конкретные файлы из настроек): `metrics_store.jsonl`,
   `metrics_dashboard.db`, `feedback.jsonl`, `history.jsonl`, каталог **`logs/cost_logs/`**.
7. **`faq_memory.jsonl`** в корне проекта (legacy/FAQ-хранилище вне Chroma, если файл есть).

### Что не удаляется

- Исходные документы корпуса в **`data/`** (PDF, MD и т.д.), кроме явно перечисленных выше
  производных файлов.
- Код, конфигурация репозитория, `.env`.
- Произвольные другие файлы в `logs/` (например, служебные логи пайплайна), если они не
  совпадают с путями из настроек.

После успешного прохода скрипт выполняет **проверку полноты**: ни один из целевых путей не
должен остаться. Код проверки доступен для тестов как `verify_deletion_complete()`.

## Повторное использование

После удаления индекс и состояние нужно **восстановить заново** (переиндексация corpus,
новая сессия обучения). См. также `scripts/backup_index.py` для резервного копирования
индексных артефактов до сброса.
