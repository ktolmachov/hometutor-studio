\# Отчёт по интеграции локальной llama.cpp-модели в процесс разработки



\*\*Проект:\*\* HomeTutor / local coding workflow

\*\*Целевая модель:\*\* `qwen3-coder-next-q4ks`

\*\*Runtime:\*\* llama.cpp OpenAI-compatible server

\*\*Режим:\*\* controlled local patch executor

\*\*Дата:\*\* 19 июля 2026 года



\---



\## 1. Цель



Интегрировать локальную coding-модель в процесс разработки приложения так, чтобы она могла:



1\. Получать узкую задачу из `doc/current\_task.md`.

2\. Читать только разрешённый `read-set`.

3\. Формировать минимальный unified diff.

4\. Не выходить за authoritative `write-set`.

5\. Применять изменения через controlled patch layer.

6\. Запускать только разрешённые targeted tests.

7\. Создавать `execution\_contract.md` исключительно из фактических evidence.

8\. Работать без внешних cloud API.



Локальная модель должна использоваться не как универсальный автономный агент, а как \*\*patch executor with gates\*\*.

\## 4. Production runtime contract

```text

```



Старые значения:



```text

qwen/qwen3-coder-next

qwen3-coder-next

```



не должны использоваться в runtime-командах, trigger, registry и smoke-проверках.



\### Canonical launcher



```powershell

cd D:\\AI\\llama\_cpp\_server\_pack\_v1



pwsh `

&#x20; -NoLogo `

&#x20; -NoProfile `

&#x20; -ExecutionPolicy Bypass `

&#x20; -File .\\Start-LocalModel.ps1 `

```



Trigger не должен самостоятельно определять GGUF, quant, GPU layers, tensor split, mmap или process priority.

\## 5. Authoritative write-set



\### Проблема



Trigger не должен проверять patch только против `WRITE\_SET`, который вернула модель. Иначе модель теоретически может расширить собственные полномочия.



\### Правильная модель



```text

task WRITE\_SET   — authoritative security boundary

model WRITE\_SET  — декларация модели

patch paths      — фактические изменения

```



Обязательные условия:



```text

model WRITE\_SET ⊆ task WRITE\_SET

patch paths      ⊆ task WRITE\_SET

```



\### Требуемая логика



```typescript

const taskWriteSet = extractMarkdownListSection(prompt, \[

&#x20; "Write-set",

&#x20; "Write set",

&#x20; "WRITE\_SET",

&#x20; "write-set",

]);



const modelWriteSet = parseWriteSet(parsed.writeSetRaw);



const modelWidening = modelWriteSet.filter(

&#x20; (path) => !taskWriteSet.includes(path),

);



if (modelWidening.length > 0) {

&#x20; return {

&#x20;   status: "error",

&#x20;   contractContent: null,

&#x20;   actualModel: model,

&#x20;   fields: {

&#x20;     error\_reason:

&#x20;       `model\_write\_set\_widening: ${modelWidening.join(", ")}`,

&#x20;     task\_write\_set: taskWriteSet,

&#x20;     model\_write\_set: modelWriteSet,

&#x20;     ...contextFields,

&#x20;   },

&#x20; };

}



validation = validatePatchAgainstWriteSet(

&#x20; diff,

&#x20; JSON.stringify(taskWriteSet),

);

```

\## 6. Authoritative targeted tests



\### Проблема



Текущий allowlist защищает от shell injection, но модель всё ещё может выбрать слишком слабый тест или исключить обязательную проверку.



\### Правильная схема



```text

Task / DoD:

&#x20; authoritative tests



Model:

&#x20; может повторить или рекомендовать тесты



Adapter:

&#x20; выполняет authoritative tests из task

```



\### Требуемая логика



```typescript

const taskTestCommands = extractTestCommands(prompt);

const proposedTestCommands = extractTestCommands(parsed.testsRaw);



const unapprovedTests = proposedTestCommands.filter(

&#x20; (command) => !taskTestCommands.includes(command),

);



if (unapprovedTests.length > 0) {

&#x20; return {

&#x20;   status: "error",

&#x20;   contractContent: null,

&#x20;   actualModel: model,

&#x20;   fields: {

&#x20;     error\_reason:

&#x20;       `model\_proposed\_unapproved\_tests: ${unapprovedTests.join(" | ")}`,

&#x20;     task\_tests: taskTestCommands,

&#x20;     model\_tests: proposedTestCommands,

&#x20;   },

&#x20; };

}



const testCommands = taskTestCommands;

```



Разрешённый allowlist:



```text

.\\.venv\\Scripts\\python.exe -m pytest ...

npm.cmd run test:trigger -- tests/trigger/<name>.test.ts

```



Запрещаются:



```text

|

;

\&

backticks

redirects

aliases

command chaining

```

\## 7. Two-root limitation



Документ описывает отдельные `CODE\_ROOT` и `DOCS\_ROOT`, но текущий trigger выполняет чтение, apply, tests и `git diff` относительно `process.cwd()`.



Следовательно, текущая реализация является single-root.



До отдельной реализации multi-root режима нужно зафиксировать:



```text

LLAMACPP\_AGENT\_ROOT\_MODE=single-root

```



Для задач, затрагивающих два репозитория:



```text

MULTI\_ROOT\_TASK\_NOT\_SUPPORTED

```



Такие задачи следует разбивать на отдельные подпакеты.

\## 8. Дополнительные gates



\### Response model



```typescript

if (response.model !== model) {

&#x20; return error("response\_model\_mismatch");

}

```



\### Finish reason



```typescript

if (choice.finish\_reason !== "stop") {

&#x20; return error(

&#x20;   `unexpected\_finish\_reason: ${choice.finish\_reason}`,

&#x20; );

}

```



\### Hidden reasoning



Запрещать:



```text

<think>

</think>

reasoning\_content

reasoning

```



\### Input budget



```typescript

const estimatedInputTokens =

&#x20; Math.ceil(packedPrompt.length / 3.5);



if (estimatedInputTokens > maxInputTokens) {

&#x20; return error(

&#x20;   `input\_budget\_exceeded: ` +

&#x20;   `${estimatedInputTokens} > ${maxInputTokens}`,

&#x20; );

}

```



В metrics записывать:



```text

input\_chars

estimated\_input\_tokens

max\_input\_tokens

context\_files\_count

context\_truncated

```



\### Dirty-worktree baseline



Перед применением patch сохранить:



```text

git diff --name-only

git diff --cached --name-only

```



Политика:



```text

existing changed path ∩ task write-set ≠ ∅

→ dirty\_write\_set

→ hard fail

```



После apply нужно вычислять новые изменения относительно baseline, а не использовать весь текущий `git diff --name-only`.

\## 10. Сгенерированные скрипты



В комплект входят четыре PowerShell-скрипта.



\### 10.1 Preflight



```text

```



Проверяет:



\* repository и `.git`;

\* Node.js, npm, npx, PowerShell, Git, ripgrep;

\* Python virtual environment;

\* наличие trigger и tests;

\* model registry;

\* alias;

\* context;

\* production runtime profile;

\* process command line;

\* `--no-mmap`;

\* `--prio 2`;

\* наличие новых security gates в trigger;

\* JSON-отчёт.



\### 10.2 API contract smoke



```text

scripts/local\_llamacpp/

Test-LlamaCppApiContract.ps1

```



Проверяет:



\* `/v1/models`;

\* alias;

\* context;

\* `/chat/completions`;

\* `response.model`;

\* `finish\_reason`;

\* отсутствие reasoning;

\* порядок секций;

\* marker;

\* единственный fenced diff;

\* строгий `# NO\_CHANGES`;

\* JSON-отчёт.



\### 10.3 One-command wrapper



```text

scripts/local\_llamacpp/

Use-LlamaCppCodingAgent.ps1

```



Выполняет:



\* запуск production coding runtime;

\* настройку env;

\* preflight;

\* запуск `llamacpp\_agent\_trigger.ts`;

\* контроль exit code.



\### 10.4 Общий integration gate



```text

scripts/local\_llamacpp/

Invoke-LlamaCppAgentIntegrationGate.ps1

```



Последовательно запускает:



```text

preflight

API contract smoke

trigger unit tests

optional live task

JSON report

```



Полные листинги всех четырёх скриптов приведены в сообщении, непосредственно предшествующем этому отчёту.

\## 12. Политика fallback



Не выполнять fallback после детерминированных ошибок:



```text

write\_set\_violation

model\_write\_set\_widening

dirty\_write\_set

targeted\_tests\_failed

execution\_contract\_invalid

input\_budget\_exceeded

response\_model\_mismatch

```



Fallback допустим при transient errors:



```text

server\_unreachable

HTTP 503 Loading model

timeout до получения ответа

process spawn failure

temporary transport error

```

\## 14. Итоговая рекомендация



Не начинать с полной автономной интеграции в orchestrator.



Правильная последовательность:



```text

1\. Синхронизировать alias и runtime contract.

2\. Сделать task write-set authoritative.

3\. Сделать task tests authoritative.

4\. Добавить model/finish/reasoning/input gates.

5\. Зафиксировать single-root limitation.

6\. Запустить preflight, no-op и disposable patch.

7\. Провести пять реальных low-risk задач.

8\. Только затем включить direct\_llamacpp.

```



Качество workflow определяется не только моделью, но и системными ограничениями:



```text

малый write-set

короткий read-set

authoritative task contract

structured output

controlled apply

targeted tests

baseline diff protection

evidence-only execution contract

отдельный reviewer gate

```
