# Сценарии — Как hometutor меняет обучение

> **Это не инструкция.** Это карта реальных моментов: от «только что поставил» до «я наконец освоил эту тему». Каждый сценарий — это живая история человека с конкретной целью и способ системы её решить.

---

## Философия документа

hometutor строится вокруг одной идеи: **твои материалы — это курс, а не архив**.

PDF в папке `data/` — не мёртвый файл. Это источник знаний, который умеет объяснять, проверять, напоминать и показывать прогресс. Каждый сценарий ниже — это момент, когда эта идея становится реальной.

SSoT по demo-артефактам: каждый номер сценария должен иметь YAML в
`doc/scenarios/scenario_NN_*.yaml`. Проверка:
`.\.venv\Scripts\python.exe scripts/check_scenario_ids.py --strict-titles`.
Синхронизация заголовков MD с YAML:
`.\.venv\Scripts\python.exe scripts/sync_scenario_doc_titles.py`.

## Сценарий 1 — Первый запуск: от папки до первого ответа
> *«У меня 12 PDF-лекций и экзамен через неделю. Хочу понять — это поможет или нет?»*

### Контекст

Пользователь только что установил hometutor. Материалы уже есть — они накапливались в виде PDF, Markdown, HTML. Но они мёртвые: искать по ним — это `Ctrl+F` по каждому файлу, и никакой возможности «спросить» у текста. Первый запуск — это проверка главного обещания продукта: **папка с файлами превращается в базу знаний, которую можно спрашивать**.

**Для кого:** Аня, 3-й курс, 12 PDF-лекций, экзамен через 7 дней.  
**Время:** 5 минут.  
**Главный вопрос:** «Оно вообще работает?»

### Шаги

1. **Положить PDF в `data/`.**  
   _Почему:_ никакой облачной загрузки, никакой регистрации — просто скопировать файлы в папку.

2. **Запустить индексацию:**
   ```bash
   python ingest.py
   ```
   Прогресс-бар показывает файл за файлом. Для 12 PDF занимает ~2–4 минуты. При повторном запуске — автоматически проверяет изменения и пропускает уже обработанные файлы (`run_kind=noop`).

3. **Открыть Streamlit UI:**
   ```bash
   streamlit run app/ui/main.py
   ```
   Главный экран показывает 6 режимов. Не пустой чат — понятная точка входа.

4. **Выбрать «Быстрый ответ» → задать вопрос:**  
   _Пример:_ «Как работает обратное распространение ошибки?»

5. **Раскрыть источники под ответом.**  
   Система показывает конкретные фрагменты из конкретных PDF с номерами страниц.

### Под капотом

- Ingest строит чанки и сохраняет их в ChromaDB (`chroma_db/`). Embeddings создаются локально через configured провайдер.
- Retrieval использует hybrid search: keyword + semantic, затем rerank. Поэтому ответ выглядит «умным», а не случайным.

### Сигнал успеха

Пользователь видит ответ, который явно пришёл из его материалов — с фрагментом, именем файла и страницей. Момент, когда человек думает: **«мой материал ожил»**.

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| `missing env variable` | Заполнить `.env` по примеру `.env.example`; минимум: `ANTHROPIC_API_KEY` или `OPENAI_API_KEY` |
| Пустой ответ «не нашёл информации» | Проверить, что PDF содержит копируемый текст (не скан); запустить `GET /index/stats` |
| UI не открывается | Убедиться, что `python main.py` запущен в отдельном терминале (API нужен для UI) |

### Следующий шаг

→ [Сценарий 2](#сценарий-2--home-mode-selector-навигация-за-30-секунд): углубиться в Quick Answer или [Сценарий 3](#сценарий-3--ответ--tutor-за-один-клик): сразу перейти к обучению.

## Сценарий 2 — Home Mode Selector: навигация за 30 секунд
> *«Мне не нужен урок. Мне нужен ответ прямо сейчас.»*

### Контекст

Максим не хочет проходить учебный цикл — у него рабочая задача, которую нужно решить здесь и сейчас. hometutor не должен навязывать обучение тому, кто пришёл за справкой. Быстрый ответ — это режим **скорости без потери прозрачности**: ответ + источники, без лишнего шага.

**Для кого:** Максим, разработчик, нужен конкретный ответ по теме из документации.  
**Время:** 30–60 секунд.  
**Главный вопрос:** «Можно ли получить ответ, не проходя урок?»

### Шаги

1. **Открыть Home → «Быстрый ответ».**

2. **Выбрать один из трёх примеров или ввести свой вопрос.**  
   Если индекс уже заполнен, hero показывает до трёх кликабельных примеров из локальных материалов. Это быстрый старт для первого ответа: пользователь может не придумывать формулировку с нуля.

3. **Ввести вопрос.**  
   _Совет:_ чем конкретнее — тем лучше retrieval. «Как работает rerank в пайплайне?» лучше, чем «расскажи про поиск».

4. **Получить ответ с trust-панелью.**  
   Confidence score, источники, фрагменты — всё рядом. Не нужно идти в другой экран.

5. **Увидеть кнопку «Учить эту тему 5 минут».**  
   Кнопка есть, но не навязана. Максим может закрыть вкладку. Аня — нажать.

### Под капотом

- Query rewrite превращает вопрос в несколько поисковых вариантов.
- `first_answer_examples()` выбирает до трёх уникальных примеров только при наличии индексированного контента.
- Hybrid retrieval: BM25 + semantic similarity → reranker отбирает топ-фрагменты.
- LLM генерирует ответ только по отобранным фрагментам — не из «головы».

### Сигнал успеха

На первом экране видны кликабельные примеры, а ответ содержит тезис, который пользователь может проверить: кликнул на фрагмент — увидел оригинальный контекст в документе.

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Ответ нерелевантен | Переформулировать вопрос более конкретно; проверить `trust score` — если низкий, в базе нет нужного материала |
| Долго ждать | Переключить на `--profile fast` в CLI или проверить latency через Debug Panel |

### Power user tip

В Debug Panel (раскрывается в нижней части UI) видно весь оркестрационный трейс: rewrite → classify → retrieve → rerank → generate. Это помогает понять, почему конкретный ответ получился именно таким.

### Следующий шаг

→ [Сценарий 3](#сценарий-3--ответ--tutor-за-один-клик): нажать «Учить эту тему» и перейти к обучению без потери контекста.

## Сценарий 3 — Ответ → Tutor за один клик
> *«Я понял ответ формально. Но хочу понять по-настоящему.»*

### Контекст

Это самый важный переход в продукте. Большинство RAG-систем заканчиваются на ответе. hometutor начинается с него. Кнопка «Учить эту тему» — это не просто «открыть чат с тьютором». Это передача полного контекста: вопрос, источники, найденные концепты. Тьютор не спрашивает «что хочешь изучить?» — он уже знает.

**Для кого:** Аня после первого ответа по обратному распространению.  
**Время:** 3 минуты.  
**Главный вопрос:** «Сохранится ли контекст при переключении режима?»

### Шаги

1. **В Quick Answer нажать «Учить эту тему 5 минут».**  
   Это не просто навигация — это передача `QueryContext` с вопросом, источниками и extracted_concepts в tutor pipeline.

2. **Тьютор открывается с бейджем текущей цели.**  
   Пользователь видит: «Тема: Backpropagation. Цель: понять, не просто прочитать». Никакого «введите тему с нуля».

3. **Первый ответ тьютора** — не лекция, а вопрос или объяснение с проверкой понимания.  
   Pedagogical router выбирает режим: `explainer` если тема новая, `quiz` если концепт уже встречался в профиле.

4. **После объяснения — предложение «Проверь себя».**  
   Переход в micro-quiz без потери темы сессии.

### Под капотом

- `tutor_orchestrator.py` получает контекст через `QueryContext` с полем `extracted_concepts`.
- Personalization policy читает `PersonalizedLearnerModel` — mastery по концептам определяет, с какого уровня начать.
- Весь трейс тьютора сохраняется в `tutor_learning_resume` — для восстановления на следующий день.

### Сигнал успеха

Первый вопрос тьютора касается именно той темы, которую пользователь только что искал. Не случайный. Не общий.

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Тьютор начинает «с нуля» | Проверить, что переход был через CTA-кнопку, а не через прямую навигацию в «Чат с тьютором» |
| Тьютор выбрал неподходящий режим | Это нормально для первой сессии — `policy clamp` калибруется по истории; с 3–5 сессии точность улучшается |

### Следующий шаг

→ [Сценарий 4](#сценарий-4--мини-квиз-проверь-понимание-за-2-мин): пройти полноценный цикл «объяснение → quiz → feedback → план».

## Сценарий 15 — UX Breakthrough
> *«Система работает. Но хочу, чтобы она чувствовалась быстрой, плавной и мотивирующей.»*

### Контекст

После достижения feature completeness (146 закрытых пакетов, все 13 моментов истины CJM покрыты), UX составляет только 15.5% от всей работы. Следующий фокус — **perceived quality** и **wow-моменты** в ключевых точках пути пользователя. UX Breakthrough Wave — это прорывная волна улучшений, которая превращает функциональную систему в приятную в использовании.

**Для кого:** любой пользователь, который хочет не просто функциональности, а качественного опыта.  
**Время:** улучшения заметны сразу при первом использовании.  
**Главный вопрос:** «Чувствуется ли система быстрой, плавной и мотивирующей?»

### Пять прорывных улучшений

#### 1. Wait UX — Skeleton Screens и Progressive Reveal

**Проблема:** Ожидание первого ответа воспринимается как зависание системы.

**Решение:**
- Skeleton screen появляется <100ms после отправки вопроса
- Progressive reveal: контент раскрывается постепенно, не дожидаясь полной генерации
- Typewriter effect: текст появляется со скоростью 50-80 символов/секунду
- Индикатор прогресса показывает текущий этап (Retrieving sources... / Generating answer...)

**Результат:** Perceived latency <2 секунд до первого видимого контента. Система чувствуется отзывчивой.

**Inspiration:** Notion AI (typewriter effect), Duolingo (skeleton screens)

#### 2. Seamless Tutor Handoff — Сохранение Контекста

**Проблема:** Переход от Q&A к тьютору теряет контекст — приходится объяснять заново.

**Решение:**
- Кнопка «Учить эту тему 5 минут» передаёт полный Context_Payload
- Payload включает: исходный вопрос, тему, источники, confidence score, learner state
- Transition animation 300-500ms сохраняет visual continuity
- Тьютор открывается с кратким резюме переданного контекста

**Результат:** Плавный переход без потери нити обучения. Тьютор «знает», откуда пришёл пользователь.

**Inspiration:** Seamless mode transitions в современных приложениях

#### 3. Celebration UX — Graduation Overlays

**Проблема:** Достижение mastery ≥ 80% — это просто обновление числа в таблице.

**Решение:**
- Graduation overlay с celebration animation 2-3 секунды
- Achievement badge с названием освоенной темы
- Ключевые метрики: время обучения, количество сессий, финальный mastery%
- Primary CTA "Следующая тема" после celebration
- Overlay пропускается через ESC или клик вне области

**Результат:** Wow-момент, который превращает достижение в событие. Мотивация продолжать.

**Inspiration:** Duolingo (celebration UX), Khan Academy (mastery challenges)

#### 4. Flashcard Session Analytics — Детальная Статистика

**Проблема:** После сессии повторения — просто "сессия завершена", без понимания прогресса.

**Решение:**
- Session analytics overlay с распределением оценок: Again/Hard/Good/Easy (количество и %)
- Learning velocity: карточек/минуту
- Retention prediction для повторённых карточек
- Timeline на 7 дней: расписание следующих повторений
- Primary CTA "Разобрать сложные темы" при ≥3 оценок "Again"

**Результат:** Осознанное повторение вместо механического. Понимание своего прогресса.

**Inspiration:** Anki (session stats), Quizlet (study session insights)

#### 5. Home Hub Visual Hierarchy — Hover States и Micro-Interactions

**Проблема:** 6 режимов на главном экране конкурируют за внимание без чёткой иерархии.

**Решение:**
- 2×3 grid layout с чёткой визуальной иерархией
- Hover states с preview действия
- Micro-interactions: scale 1.02, shadow elevation, 200ms transition
- Due badge highlighting через accent color
- Resume card как первый элемент при незавершённой сессии
- Transition animation 300ms при выборе режима

**Результат:** Быстрая навигация. Понятные приоритеты. Живой интерфейс.

**Inspiration:** Современные design systems с micro-interactions

### Под капотом — 3 Парсера с Round-Trip Гарантиями

Все улучшения построены на трёх новых парсерах с формальными гарантиями корректности:

1. **Answer Parser** (`app/answer_parser.py`)
   - Парсит LLM-ответы в структурированный AnswerObject
   - Round-trip property: `parse(format(obj)) == obj`
   - Pretty printer для консистентного форматирования

2. **Tutor Context Parser** (`app/tutor_context_parser.py`)
   - Сериализует/десериализует Context_Payload для seamless handoff
   - Round-trip property: `deserialize(serialize(obj)) == obj`
   - Валидация completeness с списком недостающих полей

3. **Session Analytics Parser** (`app/session_analytics_parser.py`)
   - Парсит flashcard review данные и вычисляет метрики
   - Round-trip property: `from_dict(to_dict(obj)) == obj`
   - SM-2 retention predictions, learning velocity, 7-day schedule

### Property-Based Testing — 11 Correctness Properties

Каждый парсер покрыт property-based tests (hypothesis library, 100 iterations each):

| Property | Что проверяет |
|---|---|

## Сценарий 16 — Интерактивный тур: 5 глав и сохранение прогресса
> *«Хочу за 20 минут обойти продукт без чтения README — с подсказками на экране.»*

### Контекст

После первого запуска **интерактивный тур** ведёт по ключевым режимам: первый ответ с доверием к источникам, переход к тьютору, возврат на следующий день, flashcards и курсовой workspace. Это не просто список функций — это **путь от новичка до эксперта** через 5 глав, каждая из которых закрывает критический момент CJM. Прогресс главы сохраняется в learner state — можно закрыть вкладку и продолжить с того же шага.

**Для кого:** Аня, первый день с hometutor, хочет быстро понять возможности без чтения документации.  
**Время:** ~20 минут на все главы (3-8 минут на главу) или по главе при каждом визите.  
**Главный вопрос:** «Как быстро освоить продукт без внешних инструкций?»

### Структура тура: 5 глав от beginner до expert

Тур построен как **прогрессивное раскрытие**: от базовых функций к продвинутым, с wow-моментами ✨ в ключевых точках.

| Глава | Тема | Уровень | Время | CJM-стадии |
|---|---|---|---|---|
| **1** | Первый ответ | Beginner | 3 мин | Discover → Install → Ingest → First Answer → Trust |
| **2** | От ответа к обучению | Beginner | 5 мин | Switch to Tutor → Tutor Session → Micro-quiz |
| **3** | Возвращаюсь завтра | Intermediate | 4 мин | Adaptive Plan → Resume → Spaced Repetition |
| **4** | Flashcards и долгая память | Advanced | 6 мин | Flashcards Gen → Review → Progress |
| **5** | Курс под ключ | Expert | 8 мин | Course Mode → Master → Export/Sync → Expert Controls |

### Шаги

1. **На главном экране нажать «Пройти интерактивный тур (5 глав)».**
   
   Появляется модальное окно с первым шагом Главы 1. Интерфейс показывает:
   - **Уровень и краткое описание главы** (например, "Beginner · Discover → Install → Ingest → First Answer → Trust")
   - **Заголовок главы и номер шага** (например, "Глава 1. Первый ответ — шаг 1/3")
   - **Название и описание текущего шага**
   - **Прогресс-индикатор** с точками (● для текущего шага, ○ для остальных)
   - **Три кнопки навигации:** "Назад" / "Начать" (или "Дальше") / "Пропустить"

2. **Пройти Главу 1: Первый ответ (3 шага)**
   
   - **Шаг 1 — Добро пожаловать:** Введение в тур, переход к режиму "Быстрый ответ"
   - **Шаг 2 — Попробуйте пример вопроса:** Задать вопрос и получить ответ с источниками
   - **Шаг 3 — Проверьте источники и доверие ✨:** Раскрыть карточки источников и confidence-панель (wow-момент)
   
   После последнего шага — кнопка "К главе 2" автоматически переключает на следующую главу.

3. **Пройти Главу 2: От ответа к обучению (3 шага)**
   
   - **Шаг 1 — Переключение в Tutor:** Нажать CTA перехода к тьютору с сохранением контекста
   - **Шаг 2 — Мини-квиз внутри сессии:** Запустить микро-квиз с адаптацией сложности
   - **Шаг 3 — Подсказка вместо тупика ✨:** При ошибке система даёт подсказку без fail-state (wow-момент)

4. **Пройти Главу 3: Возвращаюсь завтра (3 шага)**
   
   - **Шаг 1 — Resume и план дня:** Оценить resume-карточки и план на сегодня
   - **Шаг 2 — Soft-recovery после паузы ✨:** Очередь due-повторений распределяется мягко (wow-момент)
   - **Шаг 3 — Устойчивость после reindex:** Профиль обучения сохраняется после обновления индекса

5. **Пройти Главу 4: Flashcards и долгая память (3 шага)**
   
   - **Шаг 1 — Создайте колоду:** Сгенерировать карточки из документа и отредактировать
   - **Шаг 2 — Пройдите due-review:** Повторить карточки в режиме Again/Hard/Good/Easy
   - **Шаг 3 — Экспорт и graduation ✨:** Экспорт в Anki и сигнал graduation (wow-момент)

6. **Пройти Главу 5: Курс под ключ (3 шага)**
   
   - **Шаг 1 — Активируйте Course Workspace:** Включить курс в "Темах" для scoped learning
   - **Шаг 2 — Пройдите учебный цикл курса ✨:** Связка scoped QA → flashcards → tutor (wow-момент)
   - **Шаг 3 — Expert controls и полный sync:** Advanced control и export/import sync-пакета
   
   После последнего шага — кнопка "Завершить тур" закрывает модальное окно.

7. **Проверить сохранение прогресса:**
   
   - Закрыть модальное окно кнопкой "Пропустить" или крестиком (X) — прогресс сохраняется
   - Обновить страницу — лента показывает "Тур: глава N из 5 · завершено: M/5"
   - Нажать "Продолжить тур" — модальное окно открывается на том же шаге, где остановились

### Под капотом

- **Прогресс сохраняется в learner state** через `app/tutorial_service.py` — chapter_id, step_index, completed_chapters
- **Автоматическая навигация по целевым режимам:** каждый шаг указывает `target_view` (например, "Быстрый ответ", "Чат с тьютором"), и система автоматически переключает режим при переходе к следующему шагу
- **Wow-моменты помечены ✨** в интерфейсе — это ключевые точки, где пользователь видит уникальную ценность продукта
- **Модальное окно с on_dismiss handler:** закрытие модалки (X / ESC / клик снаружи) работает как "Пропустить" с сохранением прогресса
- **Прогресс-индикаторы:** точки (●○○) показывают позицию внутри главы, лента "Тур: глава N из 5" — общий прогресс
- **Кнопка "Назад"** позволяет вернуться к предыдущему шагу или предыдущей главе

### Сигнал успеха

Пользователь прошёл хотя бы одну главу без внешней инструкции; после перезагрузки тур **не сбрасывается** в начало — можно продолжить с того же места. Wow-моменты (✨) вызывают реакцию: "вот это да, так можно было?!"

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Модальное окно не открывается | Проверить, что UI запущен и session_state инициализирован |
| Прогресс не сохраняется после перезагрузки | Проверить, что `app/tutorial_service.py` имеет доступ к SQLite-базе learner state |
| Автоматическое переключение режимов не работает | Убедиться, что `target_view` в `tutorial_chapters.py` соответствует названиям режимов в UI |
| Кнопка "Продолжить тур" не появляется | Проверить, что `tutorial_progress_hydrated` установлен в session_state |

### Power user tip

Можно пропустить тур и вернуться к нему позже — прогресс сохраняется. Если нужно пройти конкретную главу заново, можно использовать кнопку "Назад" для навигации или начать тур с начала через "Пройти интерактивный тур (5 глав)".

### Связанные артефакты

- **Контент глав (SSoT):** `app/ui/tutorial_chapters.py` — все 5 глав с шагами, wow-моментами и US-ссылками
- **Runtime логика:** `app/ui/tutorial_guide.py` — управление прогрессом, навигация, сохранение
- **Persistence:** `app/tutorial_service.py` — load/save tutorial progress в learner state
- **YAML:** `doc/scenarios/scenario_16_interactive_tour.yaml`
- **E2E тест:** `tests/e2e/demos/scenario_16_interactive_tour.spec.ts`

## Сценарий 4 — Мини-квиз: проверь понимание за 2 мин
> *«Хочу не просто прочитать — хочу, чтобы меня проверили.»*

### Контекст

Марк готовится к техническому собеседованию. Он читал про алгоритмы часами, но не может объяснить их своими словами — верный признак того, что знание поверхностное. Учебная сессия — это структурированный цикл: тьютор объясняет, задаёт вопрос, ждёт ответа, даёт feedback, предлагает следующий шаг. Не бесконечный монолог — диалог с проверкой понимания в каждом такте.

**Для кого:** Марк, готовится к собеседованию, нужна проверка понимания.  
**Время:** 7–10 минут.  
**Главный вопрос:** «Это диалог или бесконечная лекция?»

### Шаги

1. **Home → «Чат с тьютором»** (или продолжение из Сценария 3).

2. **Выбрать цель сессии:**
   - «Понять тему» → режим `explainer + quiz`
   - «Подготовиться к экзамену» → режим `recall + transfer`
   - «Разобрать конкретный вопрос» → точечный разбор

3. **Получить структурированный ответ тьютора:**
   - TL;DR (1–2 предложения)
   - Объяснение с примером из материалов
   - Проверочный вопрос в конце

4. **Ответить на вопрос** (или нажать «Проверь меня»).

5. **Получить feedback:**
   - Правильно → подтверждение + усложнение
   - Неправильно → hint без прямого ответа, повторная попытка

6. **Следующий шаг:** тьютор предлагает deepened topic или переход к Adaptive Plan.

### Под капотом

- Pedagogical router (`tutor_orchestrator.py`) читает `quiz_learning_mode` и mastery vector — и выбирает между `explainer`, `quiz`, `review`.
- `policy clamp` ограничивает сложность вопросов уровнем пользователя: не даёт `transfer`-вопросы до `recall`-стабильности.
- Каждый ответ обновляет `PersonalizedLearnerModel` через `learner_model_service.py`.

### Три уровня quiz-вопросов

| Уровень | Тип | Пример |
|---|---|---|
| `recognized` | Узнавание | «Какой из этих вариантов описывает backpropagation?» |
| `recalled` | Воспроизведение | «Опиши своими словами, что делает chain rule» |
| `transfer` | Применение | «Почему vanishing gradient — проблема именно для RNN?» |

### Сигнал успеха

После сессии пользователь может объяснить тему одним абзацем без подсказки. Adaptive Plan обновился — концепт сдвинулся по mastery.

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Вопросы слишком простые | Явно указать уровень в начале: «я уже знаком с темой, проверь на transfer» |
| Вопросы слишком сложные | Router откалибруется через 2–3 сессии; или явно написать «объясни проще» |

### Power user tip

В режиме `quiz_learning_mode: understand_topic` через API можно задать `quiz_difficulty_preference: transfer` напрямую — полезно для подготовки к сложным интервью.

## Сценарий 5 — Flashcards: сгенерировать, отредактировать, сохранить
> *«Мне нужны карточки по этой главе. Я не хочу тратить час на их написание.»*

### Контекст

Аня работает по методу активного воспроизведения (active recall): она знает, что карточки работают лучше, чем перечитывание. Но написать 30 карточек по лекции вручную — это час работы. Flashcard generation — это генерация карточек с **human-in-the-loop**: LLM создаёт черновик, пользователь редактирует перед сохранением. Разница между «LLM нагенерила мусор» и «я быстро собрал хорошую колоду» — это контроль качества на этапе preview.

**Для кого:** Аня, хочет колоду по конкретной главе за 5 минут.  
**Время:** 4–6 минут.  
**Главный вопрос:** «Смогу ли я проверить карточки до сохранения?»

### Шаги

1. **Flashcards → «Создать новые».**

2. **Выбрать источник:**
   - Документ из базы → карточки по всему файлу
   - Активный курс → карточки в рамках scope курса
   - Загрузить файл → временная генерация без добавления в базу знаний

3. **Задать количество карточек** (5–20; оптимально — 10 на главу).

4. **Нажать «Сгенерировать».**  
   Под капотом: `explain_file()` → LLM по правилу Wozniak (one idea per card) → editable preview.

5. **Проверить и отредактировать preview:**
   - Перефразировать front/back одной карточки
   - Удалить неудачные карточки
   - Добавить собственную карточку вручную

6. **Сохранить колоду** → она уходит в SRS для планирования повторений.

7. **Открыть «Мои колоды».**  
   Список показывает название колоды, источник, общее число карточек и due badge. Отсюда можно сразу перейти в повторение или открыть детали.

8. **Отредактировать карточки внутри колоды или экспортировать в Anki.**  
   В деталях колоды доступны правка Front/Back/тегов, добавление карточки вручную, удаление неудачных карточек и экспорт `.apkg`.

### Под капотом

- LLM создаёт карточки по правилу минимальной информации (Wozniak): один факт — одна карточка. Длинные определения разбиваются.
- После сохранения каждая карточка получает начальные SM-2 параметры: `easiness_factor=2.5`, `interval=1`, `next_review=tomorrow`.
- Карточки хранятся в таблицах `flashcard_decks` + `flashcards` в SQLite.
- Экран колод читает прогресс и due-счётчики через flashcard read cache; после редактирования cache инвалидируется, чтобы UI не показывал старые числа.

### Сигнал успеха

Колода сохранена, видна в «Моих колодах», открывается для редактирования и может быть выгружена в Anki. На главном экране появился due badge с числом карточек к повторению — значит, SRS уже знает о новой колоде.

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Карточки слишком общие | Уменьшить количество (5–7 на сложную тему); указать конкретный раздел документа |
| Карточки содержат несколько идей | Отредактировать в preview или пересгенерировать с явной инструкцией |
| Экспорт в Anki недоступен | Открыть детали колоды: кнопка `.apkg` активна только когда у колоды есть карточки |

> **Обновлено после wave-flashcard-polish:** сценарий отражает текущий Flashcards Hub: список колод, due badge, редактирование карточек, ручное добавление и export `.apkg`.

### Power user tip

После генерации из тьютор-сессии (`quiz → create flashcard`) система автоматически помечает концепт, который вызвал затруднение — такие карточки получают приоритет в SRS.

### Следующий шаг

→ [Сценарий 6](#сценарий-6--srs-повторение-по-интервальному-алгоритму): когда появятся due — пройти первое повторение.

## Сценарий 6 — SRS: повторение по интервальному алгоритму
> *«Три дня учился. Сегодня на главном экране — 11 карточек к повторению. Поехали.»*

### Контекст

Марк третий день работает с продуктом. В его колоде 45 карточек. Спустя три дня часть из них «созрела» по алгоритму SM-2 — система рассчитала, что именно сейчас оптимальное время для повторения, пока материал ещё не стёрся из памяти. Ключевая разница от классического Anki: если карточка вызвала трудности — можно нажать «Объясни» и не просто поставить «Again», а восстановить понимание прямо здесь.

**Для кого:** Марк, 3-й день обучения, накопились due-карточки.  
**Время:** 5–8 минут.  
**Главный вопрос:** «Это Anki или что-то лучше?»

### Шаги

1. **Главный экран → due badge `N карточек`.**  
   Badge показывает число на иконке Flashcards. Это не уведомление — это расписание.

2. **Flashcards → «Повторение»** → загрузить очередь.

3. **Для каждой карточки:**
   - Прочитать front
   - Вспомнить ответ (не подглядывать!)
   - Нажать «Показать»
   - Оценить: **Снова / Трудно / Хорошо / Легко**

4. **На сложной карточке — «Не знаю, объясни»:**  
   Открывается мини-tutor в контексте этой карточки. Объяснение → возврат к очереди.

5. **После последней карточки — summary:**  
   `Снова: 2 | Трудно: 3 | Хорошо: 5 | Легко: 1`. Следующее повторение — через X дней.

### Под капотом — алгоритм SM-2

| Оценка | Что происходит |
|---|---|
| **Снова** | Interval → 1 день; easiness_factor снижается |
| **Трудно** | Interval × 1.2; easiness_factor снижается |
| **Хорошо** | Interval × easiness_factor (обычно ×2.5) |
| **Легко** | Interval × easiness_factor × 1.3; easiness_factor растёт |

Оптимальный момент повторения рассчитывается так, чтобы попасть в «кривую забывания» Эббингауза до точки потери информации.

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Слишком много due за раз | Soft-recovery: если пропустил > 3 дней, система разносит overdue по нескольким сессиям |
| Карточка всегда «Снова» | Открыть тьютор по этой теме — возможно, нужен новый уровень объяснения |

### Сигнал успеха

После сессии: прогресс-бар показывает долю карточек по статусам. Дата следующей сессии сдвинулась вперёд — алгоритм оптимизирует нагрузку.

## Сценарий 7 — Прогресс и слабые места
> *«Открыл продукт. Не помню, где остановился. Хочу продолжить, не тратя время на поиск.»*

### Контекст

Главный friction самостоятельного обучения — это не сложность материала. Это «где я был вчера?». Большинство инструментов предлагают пустой чат или список файлов. hometutor показывает три точки входа прямо на главном экране: продолжить чат с тьютором, продолжить план, повторить карточки. Второй день начинается с **действия**, а не с навигации.

**Для кого:** Аня, открывает UI на следующее утро.  
**Время:** 2 минуты до первого действия.  
**Главный вопрос:** «Придётся ли снова объяснять системе, кто я и что изучаю?»

### Resume-карточки на главном экране

| Карточка | Что показывает | Действие |
|---|---|---|
| **Тьютор** | Последняя тема, дата | «Продолжить чат» |
| **План** | Сколько шагов осталось | «Открыть план» |
| **Flashcards** | N due сегодня | «Начать повторение» |

### Шаги

1. **Открыть UI.** Главный экран — resume-карточки видны сразу, без скролла.

2. **Нажать «Продолжить чат»** → тьютор восстанавливает контекст из `tutor_learning_resume`:
   - Последняя тема
   - Последний вопрос
   - Текущий mastery по концептам

3. **Или нажать due badge → Flashcards** → сразу в очередь повторения.

4. **Или открыть «Мой прогресс» → посмотреть план на сегодня** → выбрать следующий шаг.

### Под капотом

- `tutor_learning_resume` — single-row snapshot в SQLite с WAL durability. Пишется в конце каждой сессии, не теряется при перезапуске.
- После reindex snapshot обновляется через lineage sync: концепты привязываются к обновлённому индексу.

### Сигнал успеха

От открытия UI до начала учёбы — меньше 30 секунд. Первое действие не требует навигации.

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Resume-карточка пропала | Проверить, что сессия завершилась корректно (не через принудительное закрытие браузера) |
| Тьютор не помнит прошлую тему | После reindex lineage sync нужно время; обновлённый бейдж «профиль обновлён» появится автоматически |

## Сценарий 8 — Доверие к ответу: источники и confidence
> *«25 лекций. Хочу пройти их как курс, а не как беспорядочный набор файлов.»*

### Контекст

Сергей проходит курс по машинному обучению. У него 25 файлов в папке `data/ML-Course/`. В стандартном RAG-режиме он может задавать вопросы по всей базе, но не видит прогресс по конкретному курсу, не получает план именно по этому набору материалов, карточки смешиваются с другими темами. Course Workspace решает это одним действием: **папка становится курсом** — со своим планом, карточками, прогрессом и graduation.

**Для кого:** Сергей, курс из 25 лекций, хочет структуру и прогресс.  
**Время:** 10–15 минут на активацию.  
**Главный вопрос:** «Может ли папка с файлами стать курсом?»

### Шаги

1. **Положить файлы курса в одну папку** (например, `data/ML-Course/`).

2. **Открыть Topics → найти тему, соответствующую папке → «Активировать как курс».**

3. **Система создаёт:**
   - Synthesis (обзор всего корпуса)
   - Learning plan (структурированный маршрут по темам)
   - Course scope для flashcards и tutor

4. **Открыть Course Cockpit.**  
   Кокпит показывает daily briefing: что уже сделано, какой следующий блок в курсе, сколько осталось до graduation и есть ли overdue-нагрузка.

5. **Сгенерировать flashcards по курсу** → карточки привязаны к scope курса, не смешиваются с остальными.

6. **Открыть прогресс** → отдельный фильтр по курсу: mastery по темам курса, graduation status.

7. **Пройти цикл:** Review (SM-2) → Tutor (пробелы) → Quiz → план обновляется.

8. **Graduation:** при mastery ≥ 80% по всем концептам — появляется graduation overlay с итогом курса и CTA к следующей теме.

### Retention Resilience: Возврат после пропуска

**Сценарий:** Сергей пропустил 5 дней курса. Накопилось 47 overdue карточек и 8 пропущенных тем.

9. **При возврате система показывает Recovery Banner:**
   - «Добро пожаловать обратно! Накопилось 47 карточек и 8 тем. Начнём с короткой разминки.»
   - Кнопка «Начать восстановление» вместо обычного плана

10. **Recovery Flow (3 этапа):**
   - **Этап 1: Quick Refresh (5 мин)** — 5 ключевых карточек из overdue для восстановления контекста
   - **Этап 2: Gap Remediation (10 мин)** — короткое объяснение самой проблемной темы из пропущенных
   - **Этап 3: Promise** — система предлагает конкретный план на следующую сессию: «Завтра: 10 карточек + тема "Градиентный спуск"»

11. **После recovery — Soft Landing:**
    - Overdue карточки распределяются на 3-5 дней (не все сразу)
    - Первые 2 дня — облегчённая нагрузка (15 мин вместо 30)
    - Система отслеживает repeated misses: если пропуск повторяется 3+ раза за 2 недели → предлагает переключить режим темпа с Deep на Steady

### Под капотом

- Course scope передаётся в `query_mode=tutor` как `course_filter` — retrieval ограничивает поиск файлами курса.
- Graduation overlay считается через `adaptive_plan.py`: если все концепты в статусе `transfer` стабильны > 7 дней → graduated.
- Daily briefing в начале курсовой сессии показывает: «вчера прошли X, сегодня — Y, до graduation осталось Z».
- **Recovery mechanism:** при gap > 3 дней система активирует `course_recovery_flow`:
  - Выбирает 5 highest-priority карточек через `select_recovery_cards()` (критерий: easiness_factor + recency)
  - Определяет самую проблемную тему через `identify_weakest_concept()` из knowledge graph
  - Генерирует next-session promise через `generate_session_promise()` с учётом доступного времени пользователя
- **Soft landing:** overdue карточки распределяются через `distribute_overdue_load()` — алгоритм разносит нагрузку равномерно, избегая overwhelming первой сессии
- **Churn prevention:** `repeated_miss_detector` отслеживает паттерн пропусков и предлагает адаптацию темпа или уменьшение scope

### Режимы темпа

| Режим | Нагрузка | Для кого |
|---|---|---|
| **Steady** | 20–30 мин/день | Планомерное освоение |
| **Deep** | 60–90 мин/день | Подготовка к дедлайну |

### Сигнал успеха

В Topics появился статус курса с прогресс-баром. Course Cockpit показывает daily briefing, карточки из курса имеют отдельный badge, а при достижении цели появляется graduation overlay. Тьютор при открытии говорит: «Продолжаем курс ML: сегодня — тема 4».

**После recovery:** Recovery banner исчезает, появляется обычный daily plan. Пользователь видит конкретное обещание на следующую сессию в виде preview-карточки.

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Слишком много overdue карточек (100+) | Recovery flow автоматически распределяет на 7 дней; можно вручную сбросить часть карточек через «Архивировать старые» |
| Recovery кажется слишком лёгким | Система намеренно снижает нагрузку в первые дни; через 2-3 сессии вернётся к обычному темпу |
| Repeated misses продолжаются | Система предложит переключить режим темпа или уменьшить scope курса; можно также временно приостановить курс без потери прогресса |
| Promise на следующую сессию не появляется | Проверить, что recovery flow завершён полностью (все 3 этапа); promise генерируется только после завершения |

> **Обновлено после wave-course-retention-resilience:** добавлены recovery flow, session promise, и churn prevention механизмы для возврата после пропусков.

> **Обновлено после wave-course-learning-v2:** добавлены Course Cockpit, daily briefing и graduation overlay как основной путь курсового прогресса.

## Сценарий 9 — Персональный план обучения
> *«Я устал каждый день решать, что учить. Пусть система скажет.»*

### Контекст

Вторая неделя обучения — это точка, где большинство бросают. Первая неделя — мотивация и новизна. Вторая — рутина и выбор: «за что браться?» Adaptive Daily Plan убирает этот выбор. Он читает состояние пользователя — что нужно повторить, где пробел, что можно начать — и предлагает структурированный список на день. Не произвольный, а рассчитанный.

**Для кого:** Марк, вторая неделя, усталость от планирования.  
**Время:** 5 минут.  
**Главный вопрос:** «Кто-нибудь скажет мне, что делать сегодня?»

### Из чего строится план

| Блок | Что включает | Приоритет |
|---|---|---|
| **Review** | Due карточки и концепты к повторению по SM-2 | Первый |
| **Gap** | Концепты с mastery < порога или ошибки в последних quiz | Второй |
| **New** | Следующие темы в learning path или курсе | Третий |

### Шаги

1. **Открыть «Мой прогресс» → блок «План на сегодня».**

2. **Развернуть каждый блок** (Review / Gap / New) → увидеть конкретные концепты.

3. **Нажать на первый пункт Gap** → прямой переход в тьютора с готовой целью.

4. **После сессии вернуться в план** → пункт отмечен, план обновился.

### Под капотом

- `adaptive_plan.py` строит план из трёх источников: `spaced_repetition` (due), `knowledge_graph` (mastery gaps), `learning_plan_service` (next topics).
- Plan diff: если план изменился с прошлой сессии (добавились due, закрылись пробелы) — показывается diff «что изменилось».
- Блоки кликабельны — это не таблица, это action list.

### Сигнал успеха

Пользователь не решает, что учить — он выбирает из готового списка и действует. Вечером: все Review закрыты, 1–2 Gap закрыто, 1 New начато.

### Power user tip

Через API: `GET /adaptive-plan` возвращает структуру плана в JSON — удобно для интеграции с внешними todo-инструментами.

## Сценарий 10 — Возвращение на день 2: resume + soft-recovery
> *«Я готовлюсь к экзамену. Хочу понять: что я уже знаю, а что ещё нет?»*

### Контекст

Стандартный прогресс в обучении — это количество: «прочитал 15 страниц», «прошёл 3 урока». hometutor считает **качественный** прогресс: не «прочитал», а «могу применить». Mastery vector — это карта освоения по концептам. Когда концепт достигает уровня `transfer` и остаётся стабильным 7+ дней — он получает graduation: система больше не включает его в daily gap.

**Для кого:** Аня перед экзаменом, хочет видеть реальную карту знаний.  
**Время:** 5 минут.  
**Главный вопрос:** «Есть ли разница между "прочитал" и "освоил"?»

### Уровни mastery

| Уровень | Что означает | Как достигается |
|---|---|---|
| `recognized` | Узнаёт тему при чтении | Первый контакт с концептом |
| `recalled` | Может воспроизвести | Правильные ответы на `recall` quiz |
| `transfer` | Может применить в новой ситуации | Правильные ответы на `transfer` quiz |
| `graduated` | Стабильно освоено >7 дней | Нет деградации в периоде |

### Шаги

1. **Открыть «Мой прогресс»** → mastery vector по всем концептам.

2. **Найти graduated-концепты** — зелёные, стабильные. Их можно не трогать перед экзаменом.

3. **Найти weak-концепты** — ниже `recalled`. Это приоритет для последних дней.

4. **Открыть Knowledge Graph** → визуализация связей между концептами и их mastery-статус.

5. **Нажать на weak-концепт → тьютор с целевой темой.**

### Под капотом

- Graduation contract: `transfer` стабильность > 7 дней и нет ошибок в последних 3 quiz-сессиях.
- Graduated концепт убирается из блока Gap в Adaptive Plan — система «знает», что он освоен.
- Knowledge Graph строится через `knowledge_graph.py` — концепты связаны через предпосылки (prerequisite edges).

### Сигнал успеха

Аня видит: «Graduated: 12 концептов. Weak: 4 концепта». Она знает точно, что учить в последние два дня.

### Следующий шаг

→ [Сценарий 11](#сценарий-11--flashcards--экспорт-в-anki): убедиться, что ответы системы на weak-темы достоверны.

## Сценарий 11 — Flashcards → экспорт в Anki
> *«Как я могу доверять ответу, который дала система?»*

### Контекст

Это правильный вопрос. LLM могут галлюцинировать. Но hometutor построен на другом принципе: **ответ генерируется только по найденным фрагментам** — не из предобученных знаний модели. Trust-панель делает этот принцип видимым: каждый тезис в ответе можно проследить до конкретного фрагмента конкретного файла.

**Для кого:** скептически настроенный пользователь или reviewer, проверяющий качество.  
**Время:** 3 минуты.  
**Главный вопрос:** «Откуда взялся этот ответ?»

### Шаги

1. **Получить ответ** в Quick Answer или Tutor.

2. **Раскрыть «Подробнее об источниках»:**
   - Имена файлов и страницы
   - Confidence score (насколько retrieval уверен в покрытии)
   - Топ-фрагменты с оценкой релевантности

3. **Кликнуть по фрагменту** → увидеть полный контекст из оригинального документа.

4. **Открыть Debug Panel** → полный оркестрационный трейс:
   ```
   rewrite → classify → retrieve → rerank → generate
   ```
   Каждый шаг — с входными и выходными данными.

### Что значат оценки

| Индикатор | Значение |
|---|---|
| `trust: high` | Retrieval нашёл прямой ответ, coverage >80% |
| `trust: medium` | Ответ собран из нескольких частично релевантных фрагментов |
| `trust: low` | Материалы не содержат прямого ответа; ответ сгенерирован по косвенным данным |

### Сигнал успеха

Пользователь может проследить каждый тезис до источника. Ответ с `trust: low` — это сигнал: в базе нет нужного материала, стоит добавить.

## Сценарий 17 — План на сегодня: что изменилось (diff)
> *«Вчера план был одним, сегодня другим. Это баг или адаптация?»*

### Контекст

Adaptive Daily Plan пересчитывается из due, пробелов и новых тем. Чтобы снять недоверие к «магии», в UI есть блок **«Что изменилось в плане»**: какие концепты **появились** в шагах и какие **исчезли**.

**Для кого:** Марк на второй неделе, после перерыва или всплеска повторений.

### Шаги

1. Открыть **Tutor** (или экран, где развёрнут адаптивный план).
2. Развернуть **«Адаптивный план и прогноз»** → **Adaptive Daily Plan**.
3. Раскрыть **«Что изменилось в плане»** и прочитать списки добавленных/убранных концептов.
4. Сравнить diff с причиной: due-карточки, свежая ошибка в quiz, слабый mastery или новая тема из learning path.

### Сигнал успеха

Изменение плана **объяснимо** одним экраном: добавленные и убранные концепты видны отдельно, а причина пересчёта привязана к локальным сигналам. Без diff пользователь не гадает, «сломалось» ли что-то.

`doc/scenarios/scenario_17_plan_diff.yaml`, `tests/e2e/demos/scenario_17_plan_diff.spec.ts`, smoke `tests/e2e/plan_diff_ux_wave_smoke.spec.ts`.

## Сценарий 18 — Главная как карта возврата: режимы, due и следующий шаг
> *«Утром открыл ноутбук — хочу одним взглядом понять: куда жать дальше.»*

### Контекст

Главный экран — это **Mission Control**: центр управления обучением. Вместо того чтобы заставлять пользователя выбирать из множества опций, система показывает **один лучший следующий шаг** в SSR-баннере сверху, а ниже — семь чётких направлений для явного выбора. Это закрывает friction «где я был» и «что сегодня делать» без когнитивной перегрузки.

**Для кого:** любой returning learner, особенно после первой недели использования.  
**Время:** 30 секунд до первого действия.  
**Главный вопрос:** «Что мне делать прямо сейчас?»

### До и После

**До wave-mission-control-home:**
Главный экран показывал шесть карточек режимов без приоритизации. Пользователь сам решал, куда идти — это создавало friction выбора, особенно для returning learners.

**После wave-mission-control-home:**
Mission Control показывает **SSR-баннер** с одним приоритетным действием («Повторить 8 карточек» или «Продолжить тему X») и **семь destination tiles** ниже для явной навигации. Никакого stacked ribbon выше fold — всё видно сразу.

**После wave-home-mode-selection-v2:**
Каждая destination tile теперь включает три ключевых улучшения:

1. **Intent cards** (epoch-home-mode-card-labels): каждая из 6 основных карточек режимов показывает краткое объяснение «best-for» — для какой цели подходит этот режим. Например:
   - Quick Answer: «Лучше для: быстрый ответ на конкретный вопрос»
   - Tutor: «Лучше для: глубокое изучение темы с проверкой понимания»
   - Flashcards: «Лучше для: повторение и закрепление уже изученного»

2. **Time badge** (epoch-home-mode-flashcard-time-badge): карточка Flashcards показывает оценку времени на повторение («~15 мин» или «8 карточек»), чтобы пользователь мог оценить commitment перед началом сессии. Это помогает выбрать режим, соответствующий доступному времени.

3. **Preview drawer** (epoch-home-mode-preview-drawer): при наведении на tile появляется компактное превью с кратким описанием, что ждёт внутри режима — это снижает friction переключения контекста и помогает принять уверенное решение.

**Wow-момент:**
Система сама говорит: «вот что тебе сейчас нужно». Не нужно вспоминать, где остановился — SSR уже знает. А если выбираешь явно — intent cards показывают, для чего каждый режим, time badge помогает оценить commitment, а preview drawer даёт уверенность перед переходом. Выбор режима становится осознанным, а не случайным.

### Шаги

1. **Открыть главный экран** → увидеть SSR-баннер сверху.
   
   SSR-баннер показывает:
   - **Рекомендованное действие:** «Повторить 8 карточек» или «Продолжить тему: Backpropagation»
   - **Краткое объяснение:** «Почему сейчас» — например, «8 карточек due сегодня» или «Пробел в mastery по этой теме»
   - **Кнопка действия:** прямой переход в режим

2. **Нажать на SSR-баннер** → прямой переход к рекомендованному действию.
   
   Или проигнорировать и выбрать из семи destination tiles ниже.

3. **Оценить семь destination tiles:**
   - **Quick Answer** — быстрый ответ на вопрос (intent: «Лучше для: быстрый ответ на конкретный вопрос»)
   - **Tutor** — учебная сессия с тьютором (intent: «Лучше для: глубокое изучение темы»)
   - **Flashcards** — повторение карточек (с **due badge** «8 карточек» и **time badge** «~15 мин»)
   - **Course** — активный курс (если есть)
   - **Progress** — прогресс и план на сегодня
   - **Topics** — обзор тем и Knowledge Graph
   - **Settings** — настройки и backup
   
   Каждая карточка показывает **intent card** с кратким объяснением цели режима.

3a. **Навести на tile** → увидеть **preview drawer** с описанием, что ждёт внутри режима.
   
   Preview помогает понять, куда идёшь, ещё до клика — снижает friction переключения контекста.

4. **Выбрать destination tile** → прямой переход в режим без дополнительной навигации.

5. **Вернуться на Home** → SSR-баннер обновился с учётом выполненного действия.

### Под капотом

- SSR-баннер использует локальный роутер из `app/ssr/` — анализирует due карточки, mastery gaps, последнюю сессию тьютора и предлагает один приоритетный шаг.
- Семь destination tiles — это фиксированная навигация, не зависящая от SSR. Пользователь всегда может выбрать явно.
- Никакого stacked ribbon или скрытых элементов выше fold — весь Mission Control виден без скролла.
- После выполнения действия (например, повторения карточек) SSR пересчитывает приоритет и обновляет баннер при возврате на Home.

### Сигнал успеха

До первого осмысленного действия — **меньше 30 секунд**. Пользователь либо следует SSR-рекомендации (один клик), либо выбирает из семи направлений (два клика). Никакого «где я был вчера» — система уже знает.

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| SSR-баннер показывает неподходящее действие | Выбрать destination tile явно; SSR откалибруется через 2–3 сессии |
| Не вижу due badge на Flashcards | Проверить, что карточки созданы и прошло время для повторения (SM-2 schedule) |
| Mission Control не загружается | Проверить, что API запущен (`python main.py`) и UI подключён |

### Power user tip

SSR-баннер можно временно скрыть через Settings → «Hide SSR banner» — останутся только семь destination tiles. Полезно для опытных пользователей, которые предпочитают явный выбор.

## Сценарий 20 — Адаптивный маршрут тьютора
> *«Иногда мне нужно объяснение, иногда проверка — не хочу сам переключать режимы.»*

### Контекст

Тьютор опирается на **педагогический роутер**: по состоянию learner'а и контексту сессии приоритизируются объяснение, микро-квиз или отсылка к повторению карточек. Пользователь не обязан знать названия режимов — система подталкивает к следующему полезному шагу.

**Для кого:** Аня в середине курса, уже с накопленным mastery и историей quiz.
**Время:** 3–5 минут.
**Главный вопрос:** «Подстроится ли тьютор под мой уровень без ручной настройки?»

### Шаги

1. Открыть **Tutor** с активной темой.
2. Проверить header: он напоминает, что тьютор умеет объяснить, привести пример, проверить и предложить следующий шаг.
3. Пройти предложенный шаг: объяснение, «Проверь меня» или короткий review.
4. Ошибиться в micro-quiz и получить hint без готового ответа.
5. После успешного ответа увидеть переход к закреплению, усложнению или новой теме.
6. При необходимости выбрать действие под ответом: «Объясни проще», «Дай пример», «Проверь меня», «Следующий шаг».

### Сигнал успеха

Сессия не застревает: после ошибки за один turn появляется **hint** и «следующий шаг», а не тупик; после успеха пользователь видит закрепление или переход к новой теме. Маршрут ощущается как tutor loop, а не как набор ручных кнопок.

### Под капотом

- `tutor_mastery_level` и `quiz_feedback` влияют на глубину следующего шага: explain → quiz → review.
- Intent clamps в `tutor_mastery_forecast_panel.py` не дают тьютору перескочить к неподходящей тактике: после ошибки сначала восстановление, после уверенного ответа — усложнение.
- Micro-quiz обновляет mastery estimate и spaced repetition, чтобы следующий шаг был основан на состоянии learner'а.

- **YAML:** `doc/scenarios/scenario_20_adaptive_tutor_route.yaml`.
- **E2E smoke:** `tests/e2e/agentic_tutor_depth_smoke.spec.ts`.
- **Автотесты:** demo e2e в `tests/e2e/demos/` пока нет; текущая защита — smoke-тест глубины тьютора.

## Сценарий 21 — Умный Маршрутизатор: один лучший следующий шаг
> *«Я открыл приложение утром. Не хочу выбирать между tutor, quiz, карточками и прогрессом — скажите, что делать сейчас.»*

### Контекст

У hometutor много сильных режимов, но именно это создаёт новую боль: пользователь может застрять перед меню. **Умный Маршрутизатор** превращает набор инструментов в учебный автопилот: смотрит на локальное состояние и показывает **один лучший следующий шаг** с объяснением «почему сейчас».

**Для кого:** Аня после первого ответа, квиза или возвращения на следующий день.
**Время:** 1 минута.
**Главный вопрос:** «Что мне делать прямо сейчас?»

### Шаги

1. Открыть главный экран или вкладку **«Прогресс обучения»**.
2. Найти карточку **«Умный следующий шаг»**.
3. Прочитать блок **«Почему сейчас»**: рекомендация привязана к локальным сигналам, а не к случайному совету.
4. Нажать главную CTA-кнопку или выбрать одну из безопасных альтернатив.
5. При необходимости раскрыть короткий след решения роутера и увидеть, какие сигналы повлияли на маршрут.

### Surface Parity — SSR везде, где нужен следующий шаг

**Проблема:** SSR был только на главном экране. Но «что делать дальше?» возникает в трёх местах:
- После завершения flashcard-сессии
- На вкладке «Прогресс обучения»
- После выхода из тьютора

**Решение:**
- **SSR-карточка на Progress Hub** — показывает следующий шаг после просмотра mastery vector
- **SSR-карточка на Flashcards Hub** — появляется после завершения review-сессии
- **Единый session contract** — все три поверхности используют одну и ту же логику маршрутизации

**Wow-момент:** Система понимает тебя и показывает один лучший следующий шаг **в любой точке учебного цикла**, не только на главной.

### Под капотом

Роутер собирает локальные сигналы:

- очередь карточек SM-2;
- просроченные темы;
- результат последнего quiz;
- свежий ответ с источниками;
- слабый концепт;
- незавершённая tutor-сессия;
- первый блок адаптивного плана.

Затем применяет приоритеты **retention → recovery → new learning → safe default** и отдаёт `SmartStudyRecommendation`: действие, причину, навигацию и 2–4 альтернативы.

**Surface Parity Implementation:**
- Единый компонент `SmartStudyRouterCard` рендерится на трёх поверхностях
- Session contract гарантирует консистентность: один и тот же learner state → одна и та же рекомендация
- Smoke-тесты проверяют, что SSR работает на всех трёх поверхностях с одинаковой логикой

### Сигнал успеха

Пользователь не рыщет по интерфейсу и не выбирает режим вручную. Он понимает: **вот следующий шаг, вот почему он важен, вот что можно сделать вместо него**.

Негативные пути тоже видны: при отсутствии сильных сигналов SSR возвращает
`safe_default`; при низком trust предлагает сначала Q&A sources; при «Не
сейчас» включает мягкий шаг, но не забывает retention debt.

**Surface Parity Success Signal:** SSR-карточка появляется на Progress Hub и Flashcards Hub с той же логикой, что и на главной. Пользователь видит консистентные рекомендации независимо от точки входа.

### Почему это не chatbot

SSR не генерирует скрытый совет из облачного профиля. Он показывает локальный
rule-based route, evidence ledger и безопасные альтернативы. LLM может
переписать объяснение только как текст, но не меняет `hint_kind` и CTA.

## Сценарий 22 — Умный Маршрутизатор с ИИ: 5 уровней AI Vision
> *«Сейчас роутер объясняет следующий шаг. А что будет, когда он начнёт учитывать мою память, привычки и граф знаний?»*

### Контекст

Текущий SSR уже полезен как детерминированная policy-система. Следующий уровень — **гибридный AI Router**: правила остаются страховочной сеткой, а ML/LLM-компоненты делают маршрут персональнее и понятнее. Это не отдельный чат-бот, а надстройка над тем же учебным циклом.

**Для кого:** защита проекта, product demo, roadmap discussion.
**Время:** 2 минуты.
**Главный вопрос:** «Почему это не просто кнопка, а будущий персональный учебный проводник?»

### Шаги

1. Показать базовый SSR: **следующий шаг + почему сейчас + CTA**.
2. Показать уровень 1: локальная ML-модель кривой забывания меняет приоритет карточек.
3. Показать уровень 2: LLM пишет понятное объяснение, но не принимает routing decision.
4. Показать уровень 3: недельный план строится из прогресса, целей и доступного времени.
5. Показать уровень 4: граф концепций не даёт перескочить через prerequisite.
6. Показать уровень 5: feedback loop учится на отклонённых рекомендациях.

### Outcome Receipts — Видимый результат маршрутизации

**Проблема:** Пользователь нажал на SSR-рекомендацию, выполнил действие (повторил карточки, прошёл quiz), но не видит, как это повлияло на его состояние.

**Решение:**
- **Outcome Receipt** появляется после завершения routed action
- Показывает локальный результат: сколько карточек повторено, какой mastery изменился, что обновилось в плане
- Включает «До/После» метрики: retention debt снизился с 15 до 7 карточек
- Предлагает следующий шаг на основе нового состояния

**Wow-момент:** Система не просто отправила тебя куда-то — она показала, **что изменилось** после твоего действия.

### Quiet Access Mode — Низкая когнитивная нагрузка

**Проблема:** Иногда пользователь устал и не хочет читать объяснение «почему сейчас» — просто хочет быстро выполнить следующий шаг.

**Решение:**
- **Quiet Mode toggle** в настройках SSR-карточки
- В Quiet Mode: только CTA-кнопка и иконка действия, без текста объяснения
- Объяснение доступно через раскрывающийся блок «Почему?»
- Альтернативные действия остаются видимыми — не теряются entry points

**Результат:** Пользователь может работать в режиме низкой когнитивной нагрузки, но не теряет контроль и прозрачность.

### Пять уровней AI Vision

| Уровень | Что добавляет | Почему это важно |
|---|---|---|
| 1. Local ML Layer | персональная кривая забывания | повторение назначается под конкретного learner'а |
| 2. LLM Explanation | живое «почему сейчас» | подсказка звучит как тьютор, а не шаблон |
| 3. Proactive Planner | план на неделю | система ведёт дальше одного клика |
| 4. Concept Graph Router | prerequisite-aware маршрут | сложная тема не появляется раньше базовой |
| 5. Misroute Feedback Loop | обучение на отказах | стратегия меняется, если совет не помогает |

### Сигнал успеха

Зритель понимает разницу: **SSR v2.0** — объяснимый автопилот с жёсткими правилами; **SSR AI Vision** — персональный проводник, который остаётся local-first и постепенно усиливается ML/LLM слоями.

Контрастный пример для demo: один и тот же learner state даёт `cards_due` в
SSR v2.0; Level 1 может временно поднять `quiz_failed`, если retention риск по
карточкам низкий, а recovery-риск высокий; Level 2 меняет только формулировку
«почему сейчас», оставляя route неизменным.

**Outcome Receipts Success Signal:** После завершения routed action пользователь видит outcome receipt с метриками «До/После» и понимает, как его действие изменило состояние системы.

**Quiet Mode Success Signal:** Пользователь может переключиться в Quiet Mode и работать с минимальной когнитивной нагрузкой, но entry points и альтернативы остаются доступными.

## Сценарий 23 — SSR Trust: Понятные доказательства маршрута
> *«Роутер говорит "повтори карточки". Но почему именно карточки, а не quiz? Хочу понять логику.»*

### Контекст

Базовый SSR показывает рекомендацию и короткое объяснение «почему сейчас». Но для опытного пользователя этого недостаточно — он хочет понять, **почему роутер выбрал это действие, а не другие видимые режимы**. SSR Trust Level делает routing decision прозрачным: показывает компактные локальные доказательства без облачного scoring.

**Для кого:** Марк, опытный пользователь, хочет понимать логику системы.
**Время:** 2 минуты.
**Главный вопрос:** «Почему роутер выбрал именно это действие?»

### Шаги

1. **Открыть SSR-карточку** на главном экране или Progress Hub.

2. **Увидеть рекомендацию** с кратким объяснением: «Повтори 12 карточек — retention debt растёт».

3. **Нажать "Почему именно это?"** — раскрывается Evidence Panel.

4. **Изучить Evidence Panel:**
   - **Routing Decision Tree:** визуализация приоритетов (retention → recovery → new learning)
   - **Local Signals:** конкретные числа из learner state (12 due cards, 0 failed quizzes, 3 weak concepts)
   - **Alternative Actions:** почему другие режимы не выбраны (quiz: нет failed attempts; tutor: нет active session)
   - **Confidence Score:** насколько сильный сигнал (high/medium/low)

5. **Проверить локальность:** все доказательства из локального состояния, без облачного профиля.

6. **Выбрать действие:** согласиться с рекомендацией или выбрать альтернативу с пониманием tradeoff.

### Под капотом — Evidence Ledger

**Evidence Ledger** — это структурированный лог routing decision:

```python
@dataclass
class RoutingEvidence:
    chosen_action: str  # "cards_due"
    reason: str  # "Retention debt growing"
    local_signals: Dict[str, Any]  # {"due_cards": 12, "failed_quizzes": 0, ...}
    priority_order: List[str]  # ["retention", "recovery", "new_learning"]
    alternatives: List[AlternativeAction]  # Почему не выбраны
    confidence: str  # "high" | "medium" | "low"
    timestamp: datetime
```

**Alternative Action:**
```python
@dataclass
class AlternativeAction:
    action: str  # "quiz"
    reason_not_chosen: str  # "No failed quiz attempts in last 7 days"
    would_be_chosen_if: str  # "If failed_quizzes >= 3"
```

**Ключевые принципы:**
- **Локальность:** все сигналы из SQLite learner state, без cloud scoring
- **Компактность:** Evidence Panel умещается в 200-300 слов
- **Объяснимость:** каждое решение имеет явное правило и threshold

### Wow-момент

Система не просто говорит «делай это» — она показывает **почему это, а не то**, с конкретными числами и правилами. Пользователь понимает логику и может осознанно выбрать альтернативу.

### Сигнал успеха

**Количественные метрики:**
- Evidence Panel открывается <200ms
- Все сигналы локальные (0 cloud API calls)
- Confidence score соответствует реальной полезности рекомендации

**Качественные метрики:**
- "Понимаю, почему система предлагает это действие"
- "Вижу, что будет, если выберу альтернативу"
- "Доверяю системе, потому что вижу логику"

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Evidence Panel показывает "low confidence" | Это нормально при слабых сигналах — система честно признаёт неопределённость |
| Альтернативы не показаны | Проверить, что learner state содержит данные по другим режимам (quiz history, tutor sessions) |
| Routing decision кажется неправильным | Использовать feedback кнопку "Это не помогло" — система учтёт в следующий раз |

### Power user tip

Evidence Ledger доступен через API: `GET /ssr/evidence` возвращает полный routing trace в JSON — полезно для debugging и понимания системы.

## Сценарий 24 — SSR Pedagogy: Управляемая педагогика
> *«Роутер предлагает повторить карточки. Но это для закрепления старого или для восстановления забытого?»*

### Контекст

SSR Trust Level объясняет **почему** выбрано действие. SSR Pedagogy Level объясняет **зачем** — какую педагогическую цель преследует это действие. Система различает три типа учебных действий: **retention debt** (повторение для закрепления), **recovery** (восстановление слабого концепта), **new learning** (изучение нового). Пользователь может управлять приоритетами с видимыми tradeoffs.

**Для кого:** Аня, готовится к экзамену, хочет контролировать баланс между повторением и новым материалом.
**Время:** 3 минуты.
**Главный вопрос:** «Это действие для закрепления, восстановления или изучения нового?»

### Шаги

1. **Открыть SSR-карточку** — увидеть рекомендацию с педагогической меткой.

2. **Прочитать Pedagogy Label:**
   - 🔄 **Retention Debt:** "Повтори 12 карточек — закрепи то, что уже знаешь"
   - 🔧 **Recovery:** "Разбери слабый концепт 'Backpropagation' — восстанови пробел"
   - 🆕 **New Learning:** "Начни новую тему 'Gradient Descent' — продолжи обучение"

3. **Раскрыть Pedagogy Explanation:**
   - **Retention Debt:** "Эти карточки созрели по SM-2. Если не повторить сегодня, забудешь через 2-3 дня."
   - **Recovery:** "Этот концепт показал mastery < 50% в последних 2 quiz. Нужно восстановить понимание."
   - **New Learning:** "Все retention debt закрыт, пробелов нет — можно двигаться дальше."

4. **Увидеть Learner Steering Controls:**
   - Slider: "Приоритет retention vs new learning" (по умолчанию: retention first)
   - Toggle: "Показывать recovery перед new learning" (по умолчанию: включено)
   - Visible Tradeoffs: "Если сдвинуть slider в сторону new learning, retention debt вырастет на 5-7 карточек за 3 дня"

5. **Настроить приоритеты** под свою цель:
   - Перед экзамен: retention + recovery, new learning отложить
   - Обычный режим: balanced (retention → recovery → new learning)
   - Быстрое освоение: new learning first, retention по минимуму

6. **Применить настройки** — SSR обновляет рекомендацию с учётом новых приоритетов.

### Под капотом — Pedagogy Router

**Pedagogy Router** классифицирует каждое действие по трём категориям:

```python
@dataclass
class PedagogyAction:
    action: str  # "cards_due" | "quiz_failed" | "tutor_new_topic"
    pedagogy_type: str  # "retention_debt" | "recovery" | "new_learning"
    explanation: str  # Почему это важно сейчас
    tradeoff: str  # Что будет, если отложить
    priority_score: float  # 0.0-1.0, зависит от learner steering
```

**Learner Steering:**
- Пользователь может сдвинуть приоритеты через UI controls
- Система показывает **visible tradeoffs**: "Если отложить retention, debt вырастет на X карточек"
- Steering не отменяет критические сигналы (overdue > 20 карточек всегда приоритет)

**Три педагогических режима:**

| Режим | Приоритет | Для кого |
|---|---|---|
| **Exam Prep** | Retention + Recovery | Подготовка к экзамену, закрепление |
| **Balanced** | Retention → Recovery → New | Обычный режим обучения |
| **Fast Track** | New Learning first | Быстрое освоение нового материала |

### Wow-момент

Система не просто говорит «делай это» — она объясняет **педагогическую цель** действия и позволяет управлять балансом между закреплением, восстановлением и новым обучением с видимыми последствиями.

### Сигнал успеха

**Количественные метрики:**
- Pedagogy label видна на SSR-карточке без раскрытия
- Tradeoff calculation <100ms
- Steering controls обновляют рекомендацию <500ms

**Качественные метрики:**
- "Понимаю, зачем система предлагает это действие"
- "Могу управлять балансом retention vs new learning"
- "Вижу последствия своего выбора до того, как сделаю его"

### Если что-то пошло не так

| Симптом | Решение |
|---|---|
| Tradeoff кажется неточным | Система оценивает tradeoff на основе исторических данных — точность растёт после 2-3 недель использования |
| Steering не меняет рекомендацию | Проверить, что нет критических сигналов (overdue > 20) — они переопределяют steering |
| Pedagogy label неясен | Раскрыть Pedagogy Explanation для детального объяснения |

### Power user tip

Pedagogy Router доступен через API: `GET /ssr/pedagogy` возвращает классификацию всех доступных действий по pedagogy_type — полезно для планирования учебной стратегии.

## Сценарий 13 — Course Workspace: полный цикл
> *«Добавил 5 новых лекций. Боюсь, что потеряю весь прогресс.»*

### Контекст

Марк добавил новые материалы в `data/`. Его страх понятен: в большинстве систем «пересобрать индекс» означает «начать сначала». В hometutor reindex — это **incremental**: новые файлы добавляются, уже проиндексированные пересчитываются только при изменении. Learner profile не трогается.

**Для кого:** Марк, хочет добавить материалы без потери обучения.  
**Время:** 4 минуты (+ время индексации).

### Шаги

1. **Добавить новые файлы в `data/`.**

2. **Запустить:**
   ```bash
   python ingest.py
   ```
   Для неизменных файлов — `run_kind=noop`, они пропускаются. Только новые файлы обрабатываются.

3. **Открыть «Мой прогресс»** → проверить бейдж «Профиль обновлён».

4. **Открыть план** → новые концепты из новых файлов появились в блоке New.

### Под капотом

- Manifest diff: ingest сравнивает hash файлов с прошлым запуском — `Document`-объекты кэшируются в `chroma_db/ingestion_extracted_documents.json`.
- Mastery, история сессий, flashcards — хранятся в отдельной SQLite, не в Chroma. Reindex не трогает SQLite.

### Сигнал успеха

Mastery прежних концептов не изменилась. Новые концепты из новых файлов появились в Adaptive Plan в блоке New.

## Сценарий 14 — Полный export/sync и восстановление
> *«Всё моё обучение — в одном месте. Хочу уметь перенести его и работать без интернета.»*

### Контекст

Елена работает с конфиденциальными материалами. Она хочет: никакого облака, никакого account lock-in, полный контроль над данными. hometutor — local-first по архитектуре: все данные на локальном диске, LLM может быть локальным через Ollama. Backup — это один JSON-файл, который переносит всё состояние обучения.

**Для кого:** Елена, конфиденциальные материалы, хочет полный контроль.  
**Время:** 5 минут (настройка offline), backup — 30 секунд.

### Шаги

#### Полный офлайн

1. **Установить Ollama:**
   ```bash
   ollama pull llama3.2
   ollama pull nomic-embed-text
   ```

2. **Настроить `.env`:**
   ```
   LLM_PROVIDER=ollama
   OLLAMA_MODEL=llama3.2
   EMBED_PROVIDER=ollama
   ```

3. **Отключить интернет → задать вопрос** → ответ приходит от локальной модели.

#### Backup и restore

1. **Backup:** UI → «Backup my learning» → скачать `learning_backup.json`.

2. **На новой машине:** установить продукт → UI → «Restore» → загрузить JSON.

### Что входит в backup

| Данные | Включены |
|---|---|
| Mastery по концептам | ✅ |
| Flashcard decks + карточки | ✅ |
| История tutor-сессий | ✅ |
| Adaptive plan snapshot | ✅ |
| Документы (`data/`) | ❌ переносить вручную |
| Chroma-индекс | ❌ пересобирается через `ingest.py` |

### Сигнал успеха

Air-gapped: вопрос → ответ с источниками → тьютор → прогресс — всё без интернета. Backup/restore: mastery не изменился после переноса.
