## 📝 Текст выступления по слайдам

### 🎯 Слайд 1 — Проблема и решение (30 сек)

Добрый день! Представляю проект **home-rag_v2** — персональный учебный ассистент на основе ваших материалов.

**Проблема знакома каждому студенту:** у вас папка с лекциями, статьями, заметками. Вы хотите быстро найти ответ, разобраться в теме и не забыть через неделю. Но приходится переключаться между поиском по файлам, ChatGPT, который не знает ваших материалов, и ручным созданием карточек.

**Решение — один инструмент, полный цикл, локально.** От вопроса до освоения: индексация → ответ с источниками → разбор через диалог → квиз → интервальное повторение → отслеживание прогресса.

Ключевой принцип — **local-first**: всё на вашей машине, полный offline возможен.

---

### 🏗️ Слайд 2 — Архитектура (20 сек)

Архитектура — четыре слоя:
- **Интерфейсы:** Streamlit UI, CLI, Telegram бот
- **API:** FastAPI на порту 8000
- **Сервисы:** обработка запросов, учебная оркестрация, граф концепций
- **Persistence:** SQLite для прогресса, Chroma для векторного поиска

Ключевые решения: local-first инвариант, единая точка для LLM, типизированный pipeline, guardrails на всех входах.

---

### 🔍 Слайд 3 — RAG Pipeline (50 сек)

RAG — технология, позволяющая LLM отвечать на основе ваших документов.

**Ingest:** документы парсятся, разбиваются на чанки по 700 токенов, обогащаются метаданными через LLM, векторизуются и сохраняются в две коллекции — для документов и фрагментов. Плюс строится граф концепций.

**Query Pipeline — 5 ступеней:**
1. **Classify** — определяем тип запроса
2. **Rewrite** — улучшаем для поиска
3. **Retrieve** — гибридный поиск: BM25 + векторный
4. **Rerank** — cross-encoder оценивает релевантность, опциональная self-correction
5. **Generate** — LLM создаёт ответ с атрибуцией источников

Уникально: двухуровневая индексация, multi-prompt архитектура, self-correction loop, граф концепций.

Качество — трёхслойно: детерминированные проверки, LLM-as-Judge, человеческий фидбек.

---

### 🎓 Слайд 4 — Учебный цикл (25 сек)

Как это работает:

**Шаг 1:** Задаёте вопрос — получаете ответ с тремя кликабельными источниками и retrieval confidence chip.

**Шаг 2:** Нажимаете "Учить эту тему 5 минут" — запускается квиз. Система генерирует вопросы по теме из предыдущего ответа. Контекст передаётся без разрыва.

После квиза создаётся flashcard и обновляется mastery score. Один клик от ответа до проверки знания.

### 📊 Слайд 5 — Прогресс (20 сек)

**Режим 1 — Слабые места:** визуализация концептов по трём уровням — recognition, recall, transfer. Концепты со стабильным transfer получают статус graduated и выпадают из повторений. Видите mastery timeline и streak.

**Режим 2 — Course Workspace:** активируете папку как курс с изолированным прогрессом. Можно учить несколько курсов параллельно.

### 🧭 Слайд 6 — Smart Study Router (35 сек)

Это прорывная фича. **Проблема:** слишком много режимов — вы теряетесь, что делать дальше.

**Решение:** система анализирует ваше состояние и предлагает лучшее действие с объяснением.

Семь сценариев: карточки к повторению, просроченные темы, ошибки в квизе, свежий ответ, давно не было проверки, адаптивный план, незавершённая сессия.

Роутер показывает основную кнопку и альтернативы. **Не вы ищете режим — система ведёт вас.**

Это превращает продукт из "набора функций" в "учебного проводника".

### 🖥️ Слайд 7 — Local vs Cloud (20 сек)

Проект поддерживает оба варианта.

**Local LLM через Ollama:** данные не покидают машину, нулевая стоимость API, работает offline. Требует 8+ GB RAM.

**Cloud LLM:** высокое качество, быстрая генерация, но данные передаются провайдеру и есть стоимость.

Переключение — через `.env` без изменения кода. Возможен гибрид: embeddings локально, генерация в облаке.

### 🔄 Слайд 8 — Процесс разработки (30 сек)

Интересная часть — как разрабатывался сам проект.

**Было:** каждый пакет — 6 документов, 8 команд вручную, 2-3 часа, высокий риск пропустить шаг.

**Стало:** Smart Router + Cursor SDK = автопилот. Одна команда запускает весь конвейер от контракта до закрытия с проверкой DoD.

Роутер анализирует сложность: лёгкий пакет — прямо в код, тяжёлый — полный конвейер с 8 ролями.

**Результат:** 15+ ручных шагов → 1 (принять контракт). Время: 2-3 часа → 15-75 минут.

## ❓ Подготовка к вопросам

### Ожидаемые вопросы и ответы

**1. "Как измеряете качество RAG?"**
> Трёхслойно: детерминированные проверки каждый запрос, LLM-as-Judge опционально для Faithfulness и Answer Relevancy, человеческий фидбек через кнопки в UI. Есть eval dataset и regression detection.

**2. "Почему не использовали NotebookLM?"**
> NotebookLM — только Q&A по документам, нет учебного цикла. Нам нужен полный цикл от вопроса до mastery: tutor, quiz, SRS, adaptive plan.

**3. "Ограничения локальных LLM?"**
> Качество ниже cloud frontier-моделей, требуют 8+ GB RAM, медленнее на CPU. Но зато приватность и нулевая API-стоимость. Для учебных задач достаточно.

**4. "Как доказываете эффективность обучения?"**
> Сейчас есть инфраструктура для измерений: mastery tracking, retention metrics, quiz scores. Следующий шаг — longitudinal study с контрольной группой для научного доказательства.

**5. "Стоимость индексации?"**
> ~$0.002 на документ при cloud LLM (summary + enrichment). При Ollama — бесплатно. Для корпуса в 100 документов — $0.20 или 0 ₽.

**6. "Масштабируется ли на большие корпуса?"**
> Рассчитано на локальные корпуса с профилированием bottleneck'ов. Есть partial reindex и кеши. Ограничения: BM25 in-memory, RAM для embeddings. Для тысяч документов нужна оптимизация.

**7. "Почему не multi-user?"**
> Фокус на local-first для одного пользователя. Multi-user требует отдельной инфраструктуры: auth, ACL, cloud state. Это следующий этап развития.

**8. "Сравнение с Anki?"**
> Anki — лучший для SRS, но нет Q&A по документам и автоматического создания карточек из RAG-ответов. Мы интегрируем RAG + SRS в один цикл.
