Часть 1 · Суть

Чего по-настоящему хочет студент, когда система под нагрузкой

Не «быстрого ответа» и не «стопроцентного аптайма» — эти вещи локальный стек честно не может обещать. Студент хочет предсказуемости деградации: если что-то не работает как обычно, он должен узнать об этом за секунды, а не после полной выдержки таймаута в тишине. Доверие под нагрузкой — это не про то, что система никогда не подведёт, а про то, что она никогда не подведёт молча.

Отсюда первый принцип: деградация — это интерфейс, а не побочный эффект. Второй принцип, уже открытый предыдущими разборами серии, находит здесь свой корень: molчаливые сбои LM Studio ломали первый ответ (разбор №2) и публикацию карты (разбор №3) по одной и той же причине. Этот разбор ищет саму причину, а не очередное её проявление.

Часть 2 · Реальность

Три профиля, один предохранитель — и он не подключён к сети

Инвентаризация provider-слоя показывает продуманную архитектуру: три явных профиля маршрутизации (local_strict / balanced / cloud_fast), circuit breaker с auto half-open, честный источник-бейдж с privacy-notice для облака. Но у самого нагруженного пути — primary chat — предохранитель ни разу не запитан от реального трафика.

llm_local_circuit 3 сбоя/30с → open на 60с → auto half-open механизм верный; runtime-питание отсутствует для primary chat

Что уже образцово — не трогать

Три профиля продуманы верно: local_strict честно отказывает вместо тихой деградации, cloud_fast прозрачен для demo-режима. Честный бейдж источника (llm_source_badge_text, llm_source_privacy_notice) — не за экспертным тиром, виден по умолчанию, прямо отвечает на «откуда этот ответ» — редкая прозрачность. Adaptive-plan explanation — образцовая реализация circuit breaker с полным циклом обратной связи, ровно то, чего не хватает primary chat. Auto half-open в самом circuit breaker — по-настоящему хорошая инженерия, просто не подключённая к нужному источнику сигнала.

Часть 3 · Противоречия

Три напряжения — и синтез для каждого

Local-first ↔ мгновенная отзывчивость

Локальная модель — правильный выбор по приватности и офлайн-автономии, но она честно не может быть мгновенной на слабом железе. Синтез уже в коде: профиль balanced + опциональный cloud-fallback — снаружи выбор не «локально или облако», а «локально, с явно согласованной страховкой». Ничего не менять в этом выборе — почему он не срабатывает, разобрано в Боли-якоре.

Наблюдаемость для разработчика ↔ честность для студента

Латентность, источник LLM, circuit state — всё это существует как измеримые сигналы. Бейдж источника уже сделал правильный выбор: показывать по умолчанию, не прятать за debug-тиром. Синтез: тот же принцип должен распространиться на Tutor chat — сейчас бейдж есть только в «Быстром ответе», хотя данные для него собираются для любого ответа единым кодом (_build_generation_model_and_metadata).

Простой предохранитель ↔ полная петля обратной связи

Circuit breaker спроектирован просто и без блокировок (намеренно process-local, lock-free) — это правильно для UI hot path. Но простота реализации не должна означать «подключен только к одному источнику сигналов». Синтез: не переусложнять breaker — просто дать ему второй источник данных (реальные сбои `llm_resilience.py`), тем же интерфейсом `record_failure`/`record_success`, что уже используется в `adaptive_plan_llm_explanation.py`.

Часть 4 · Боль-якорь

Предохранитель, который никогда не видит реальную проводку

Точная причина в коде, не в теории

LM Studio падает через 10 минут после старта Streamlit. Приложение продолжает работать — но каждый вопрос студента теперь молча ждёт полный таймаут:

# Единственные записи в circuit breaker для primary chat — api.py:135-170,
# запускаются РОВНО ОДИН РАЗ при старте процесса (daemon-thread):
result = probe_local_llm(base, model, …)
if result.get("reachable"): record_success(base)
else: record_failure(base, error_type="startup_probe_failed")

# provider_openai.py и llm_resilience.py — реальные HTTP-вызовы чата —
# НИ РАЗУ не упоминают circuit ни в одной ветке (grep "circuit" → 0 строк):
grep -n "circuit" app/provider_openai.py app/llm_resilience.py → (пусто)

# Итог: is_open(local_base) остаётся тем же, каким был при старте,
# сколько бы реальных таймаутов ни случилось за время работы процесса.

Даже если бы circuit открылся — например, стартовая проба случайно упала раньше, чем LM Studio успела подняться — поведение get_llm() в дефолтной конфигурации (balanced + HOME_RAG_LLM_FALLBACK_ENABLED=false, оба — tracked-дефолты config.env) не меняется: код логирует balanced_primary_chat_circuit_open_no_fallback_using_local_attempt и всё равно строит OpenAI(api_base=local_base_norm, …) — ту же самую мёртвую точку. Предохранитель существует, но в дефолтной поставке он либо не видит проблему (нет runtime-питания), либо видит и ничего не делает (нет включённого fallback).

Контраст рядом, в том же коммите: adaptive_plan_llm_explanation.py делает ровно то, чего не хватает primary chat — проверяет is_open, оборачивает попытку, вызывает record_failure/record_success по факту. Решение уже написано, просто не в том файле.

Часть 6 · Ценность

Честная деградация — измеримое доказательство

Уникальность hometutor здесь — не «локальная модель работает быстро» (это не всегда правда), а студент никогда не остаётся в неведении дольше одного запроса. Ни ChatGPT, ни облачные RAG-сервисы не показывают, откуда пришёл ответ и почему он задержался — у hometutor эта честность уже частично построена. Метрика: доля запросов во время реального сбоя local endpoint, которые получают ответ (fallback или честный отказ) быстрее полного hard-таймаута.

СценарийСейчасПосле P0
LM Studio падает после старта, local_strict2-й запрос ждёт полный timeoutмгновенный честный отказ после 3 сбоев
LM Studio падает, balanced, fallback выключенкаждый запрос — полный timeout (до 45с)быстрый фейл после порога, без ложного ожидания
Источник ответа виден студентутолько «Быстрый ответ»+ Tutor chat

North star: после первых 3 сбоев подряд ни один следующий запрос в той же сессии не ждёт больше нескольких секунд перед честным сообщением — вне зависимости от того, включён ли cloud-fallback.

Часть 7 · Путь

Один золотой путь и лестница уровней

Локальная модель работает → студент получает ответ с честным бейджем источника → модель падает (крашнулась/выгружена) → первые 1-3 запроса замечают реальный сбой → предохранитель размыкается → следующие запросы получают быстрый честный ответ (fallback, если настроен, или ясное «сейчас недоступно») вместо тишины на 45 секунд → модель возвращается → auto half-open проверяет и молча восстанавливает нормальный режим.

  • L0Ответ приходит — студент видит источник и задержку одной строкой, не только в «Быстром ответе»
  • L1Локальный сбой — первые запросы честно фиксируют проблему (runtime record_failure)
  • L2Предохранитель разомкнут — следующие запросы не ждут впустую
  • L3Восстановление — auto half-open проверяет и возвращает local без вмешательства студента
  • L4Владелец видит состояние breaker в диагностике — не только в JSONL-логе, а в понятной панели
Часть 8 · План

P0–P2 по соотношению «эффект / усилие»

P0 · ХОД 1

Runtime record_failure/record_success для primary chat

Обернуть реальные HTTP-вызовы в llm_resilience.complete_with_resilience/ chat_with_resilience тем же паттерном, что уже работает в adaptive_plan_llm_explanation.py: успех → record_success, ошибка соединения → record_failure. Предохранитель наконец получает питание от реального трафика, а не только от одноразовой стартовой пробы.

P0 · ХОД 2

Быстрый честный отказ при открытом CB без fallback

В get_llm(): при balanced + cb_now + not fallback_ready — не тихо пытаться local endpoint с полным таймаутом, а использовать короткий read-timeout для этой попытки (или сразу честную ошибку с понятным сообщением) вместо штатного hard_timeout_sec. Не включает fallback насильно — это решение остаётся за владельцем; сокращает только бесполезное молчаливое ожидание.

P1

Бейдж источника в Tutor chat

llm_source_badge_text/llm_source_privacy_notice подключаются к рендеру tutor-ответа — те же helper'ы, тот же принцип «видно по умолчанию, не за debug-тиром», данные уже собираются единым кодом для любого query_mode.

P2

Панель состояния breaker для владельца

llm_local_circuit.snapshot() уже существует для диагностики — вывести его на существующую debug/metrics поверхность человеческим языком («локальный endpoint: открыт N секунд назад, N сбоев»), не только в JSONL.

Планка неизменна: гениальность — в простоте. P0 не меняет архитектуру профилей и не создаёт новый механизм — переносит уже написанный, работающий паттерн из одного файла (adaptive_plan_llm_explanation.py) в другой (llm_resilience.py), где он нужнее всего.
