🔥 §3

Боль-якорь (E1): подтверждённый разрыв контроля

🚨
E1 — Разрыв между промптом и парсером
QUIZ_SCOPED_PROMPT требует качественных вопросов с привязкой к источнику, но parse_scoped_quiz_json принимает любой структурно валидный JSON без проверки содержательного качества. После успешного парсинга — немедленная запись в SM-2 и quiz_mastery.
✅ Что QUIZ_SCOPED_PROMPT требует
  • Только на основе source material
  • Микс recognition / recall / transfer
  • Один правильный ответ
  • Explanation для каждого вопроса
🔍 Что parse_scoped_quiz_json проверяет
  • Количество вопросов (5-8)
  • Непустой текст вопроса
  • Ровно 4 варианта ответа
  • correct_index ∈ 0..3
  • difficulty ∈ {recognition, recall, transfer}
  • explanation (непустая строка)
❌ Что парсер НЕ проверяет
🔴 Span matching — что вопрос следует из source scope
🔴 Что correct_index действительно верен (нет verification)
🔴 Что остальные 3 варианта действительно неверны
🔴 Что explanation подтверждён материалом
🔴 Что вопрос с transfer реально требует transfer thinking
🔴 Дублирование вопроса внутри одного набора
⚠️
Fallback Micro Quiz: структурно валиден, содержательно невалиден
_fallback_micro_quiz(topic, band) всегда возвращает «B» как правильный ответ с шаблонным вопросом «Рабочее определение и типичный сценарий применения». Ответ не связан с реальным материалом, но process_micro_quiz_outcome() немедленно записывает результат в SM-2 и quiz_mastery.

📋 Rubric v1 (8 пунктов, подготовлен до будущего live-прогона)
#КритерийТипВес
1Stem grounding — вопрос подтверждён конкретным source spanHard FailFatal
2Correct-answer grounding — правильный вариант подтверждёнHard FailFatal
3Uniqueness — нет второго допустимого правильного ответаHard FailFatal
4Distractor validity — дистракторы правдоподобны, но опровержимыSoft0–1
5Explanation grounding — объяснение следует из материалаSoft0–1
6Cognitive level — independent classification matches claimed levelSoft0–1
7Clarity — вопрос самодостаточен, не раскрывает ответSoft0–1
8Duplication — не дублирует другой вопросSoft0–1
📉 Гипотезы для проверки — не измеренные частоты
🔴 Stem grounding — промпт не имеет enforcement механизма не измерено
🟠 Uniqueness — нет требования доказать неверность остальных вариантов не измерено
🔴 Cognitive level — нет enforcement, что transfer реально требует transfer не измерено
🟠 Distractor validity — нет инструкций по опровержимости не измерено
⚡ Fallback micro quiz — B=correct, тема не связана с материалом, структурно valid не source-grounded по конструкции
📌 Что требуется для закрытия E2

Живой семпл: LLM endpoint + 5 scope × 3 повтора = 15 генераций с оценкой каждого вопроса. Raw outputs сохраняются полностью; ключевые hard fail проверяются человеком или независимым evaluator.

Противоречия и синтезы

⚡ Скорость генерации
Быстрый quiz в realtime без дополнительной проверки
vs
🔬 Доказуемое качество
LLM-судья или content gate перед mastery write
Синтез: Детерминированный gate (наборы правил, не LLM) before mastery write. LLM eval только для CI/offline семплов.
🧠 Проверка понимания
Transfer-level вопросы требуют настоящего когнитивного усилия
vs
📚 Проверка запоминания
Recognition вопросы надёжнее для comparison
Синтез: difficulty label + независимая классификация после генерации. Не блокировать, но логировать расхождение.
🎯 Qualitative distractors
Дистракторы должны быть правдоподобны для педагогической ценности
vs
✅ Однозначный ответ
Только один вариант должен быть объективно правильным
Синтез: Добавить в промпт «для каждого дистрактора укажи почему он неверен» — без LLM-judge, только prompt engineering.
📈 Адаптивная сложность
Mastery должна быть актуальной для adaptive quiz generation
vs
📊 Сопоставимость mastery
Mastery сопоставима только если вопросы одного качества
Синтез: Mastery обновляется только после validated вопроса. Fallback не обновляет mastery.
🤖 Детерминированный gate
Правила без LLM — быстро, надёжно, без latency
vs
🧩 LLM-судья
LLM лучше оценивает семантическое качество
Синтез: Для P0 только детерминированные правила. LLM-judge → P1/CI только, не realtime.
🔒 Strict fail-closed
Блокировать показ и mastery при любом quality fail
vs
🏠 Local-first доступность
Система должна работать в offline режиме без деградации UX
Синтез: Confidence tier: validated / limited / fallback. Показывать всегда, но mastery update только для validated.
🚫 §10

НЕ делать

  • Не запускать LLM-судью на каждом пользовательском вопросе в realtime без доказанного эффекта и приемлемой latency
  • Не использовать самооценку той же модели как единственное доказательство качества
  • Не строить universal quiz quality framework — достаточно минимального contract + проверяемого evidence binding
  • Не менять mastery/persistence до подтверждения причинной цепочки на живом семпле
  • Не добавлять новую DB-схему только ради eval — использовать существующие quiz_results + metrics_db
  • Не объявлять PAIN-04 (две линии производства) подтверждённой — пути расходятся по промпту/схеме, но mastery pipeline един
  • Не блокировать показ quiz при fail content gate — показывать с confidence label, блокировать только mastery write
  • Не делать pop-рефакторинг смежных модулей в рамках этого write-set
