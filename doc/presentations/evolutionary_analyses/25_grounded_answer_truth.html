⚖️ §1

Суть — пользовательский контракт

💬
Контракт одним предложением
«Каждое существенное утверждение ответа либо доказуемо следует из конкретного фрагмента учебного материала, который можно открыть и проверить, либо явно маркируется как недостаточно подкреплённое; если данных в корпусе нет — система говорит это прямо.»
✅ Что студент видит сейчас
  • Ответ с inline-цитатами [1], [2,3]
  • Карточки источников с файловым путём
  • Иногда: «В доступных материалах недостаточно информации…»
  • debug.citation_coverage в API-ответе
❌ Чего студент не может проверить
  • Действительно ли утверждение следует из указанного чанка
  • Сохранены ли числа, единицы, отрицания точно
  • Не создал ли синтез нескольких источников новый непроверяемый факт
  • Прошёл ли ответ ту же проверку, что cache-miss
⚠️
Центральный разрыв разбора
citation validity ≠ evidence binding ≠ semantic entailment. Текущая система гарантирует, что [2] указывает на существующий источник в списке, но не гарантирует, что текст чанка источника поддерживает конкретное утверждение рядом с маркером. Провенанс-ледже не содержит текст чанка рядом с claim, но тот же API-ответ уже несёт sources[*].text. Offline join возможен по cite_index; live-path его не выполняет.
🔍 Уточнение области

№5 исследовал устойчивость доставки ответа. №10 — видимость trace. №11 — сценарии использования. №24 — честность mastery и качество квизов. Впервые: end-to-end связь «утверждение ответа → цитата → реальный текст источника».

🗺️ §2

Реальность — аудит end-to-end

ℹ️
Уровень доказательности
Все факты в §2 — уровень E1: перепроверены в коде на runtime HEAD 1c9c56961. Номера строк указаны для навигации; функциональный anchor важнее номера строки.
Вопрос → Retrieval (hybrid BM25 + vector, Chroma)
Приносит релевантные чанки в context window. Возвращает source-объекты с полями relative_path, cite_index, text. Гарантирует: релевантность по сходству. Не гарантирует: что топ-чанк содержит конкретное утверждение, которое генерирует модель.
Generation (LLM) → inline citations [N]
Модель добавляет маркеры [1], [2,3] в текст. Канонические app.prompts._impl.QA_PROMPT и SYSTEM_RULES инструктируют цитировать источники; отдельная revision-метка в коде не подтверждена. Гарантирует: наличие маркеров по инструкции. Не гарантирует: что процитированный чанк действительно поддерживает утверждение — это решение модели, а не верифицированный факт.
normalize_inline_numeric_citations
app/query_response_postprocessing.py: заменяет [5] → [1] если source_count == 4. Исправляет выход за границы диапазона. Гарантирует: индекс в диапазоне 1..len(sources). Не гарантирует: что замена выбрала верный источник. Семантики нет.
_build_facts_from_text (grounded_answer.py:183)
Сегментирует ответ на блоки (параграфы / длинные предложения). Для каждого блока парсит маркеры [N] → CitationProvenance{cite_index, relative_path}. Если нет маркеров — missing_provenance. Если маркер вне диапазона — invalid_cite_index:N. Гарантирует: структурную валидность провенанса. Не гарантирует: что текст блока семантически следует из чанка по relative_path.
build_provenance_ledger (grounded_answer.py:218) — КЛЮЧЕВОЙ РАЗРЫВ
Строит список записей {fact_text, cite_index, relative_path, provenance_type}. Отсутствует: co-located поле chunk_text в ledger. Однако API source уже содержит evidence excerpt, поэтому offline-eval может соединить ledger и sources без запроса к индексу; runtime semantic comparison отсутствует.
apply_grounded_validation (grounded_answer.py:461) — CACHE-HIT BYPASS
Первая строка функции: if not settings.grounded_answer_contract_enabled or cache_hit: return GroundedValidationResult(answer_text=answer_text, skipped=True)
Все cache-hit ответы возвращаются без провенанс-проверки. Дополнительные bypass: answer_path_mode == "two_stage_early" (строка 471), not sources (строка 464), guardrail violation (строка 476–482).
_citation_coverage (grounded_answer.py:247)
Вычисляет долю fact-блоков с хотя бы одной provenance: cited/total. Добавляется в debug.citation_coverage в API-ответе. Гарантирует: метрика полноты цитирования доступна операторам. Не гарантирует: семантического entailment; не видна в UI студенту.
home_rag_integration_gate_v1 → CI
15 тест-кейсов в eval_data/home_rag_gate/home_rag_cases_v1.json. Проверяет: keyword presence в ответе, source paths, citation index в bounds, no-citation для OOC случаев. Gate запускается с GROUNDED_ANSWER_STRICT_QA=0 (строка 184 скрипта) — намеренно relaxed для integration тестирования. Не в CI: .github/workflows/ci.yml запускает ruff, pytest, arch_regression_guards. home_rag_integration_gate_v1.py не упоминается.

🔑
Главный E1-факт разбора
Source-объект возвращаемый retrieval содержит поле text (текст чанка). Функция _sources_text() в gate-скрипте это использует. Но build_provenance_ledger() (grounded_answer.py:218–244) строит записи из fact.provenance — цепочки CitationProvenance, которые содержат только cite_index и relative_path. Текст чанка при построении ledger-записи не передаётся и не сохраняется. Пара (утверждение, текст-источника) доступна как две структуры одного API-ответа и может быть соединена по cite_index, но runtime не выполняет ни этот join, ни semantic entailment. Это и есть подтверждённый gap.
⚡ §3

Противоречия

Полезный связный ответ
Студент хочет объяснение, а не список атомарных фактов с отдельными цитатами на каждую фразу.
VS
Атомарно проверяемые утверждения
Trustworthy ответ требует, чтобы каждое существенное утверждение можно было сопоставить с источником.
Синтез: Claim-level granularity нужна только для существенных утверждений. Вводные фразы, примеры и связки не требуют провенанса. Правило: «если утверждение можно запомнить и воспроизвести — оно существенно и требует цитаты».
Дешёвый детерминированный gate
Citation index check, keyword presence — работают без LLM, быстро, в CI.
VS
Семантическая оценка entailment
Настоящий entailment требует LLM-judge, который дорог, медленен и циркулярен для той же модели.
Синтез: Детерминированный gate — в P0 и CI. LLM-judge — только offline, только как proxy, явно помечен. Разделить инфраструктуру: structural CI gate + offline semantic eval sample. Не смешивать.
Высокая полнота ответа
Студент хочет полный охват темы. Strict abstention при недостатке данных снижает полноту.
VS
Честный отказ вне корпуса
Ответ с фактической ошибкой хуже отказа. Особенно критично в учебном контексте.
Синтез: AbstainResponse + is_abstain_phrase уже реализованы корректно. Приоритет: не тратить P0 на abstention — оно работает. Следующий шаг: валидировать abstention качество на offline-семпле (P1).
Скорость и cache
LLM_REQUEST_CACHE_PERSIST=true по умолчанию. Cache-hit экономит время и деньги.
VS
Единый контракт groundedness
Cache-hit ответ обходит всю grounded-validation. Разный контракт для одинакового вопроса.
Синтез: Повторная LLM-генерация не нужна, но простой флаг skip недостаточен. Cache entry должен доказывать ревизию grounded-контракта и validation outcome; stale/legacy entry revalidate детерминированно либо становится miss. Точный ход gated E2 и аудитом cache payload.
Фиксированный regression corpus
15 кейсов home_rag_gate с синтетическим корпусом — стабильны и воспроизводимы.
VS
Живой пользовательский курс
Реальный студент учится на своём data/. Синтетический корпус не покрывает специфику курса.
Синтез: Два отдельных layer: (1) синтетический gate для structural regression в CI, (2) живой E2 sample из реального data/ для semantic eval. Не заменять один другим.
Автоматическая оценка
Полностью автоматизированный eval pipeline — воспроизводим, не требует человека.
VS
Независимая человеческая разметка
Только человек может окончательно определить, является ли утверждение entailed из источника.
Синтез: Автоматика для structural проверок (всегда). LLM-judge как дешёвый proxy для offline batch. Человек — для валидации рубрики и calibration judge-а. Не заменять человека автоматикой при первом E2 замере.

🏆 §7

Золотой путь студента и лестница качества

🎓
Золотой путь — один прогон студента
Студент задаёт вопрос по своему курсу → получает короткий полезный ответ с inline-цитатами → видит проверяемые источники для каждого существенного утверждения → открывает конкретный фрагмент учебного материала и может сам проверить → если данных нет — система явно сообщает об этом и предлагает следующий полезный шаг → при повторном вопросе (cache-hit) результат семантически не отличается от первого прогона.

Лестница качества (наблюдаемые уровни)

L1
Структурная валидность — citation validity
Citation index в bounds. CitationProvenance с relative_path. Abstain при zero facts. Текущее состояние.
сейчас
L2
Course-specific evidence packet
Offline runner соединяет существующие ledger и API sources по cite_index, сохраняет raw output и разметку. Публичный payload не расширяется.
P0-A
L3
Cache contract revision parity
Current/stale/legacy cache entries различаются; устаревший entry проходит deterministic revalidation либо становится cache miss.
P0-B
L4
Deterministic evidence checks — не SGAR
Citation resolution, coverage и точность чисел проверяются автоматически. Token overlap не считается entailment и не входит в числитель SGAR.
P1
L5
Operator gate — home_rag_gate в протоколе
home_rag_integration_gate_v1 обязателен перед реиндексом и сменой модели/промпта. Задокументировано в operator runbook. 15 кейсов покрывают структурный regression.
P1
L6
Offline LLM-judge — semantic entailment proxy
LLM-judge оценивает рубрику R3/R5 на E2 sample. Результат — первый SGAR baseline. Judge той же модели помечен как proxy, не ground truth.
P2
