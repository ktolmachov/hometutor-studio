1Суть

Чего по-настоящему хочет разработчик, отдавая задачу модели

Не «локальная модель в IDE» — а дешёвый, контролируемый и доказуемый способ превращать часть плана в рабочий код. Ценность локальной модели не в бесплатности токенов вообще, а в дешёвых draft-токенах внутри контура, где границы держит не модель, а детерминированный код вокруг неё. Облачные токены дороги на массовую генерацию, но дёшевы на компиляцию задач и ревью диффов; локальные токены бесплатны, но по умолчанию недоверенны. Оптимум: облако думает и проверяет (мало токенов, высокая ставка), локальная модель печатает (много токенов, низкая ставка), а границы держит код, у которого ставка — ноль. Сегодняшняя находка: этот контур уже физически существует и работает за секунды — разбор фиксирует, где именно он сейчас доверяет не тому и не тем.

Три реальных входа к локальной модели — и где живёт инверсия доверия

Три разных клиента ведут не к одному общему runtime (правка 2026-07-22: исходная формулировка «делят один production-runtime» была неточна — видно уже из таблицы ниже): primary фиксированно использует qwen3-coder-next-q4ks через llama.cpp; A шлёт задачу в облачный CURSOR_MODEL (дефолт composer-2.5) — совсем другую модель; B маршрутизируется через релей на выбираемый upstream и получает qwen3-coder-next-q4ks только при одной конкретной конфигурации. Общее — разные trust-модели, не общий runtime. Инверсия доверия §4 — свойство только primary local-executor path; два клиентских входа A/B её не наследуют, но и не чинят — у них своя.

A — не HTTP в relay. cursor_agent_trigger.ts берёт задачу из argv[2] | WORKFLOW_CURRENT_TASK_PATH | doc/current_task.md и шлёт её в облачный Cursor Agent API с локальным cwd; это отдельный от llama.cpp и от kilo_relay путь. Пишет execution_contract.md, но requireSubstantiveContract: false, а границы держит платформенный auto-approval. Значит A — ручной/исследовательский вход (как Kilo IDE), не P0-executor: он не проходит packet_policy.ts и revert-before-review, поэтому его прогоны нельзя засчитывать в Execution VLCR.

B — kilo_relay ортогонален инверсии, а не чинит её. Релей — токен- компрессор + char-guard, не write-set/test gate. Два footgun'а из review 2026-07-21: C1 — strip-регэкспы заточены под Cursor-XML, на Kilo (Markdown-секции) strip_actions=[], экономия ≈ 0; C2 — дефолт SLIM_MODE=local держит allowlist Cursor-имён (Shell,Read,Write,Grep), а реальные tools Kilo (read_file,write_to_file,execute_command,…) под него не попадают → tools выкидывается, tool-calling ломается. Вывод для контура: write-set/test gate обязан жить в executor/Runner, не в релее — не путать с secret/transport-периметром, который релей всё же держит (ниже).

Обновление 2026-07-22 — DeepSeek preset в kilo_proxy_relay.py. Путь B научился маршрутизировать на облачный DeepSeek вместо локальной llama.cpp. Найден и закрыт критический баг: протухший KILO_RELAY_UPSTREAM_PRESET=deepseek вместе с явным raw-override мог отправить реальный DEEPSEEK_API_KEY на произвольный upstream — закреплено handler-level regression-тестом. Реализованы 4 подтверждённых DeepSeek-doc payload-фикса (developer-роль, tool_choice в thinking-режиме, max_completion_tokens, content:null на tool-call). Не закрыто: DeepSeek требует прокидывать reasoning_content между шагами разговора после tool call, иначе HTTP 400 — состояние между запросами, релей его не хранит; путь B через DeepSeek preset тем более не годится как P0-executor для agentic tool-calling, пока это не проверено end-to-end (усиливает уже сделанный вывод, не меняет его).

3Противоречия

Напряжения — и их синтез

Удобство IDE ↔ воспроизводимый headless-конвейер

Kilo незаменим для итеративной, исследовательской работы; headless-Runner незаменим для measurable auto-routing. Синтез: разные плоскости одного контракта — при конкретной конфигурации (релей с upstream=llama.cpp, не DeepSeek/ cloud_budget) Kilo и Runner могут использовать один alias/runtime и один формат packet/evidence contract (правка 2026-07-22: не гарантированно «один runtime» — зависит от того, куда указывает upstream релея). Kilo для человека, Runner для машины; артефакты неразличимы для reviewer, когда этот выбор конфигурации сделан.

Автономность ↔ жёсткая безопасность

Синтез: gates неизменны навсегда; растёт только routing-политика, и только по данным ledger (VLCR), а не по ощущению «сегодня прошло».

Большой контекст ↔ узкая понятная задача

64K — свойство сервера, не разрешение раздувать промпт. Синтез: context budget — обязательное поле пакета (жёсткий gate до вызова модели), 128K остаётся отдельным ручным профилем для редких long-repo задач.

Экономия cloud-токенов ↔ стоимость повторных исправлений

Синтез: облако платит за маленькие артефакты (компиляция пакета, ревью диффа), локальная модель — за большие (черновик + повторы). Полный cloud-executor включается по policy, а не по нетерпению разработчика.

Два репозитория ↔ single-root исполнитель

Синтез: Task Compiler режет двурепную задачу на два single-root пакета с явной dependencies-связью; полномочия исполнителя не расширяются ни на строку.

4Боль-якорь

Одна конкретная проверяемая боль

Проверяемый факт: живой прогон триггера с параметрами по умолчанию против реально запущенного production-сервера падает мгновенно — не из-за качества модели, а из-за строки конфигурации, которую никто давно не сверял с registry.

Точная причина в коде:

  • llamacpp_agent_trigger.ts:22 — DEFAULT_MODEL = "qwen/qwen3-coder-next", а production /v1/models отдаёт единственный alias qwen3-coder-next-q4ks;
  • llamacpp_agent_trigger.ts:318,748 — validatePatchAgainstWriteSet(diff, parsed.writeSetRaw) сверяет диф с декларацией ответа модели, а не с write-set задачи — извлечения task-write-set в файле нет вообще;
  • llamacpp_agent_trigger.ts:804 — extractTestCommands(parsed.testsRaw) берёт команды тестов из секции TESTS того же ответа.

Живая проверка: прогон с дефолтным alias против запущенного сервера падает за 54 мс с error_reason: model_alias_not_found (exit 2). После правки одной переменной окружения (LLAMACPP_MODEL=qwen3-coder-next-q4ks) тот же процесс на том же strict no-op задании проходит целиком за 2860 мс: применяет диф (в данном случае — «нет изменений»), пишет evidence-only execution_contract.md, помечает proof как substantive. Параллельно npm.cmd run test:trigger — 98/98 тестов зелёные. Контур жив и быстр; сломан не он, а то, кому он верит на входе и на выходе.

6Ценность

7Путь

Золотой путь пакета и лестница зрелости

Золотой путь одного пакета: готовый план → облачный Task Compiler режет его на Execution Packet (doc/packets/, studio) → детерминированный packet_policy.ts (studio) делает pre-flight (path-safety, денилист, risk-пересчёт) и выбирает local-direct → Runner спавнит hardened llamacpp_agent_trigger с явным LLAMACPP_TARGET_REPO_ROOT=target_repo (вместо переезда кода — один env поверх уже существующего параметра repoRoot) → repo-lock + baseline → patch/tests проверяются против задачи, не модели → journal prepared → applied → targeted tests → Runner откатывает успешный patch до reverted_pending_review → независимый cloud-reviewer → finalize_execution_packet.ts при APPROVE повторно применяет frozen patch → verified.
