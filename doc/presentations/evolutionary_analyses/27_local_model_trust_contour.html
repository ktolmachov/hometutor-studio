1Суть

Чего по-настоящему хочет разработчик, отдавая задачу модели

Не «локальная модель в IDE» — а дешёвый, контролируемый и доказуемый способ превращать часть плана в рабочий код. Ценность локальной модели не в бесплатности токенов вообще, а в дешёвых draft-токенах внутри контура, где границы держит не модель, а детерминированный код вокруг неё. Облачные токены дороги на массовую генерацию, но дёшевы на компиляцию задач и ревью диффов; локальные токены бесплатны, но по умолчанию недоверенны. Оптимум: облако думает и проверяет (мало токенов, высокая ставка), локальная модель печатает (много токенов, низкая ставка), а границы держит код, у которого ставка — ноль. Сегодняшняя находка: этот контур уже физически существует и работает за секунды — разбор фиксирует, где именно он сейчас доверяет не тому и не тем.

3Противоречия

Напряжения — и их синтез

Удобство IDE ↔ воспроизводимый headless-конвейер

Kilo незаменим для итеративной, исследовательской работы; headless-Runner незаменим для measurable auto-routing. Синтез: разные плоскости одного контракта — Kilo и Runner используют один production runtime, один alias и один формат packet/evidence contract. Kilo для человека, Runner для машины; артефакты неразличимы для reviewer.

Автономность ↔ жёсткая безопасность

Синтез: gates неизменны навсегда; растёт только routing-политика, и только по данным ledger (VLCR), а не по ощущению «сегодня прошло».

Большой контекст ↔ узкая понятная задача

64K — свойство сервера, не разрешение раздувать промпт. Синтез: context budget — обязательное поле пакета (жёсткий gate до вызова модели), 128K остаётся отдельным ручным профилем для редких long-repo задач.

Экономия cloud-токенов ↔ стоимость повторных исправлений

Синтез: облако платит за маленькие артефакты (компиляция пакета, ревью диффа), локальная модель — за большие (черновик + повторы). Полный cloud-executor включается по policy, а не по нетерпению разработчика.

Два репозитория ↔ single-root исполнитель

Синтез: Task Compiler режет двурепную задачу на два single-root пакета с явной dependencies-связью; полномочия исполнителя не расширяются ни на строку.

4Боль-якорь

Одна конкретная проверяемая боль

Проверяемый факт: живой прогон триггера с параметрами по умолчанию против реально запущенного production-сервера падает мгновенно — не из-за качества модели, а из-за строки конфигурации, которую никто давно не сверял с registry.

Точная причина в коде:

  • llamacpp_agent_trigger.ts:22 — DEFAULT_MODEL = "qwen/qwen3-coder-next", а production /v1/models отдаёт единственный alias qwen3-coder-next-q4ks;
  • llamacpp_agent_trigger.ts:318,748 — validatePatchAgainstWriteSet(diff, parsed.writeSetRaw) сверяет диф с декларацией ответа модели, а не с write-set задачи — извлечения task-write-set в файле нет вообще;
  • llamacpp_agent_trigger.ts:804 — extractTestCommands(parsed.testsRaw) берёт команды тестов из секции TESTS того же ответа.

Живая проверка: прогон с дефолтным alias против запущенного сервера падает за 54 мс с error_reason: model_alias_not_found (exit 2). После правки одной переменной окружения (LLAMACPP_MODEL=qwen3-coder-next-q4ks) тот же процесс на том же strict no-op задании проходит целиком за 2860 мс: применяет диф (в данном случае — «нет изменений»), пишет evidence-only execution_contract.md, помечает proof как substantive. Параллельно npm.cmd run test:trigger — 98/98 тестов зелёные. Контур жив и быстр; сломан не он, а то, кому он верит на входе и на выходе.

5Вердикты

Решение по каждому элементу карты

не делатьKilo как control plane. Auto-approval режет по инструментам, не по diff-границам; evidence остаётся самодекларацией агента даже в headless --auto-режиме.
не делатьВстраивание llama.cpp в trigger_orchestrator.ts сейчас. Метрики раздвоены — оркестратор читает last-row trigger_metrics.jsonl, а llamacpp-триггер пишет в собственный llamacpp_agent_trigger.jsonl; интеграция as-is сломает чтение error_reason молча, и promotion-данных для llama.cpp пока нет (0 authoritative прогонов в истории).
оставитьHardened trigger как ядро local-adapter'а. Контур уже живой, быстрый (2,9 с) и качественно спроектирован (write-set gate, hunk normalize, controlled apply, evidence-only contract) — чинить входы и выходы доверия, не переписывать заново.
починитьAlias и runtime defaults. Одна строка правки (DEFAULT_MODEL + guide/`.env.example` sync) — ROI огромный: контур мгновенно из «падает на старте» переходит в «работает».
починитьTrust inversion write-set/tests. Центральный фикс: task write-set/tests становятся authoritative, model-декларация — только предложением, которое обязано быть подмножеством.
не делатьПеренос контура в hometutor. Проверено дважды и отменено: hometutor/AGENTS.md:11-13 и CLAUDE.md:39-43 явно исключают workflow-тулинг из этого репозитория («backlog... ведутся в отдельном репозитории hometutor-studio»). Настоящий cwd-фикс дешевле — пять функций триггера (applyPatch, runTestCommands, gitChangedFiles, revertPatch, buildReadSetContext) уже принимают repoRoot параметром; жёстко закодирован только process.cwd() в пяти call sites. Решение: один env LLAMACPP_TARGET_REPO_ROOT, а не переезд 910-строчного модуля в другой язык экосистемы.
починитьKilo — оставить интерактивным рабочим местом и ручным escape hatch, не контуром принятия решений; production alias там уже верный, менять нечего.
оставитьworkflow.py — перевести в compatibility mode, не убирать. Backlog-driven работа продолжает его использовать; --trigger-cmd уже принимает произвольную команду, значит llamacpp-триггер можно дёрнуть из старого loop уже сегодня без единой правки.
не делатьПолную автономность / auto-routing до shadow-серии. Один удачный smoke — не основание; нужны ≥10 реальных пакетов с cloud-ревью каждого диффа, прежде чем local-direct станет маршрутом по умолчанию хоть для одного класса задач.
6Ценность

Verified Local Completion Rate — и честный ноль

Уникальность хода — не «ещё один локальный LLM-адаптер», а перенос доверия с исполнителя на контракт: local/cloud/Kilo/человек становятся взаимозаменяемыми деталями за одним и тем же Evidence Gate. Единственная метрика, которая доказывает это измеримо — доля пакетов, реально доведённых локальной моделью до independent-approved состояния.

7Путь

Золотой путь пакета и лестница зрелости

Золотой путь одного пакета: готовый план → облачный Task Compiler режет его на Execution Packet (doc/packets/, studio) → детерминированный packet_policy.ts (studio) делает pre-flight (path-safety, денилист, risk-пересчёт) и выбирает local-direct → Runner спавнит hardened llamacpp_agent_trigger с явным LLAMACPP_TARGET_REPO_ROOT=target_repo (вместо переезда кода — один env поверх уже существующего параметра repoRoot) → patch/tests проверяются против задачи, не модели → git apply + targeted tests → per-run артефакты (proposed.patch/applied.patch/receipt.json) → событие executed_pending_review → независимый cloud-reviewer APPROVE → терминальное verified в append-only ledger.jsonl.

  1. 0Disposable no-op smoke. Пройден сегодня живьём: 2,9 с, evidence-only contract, 98/98 unit — фундамент есть.
  2. 1Disposable real patch. Один файл, один failing test, authoritative write-set/tests — первый прогон после P0-1.
  3. 25 реальных low-risk пакетов подряд. Promotion-порог из integration report — ещё не начат.
  4. 3Shadow-серия ≥10 пакетов. Cloud-reviewer проверяет каждый дифф; auto-merge выключен — P1.
  5. 4Auto-routing «зелёных» классов. Только после promotion gates §8 — P2, не раньше.
