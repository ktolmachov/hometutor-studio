Часть 1 · Суть

Чего по-настоящему хочет студент от агента

Не «инструмента с tool-calling» и не «универсального ассистента». Студент хочет одного: нажать кнопку и получить готовую учебную сессию — диагностика, план на 10–20 минут, проверочные вопросы, карточки-кандидаты — вместо того, чтобы самому решать, что спросить, какой квиз запустить и что положить в колоду. Агент — это сжатие пяти ручных действий студента в одно решение системы.

Отсюда первый принцип: агент оценивается не по автономности, а по разнице усилий — сколько кликов и решений студента заменяет одна кнопка. Второй: система уже сама выбрала честный путь — «продуктизировать как study session coach, а не как универсальный агент» (буквальная цитата из докстринга роутера сценариев). Это верная суть — просто она пока живёт только в коде, не в интерфейсе.

Часть 2 · Реальность

Полный харнесс построен — и упирается в стену без двери

Инвентаризация показывает зрелую инженерию агентного цикла: контракты, стоп-контроллер, три продуктовых сценария, персистентность с санитайзингом. И ровно один пробел — в конце цепочки нет ничего, что подключало бы это к студенту.

Что уже образцово — не трогать

Stop controller в харнессе, не в модели — редкая дисциплина: агент не может зациклиться или сгенерировать неограниченную стоимость незамеченным. Recursion guard явно задокументирован: rag.answer обязан идти non-agent путём, не через себя же. Санитайзинг персистентности (_is_sensitive_key, _truncate_text) — данные прогонов пишутся безопасно с первого дня. Честная продуктизация уже выбрана в докстринге detect_study_session_intent: «agent mode is intentionally productized as study session coach, not exposed as a generic autonomous agent» — сути №4 не нужно изобретать, она уже написана словами в коде.

Часть 3 · Противоречия

Три напряжения — и синтез для каждого

Автономность ↔ управляемость

Агент может вызвать до 6 инструментов и потратить бюджет токенов/стоимости за один клик — то, чего линейный RAG-пайплайн никогда не делал. Это мощь, но и риск непредсказуемой стоимости на кнопку. Синтез уже реализован кодом: жёсткие бюджеты (agent_max_steps=6, agent_max_run_cost_usd, agent_max_run_seconds) и `agent_enabled=False` по умолчанию. Открывать кнопку — не значит менять эту защиту, значит подключить к уже готовому предохранителю.

Три сценария ↔ одна кнопка

Роутер выбирает сценарий по ключевым словам вопроса (_KONSPEKT_INTENT_MARKERS, _GRAPH_GAP_INTENT_MARKERS), а study_session — универсальный fallback («route every non-empty request»). Студенту не нужно знать про три сценария — ему нужна одна кнопка «Собери мне занятие». Синтез: кнопка не выбирает сценарий явно — она просто отправляет текущий контекст (тема/курс) в уже работающий роутер; разные сценарии — это variety ответа, не разные UI-элементы.

Черновик ↔ действие

Карточки-кандидаты (cards.propose) — «read/draft», никогда не сохраняются сами. Это правильно для Wave 1 (никакого автосохранения без HITL) — но черновик, который нельзя принять одним кликом, обесценивает работу агента. Синтез: не расширять write-доступ агента (Wave 5 — отдельный трек с idempotency и approval), а дать кнопке «Сохранить как карточку» рядом с черновиком — тот же жест, что уже спроектирован в разборе №1 (кандидат B1 «→ в карточку»), просто с агентным draft вместо ответа тьютора.

Часть 4 · Боль-якорь

Полностью работающий агент, недостижимый ни для одного студента

Точная причина в коде, не в теории

Три сценария реализованы, протестированы golden-кейсами, пишут персистентную историю прогонов. Ни один человек, использующий приложение через Streamlit UI, не может их вызвать:

# Единственный вход в агентную ветку (query_service.py:940):
if (options.query_mode or "").strip().lower() == "agent" and get_settings().agent_enabled:

# query_mode нигде в app/ui не устанавливается в "agent" —
# grep по всему app/ui даёт только query_mode="tutor" (tutor_chat_session.py:76)

# Фича-флаг подготовлен, но не подключён (feature_registry.py:154-158):
elif requirement == "agent_enabled":
    if not get_settings().agent_enabled: …
# — а ни один FeatureSpec в FEATURES не объявлен с requires=("agent_enabled",)

# Персистентность пишется, но список/чтение прогонов нигде не читается:
# list_agent_runs() / get_agent_run() — 0 caller'ов вне app/agent и тестов

Путь до ответа существует ровно один: прямой HTTP-запрос к /ask с {"query_mode": "agent"} в теле — это API, доступный только через прямой вызов, не через какой-либо клиент приложения. `AskRequest.query_mode` — свободная строка без enum (api_models_main.py:141), так что технически дыры в безопасности нет (`agent_enabled=False` по умолчанию гасит всё), но продуктовой ценности тоже нет: три месяца инженерии заканчиваются тупиком без двери.

Вторая створка: даже если бы кнопка появилась, у команды нет способа увидеть, как агент справляется в проде — GET /agent/runs/{run_id} из дорожной карты не реализован, роутера app/routers/agent.py не существует. Персистентность есть, обозреваемости нет — данные копятся, но никто их не смотрит.

Часть 6 · Ценность

Сжатие усилия — измеримое доказательство

Уникальность агента здесь не «ИИ вызывает инструменты» (это делает половина рынка), а сжатие пяти решений студента в одно: что спросить → какой квиз пройти → что добавить в колоду → как называется следующий шаг — за одну кнопку, без потери связи с источниками (тот же provenance-принцип, что в петле памяти разбора №1). Измеримо это через коэффициент замены действий: сколько отдельных экранов/кликов студент обычно делает вручную для того же результата, что даёт один прогон агента.

Часть 7 · Путь

Один золотой путь и лестница уровней

Студент на Mission Control видит одну кнопку с понятным обещанием («Собери мне занятие по этой теме») → агент за кулисами выбирает сценарий сам → черновик с диагностикой, планом, проверкой и карточками-кандидатами → студент жмёт «Сохранить карточки» там, где согласен → переходит в quiz/flashcards с уже готовым контекстом. Ни одного нового понятия для студента — три сценария и восемь инструментов остаются деталью реализации.

  • L0Кнопка появляется за флагом (agent_enabled=true для избранных) — первый черновик занятия
  • L1Черновик проверяется на живых студентах, golden-набор дорастает до 8–10 кейсов
  • L2Кнопка «Сохранить как карточку» поверх cards.propose — черновик становится действием
  • L3История прогонов видна студенту («что агент делал для меня») — доверие через прозрачность
  • L4Массовый rollout: кнопка на каждом экране, где уместна (Mission Control, Tutor chat, Living Konspekt)
Часть 8 · План

P0–P2 по соотношению «эффект / усилие»

P0 · ХОД 1

Одна CTA-кнопка за флагом

Mission Control получает CTA «Собрать учебную сессию» (текст уже предусмотрен дорожной картой), видимую при agent_enabled=true (сегодня — выключено глобально, включаемо для тестовой группы). Клик отправляет текущую тему/курс в /ask с query_mode="agent", результат рендерится как draft без записи в базы — контракт секций уже готов, рендерить можно как обычный markdown-ответ.

P0 · ХОД 2

Минимальный роутер чтения прогонов

app/routers/agent.py с GET /agent/runs / GET /agent/runs/{run_id} поверх уже существующих list_agent_runs/get_agent_run — только чтение, без HITL/resume. Данные, которые честно пишутся с первого дня, наконец становятся видимыми — сначала для отладки, затем как база для L3 «история прогонов».

P1

Golden-набор до 8–10 + кнопка «Сохранить как карточку»

Расширить agent_scenarios_golden_v1.json до целевого объёма (сейчас 4/8–10) — до расширения rollout за флагом. Параллельно: жест сохранения черновика карточки поверх cards.propose, использующий тот же add_flashcard(), что и кандидат B1 разбора №1.

P2

История прогонов на поверхности студента

Компактная лента «что агент делал для меня» (из list_agent_runs) на экране прогресса — не диагностика для разработчика, а учебный журнал: какие сессии агент собрал, что из них было сохранено.

Планка неизменна: гениальность — в простоте. P0 не добавляет ни одной новой строчки агентной логики — три сценария, стоп-контроллер и персистентность уже готовы. Ход №1 — это буквально одна кнопка, ход №2 — один тонкий read-роутер поверх уже написанных функций.
