## Вердикт экспертной панели

**Сильная сторона:** проект — это не «RAG-чат с красивым UI». Это замкнутый учебный контур с архитектурными границами, где каждый компонент выполняет конкретную педагогическую функцию. Это защищаемо.

**Главная уязвимость на защите:** есть retrieval-only eval (run_id=`eval-2026-05-20`, N=15 вопросов, recall@3 87 % hybrid), но **faithfulness (LLM-as-judge) не запущен**. Нельзя утверждать «система не галлюцинирует» с числами — только через retrieval-confidence. Честная позиция: «retrieval eval есть; faithfulness-run — следующий шаг после защиты».

**Что нужно признать явно на защите:**
- SM-2 — baseline, не оптимальный алгоритм для нового пользователя (cold start проблема)
- BM25 in-memory — узкое место при больших корпусах
- Качество академических PDF — зона риска (формулы, многоколоночная вёрстка)
- Retrieval eval есть (87 % hybrid recall@3, N=15); faithfulness eval — в плане
- Telegram whitelist chat_id — не реализован в текущей версии (в roadmap)

## Вопросы и ответы

### RAG Architect

**Вопрос 1:** Chunk size ~700 и overlap 50 — откуда эти числа? Почему они оптимальны для учебных материалов?

**Ответ:** Они не оптимальны — они рабочие. 700 токенов — эмпирический компромисс между информационной плотностью чанка и стоимостью embedding. Overlap 50 снижает риск разрезать ключевую мысль на границе. Для защиты честный ответ: параметры вынесены в `RetrievalSettings`, их нужно сравнивать через retrieval metrics (recall@k, chunk coherence) на репрезентативном corpus. Учебные материалы разнородны — PDF-лекции, статьи, заметки — поэтому один chunk size не оптимален для всех типов. Adaptive chunking по структуре документа (заголовки, параграфы) — следующий шаг.

---

**Вопрос 2:** Двухуровневый retrieval (document + chunk) усложняет систему. Когда он реально помогает, а когда только добавляет latency?

**Ответ:** Помогает при запросах типа «дай обзор темы» или «какие документы покрывают X» — document-level находит правильные источники, потом chunk-level извлекает детали. Добавляет latency без выигрыша при точечных вопросах: «что такое attention» — достаточно chunk-level. Поэтому это не всегда включено: `query_type=overview` триггерит двухуровневый режим, `query_type=qa` идёт прямо в chunk-level. Разделение по типу запроса — это ключевое место, где классификация на шаге 1 окупает себя.

---

**Вопрос 3:** Self-Correction Loop — retry при низком confidence. Что если система бесконечно пытается и тратит токены?

**Ответ:** Retry ограничен: максимум 1 попытка. Если после retry confidence всё ещё ниже порога — система возвращает ответ с явным дисклеймером «источников недостаточно, ответ может быть неточным». Бесконечный retry архитектурно запрещён — это один из проектных инвариантов (token budget и retry safety). Стоимость одного retry — это цена одного дополнительного LLM-вызова. При cloud LLM это осознанный trade-off: лучше потратить $0.002, чем выдать галлюцинацию.

---

**Вопрос 4:** Граф концептов — как строится, как обновляется, что делать при конфликтах?

**Ответ:** Граф строится при индексации: LLM извлекает концепты из каждого чанка и определяет prerequisite-связи. Хранится в SQLite как adjacency list. Обновляется при частичной переиндексации изменённых файлов — новые концепты добавляются, старые сохраняются (нет автоматического удаления, чтобы не ломать mastery history). Конфликты (один концепт назван по-разному в разных документах) — это зона риска: нет автоматического entity resolution. Для надёжности нужен controlled vocabulary или ручная валидация графа.

### ML Engineer

**Вопрос 1:** SM-2 — алгоритм 1987 года. Почему не FSRS-4.5, который Anki перешёл на него в 2023?

**Ответ:** FSRS-4.5 лучше SM-2, особенно в первые дни изучения и при нерегулярных повторениях. Но FSRS требует больше исторических данных для персонализации и сложнее для понимания и объяснения на защите. SM-2 выбран как pedagogy-validated baseline: его поведение предсказуемо, алгоритм читабельный, проект работает с нулевой историей пользователя. В roadmap: ML-модель кривой забывания поверх SM-2 данных — это шаг от SM-2 к данными-управляемому алгоритму. Для production-продукта — FSRS правильный выбор.

---

**Вопрос 2:** ML-модель кривой забывания в roadmap. Но откуда данные для обучения при новом пользователе (cold start)?

**Ответ:** Cold start — реальная проблема. Стратегия: первые N повторений (N~10-20 на концепт) — правило SM-2. После накопления данных — переключение на ML-предсказание. Synthetic data как bootstrap: SM-2 генерирует исторические данные для разных профилей (быстрый learner, медленный, нерегулярный), модель предобучается на них. При retraining на реальных данных пользователя synthetic data вытесняется. AUC-ROC < 0.75 на реальных данных — blocker, модель не активируется (порог из roadmap-контракта).

---

**Вопрос 3:** Mastery score — как вы убедитесь, что он не завышает уверенность пользователя (Dunning-Kruger эффект)?

**Ответ:** Хороший вопрос о педагогическом риске. Mastery score растёт только через успешные квизы и SM-2 оценки — не через количество вопросов или прочтений. Три уровня (recognition → recall → transfer) требуют разного типа доказательств: recognition — узнать из вариантов, recall — воспроизвести без подсказки, transfer — применить в новом контексте. Переход между уровнями требует стабильного прохождения — не одного успешного квиза. Graduation (статус «освоен») — только после 7+ дней стабильного transfer. Это снижает, но не устраняет риск: реальный transfer на новые задачи требует отдельного тестирования.

---

**Вопрос 4:** Reranking через cross-encoder — это дорого по latency. Измеряли ли вы p95 latency?

**Ответ:** Reranker (BAAI/bge-reranker-base) добавляет 200–400ms на CPU при top-10 кандидатах. P95 для полного pipeline с reranker — около 3–5 секунд на CPU с Ollama. Без reranker — 1–2 секунды. Reranker настраивается через `.env` и отключается при необходимости. Для production-latency требований: можно переключиться на lighter cross-encoder или убрать reranker и использовать только RRF. Измеренные цифры — на конкретном тестовом железе, не universal SLO.

### Educational AI Researcher

**Вопрос 1:** Сократический метод в тьюторе — это серьёзное утверждение. Как конкретно вы его реализовали?

**Ответ:** Реализован через структурированный промпт с pedagogical scaffolding. Тьютор: не даёт готовый ответ, задаёт уточняющий вопрос; отвечает на ответ студента подтверждением/коррекцией + следующим вопросом; ведёт от конкретного к абстрактному; заканчивает проверкой понимания. Это не настоящий сократический метод — это аппроксимация через промпт-инжиниринг. Настоящий Сократ адаптировался к каждому ответу с глубоким пониманием — LLM делает это хуже. Честная формулировка: «scaffolded Q&A диалог в стиле сократического метода».

---

**Вопрос 2:** Zone of Proximal Development (ZPD) — учитываете ли вы её при генерации квиза?

**Ответ:** Частично. Квиз генерируется из контекста последнего RAG-ответа — это материал, с которым студент только что взаимодействовал, что близко к ZPD. Но адаптации сложности к текущему уровню студента нет: новичок и эксперт получат одинаковый квиз по одному документу. Adaptive difficulty на основе mastery score — это следующий уровень после ML-маршрутизатора. Сейчас: генерация квиза + контекст = приближение к ZPD, не точное попадание.

---

**Вопрос 3:** Есть ли риск, что система создаёт иллюзию обучения — пользователь отвечает на квизы, но реального transfer learning нет?

**Ответ:** Да, это реальный педагогический риск. Система снижает его через три механизма: три уровня mastery (recognition не равен recall не равен transfer), разные типы вопросов в квизе (узнать → объяснить → применить), и spaced repetition (проверка через дни, не минуты). Но «illusion of knowing» полностью не устраняется — это известная проблема всех SRS-систем. Реальный transfer требует применения знаний вне системы, что инструмент не контролирует. Честная позиция: снижаем риск, не устраняем.

---

**Вопрос 4:** Когнитивная нагрузка — не перегружает ли пользователя ваш «полный цикл» из 8 шагов?

**Ответ:** Это правомерное опасение. Защита: в любой момент пользователь видит один рекомендованный следующий шаг (умный маршрутизатор), а не все 8 сразу. Это принцип progressive disclosure: система не показывает граф концептов, SM-2 schedule и mastery одновременно. Дашборд — отдельный экран, не постоянный overlay. «Учить тему 5 минут» — это конкретный call-to-action, не меню из 8 опций. Риск когнитивной перегрузки реален для новых пользователей — онбординг должен показывать по одному экрану за раз.

### AI Safety Engineer

**Вопрос 1:** RAG-системы уязвимы к prompt injection через документы. Пользователь загрузил документ с «Ignore previous instructions». Что происходит?

**Ответ:** Это документ-уровневая атака. Текущая защита: guardrails.py проверяет пользовательский ввод, но не содержимое загружаемых документов при индексации. Если злонамеренный текст попадёт в чанк и будет извлечён как контекст — LLM его увидит. Дополнительная защита: промпты сформулированы с явным system instruction «отвечай только по источникам», что снижает вероятность следования injection. Но полной защиты нет. Следующий шаг: content-level sanitization при индексации и adversarial eval corpus для тестирования. Это честное ограничение текущей версии.

---

**Вопрос 2:** Data exfiltration — может ли LLM «слить» содержимое чужих документов через API?

**Ответ:** В single-user local deployment нет «чужих» документов — все документы принадлежат одному пользователю. В гипотетическом multi-user сценарии: изоляция пользовательских индексов критична. Сейчас ChromaDB и SQLite не имеют row-level security — это known limitation для single-user MVP. При переходе к multi-user: нужны отдельные ChromaDB коллекции per user и user_id isolation в SQLite. Текущая архитектура предполагает один пользователь — одна машина.

---

**Вопрос 3:** Выходные данные LLM могут содержать вредоносный контент (XSS в Streamlit, небезопасные ссылки). Как защищаетесь?

**Ответ:** Streamlit по умолчанию escapes HTML в `st.write` и `st.markdown` — прямой XSS через markdown маловероятен. Для кликабельных ссылок на источники: URL формируется из проверенных метаданных документа (имя файла + путь), а не из LLM-сгенерированного текста. Output guardrails проверяют формат ответа, но не делают full content audit на безопасность. Для production с публичным доступом нужен content security policy и output filtering слой.

### MLOps Lead

**Вопрос 1:** Что происходит, если Ollama упал во время сессии пользователя?

**Ответ:** FastAPI endpoint вернёт 503 или 500 с сообщением об ошибке. В Streamlit UI — friendly error message «Сервис временно недоступен». Retry на уровне LLM-вызова не реализован (чтобы не скрывать проблему). Пользовательский state не теряется — прогресс, карточки, mastery в SQLite сохранены. При восстановлении Ollama — сессия продолжается. Для production: healthcheck endpoint, graceful degradation (показывать только карточки/дашборд без LLM-функций) — это следующий уровень надёжности.

---

**Вопрос 2:** Как мониторить quality drift — если модель начала отвечать хуже?

**Ответ:** Сейчас мониторинга quality drift нет в production-смысле. Есть: confidence score как прокси retrieval quality, логирование запросов/ответов в SQLite для post-hoc анализа, regression test suite на eval-cases. Для drift detection нужен: baseline eval dataset с ground truth, периодический автоматический прогон, алерт при деградации метрик. Это следующий уровень MLOps — eval-pipeline как scheduled job.

---

**Вопрос 3:** Partial reindex — как вы определяете «изменился ли файл»?

**Ответ:** По хешу файла (MD5/SHA) и timestamp последней модификации. При запуске реиндексации сравниваются хеши текущих файлов с сохранёнными в метаданных индекса. Файлы с изменённым хешем переиндексируются, остальные — нет. Удалённые файлы из корпуса — отдельный сценарий: их чанки остаются в ChromaDB до явной очистки (garbage collection индекса). Для production нужен TTL на удалённые источники.

### LLM Evaluation Lead

**Вопрос 1:** Faithfulness — как проверяете, что ответ не выходит за рамки найденных источников?

**Ответ:** На уровне промпта: системная инструкция явно требует «отвечай только по предоставленным источникам, не используй внешние знания». На уровне output: guardrails проверяют наличие ссылок на источники в ответе. На уровне eval: для корректного измерения faithfulness нужен eval-dataset с парами (контекст, вопрос, эталонный ответ) и автоматическая проверка через LLM-as-Judge или NLI-модель. Этот eval-pipeline есть в архитектуре (`eval/` директория), но не наполнен eval-набором. Без run-id — нет числа.

---

**Вопрос 2:** Как измерить Context Recall — что система не упустила важный фрагмент?

**Ответ:** Context Recall = процент ключевых утверждений из ответа, которые можно найти в retrieved контексте. Для измерения: нужен labeled eval-set с вопросами, ground-truth ответами и ground-truth relevant chunks. LLM-as-Judge сравнивает retrieved chunks с expected chunks. Без этого eval-набора — нельзя утверждать конкретное число. Методика понятна, реализация — следующий шаг после сбора eval-данных.

---

**Вопрос 3:** Answer Relevancy — как убедиться, что система отвечает на вопрос, а не уходит в сторону?

**Ответ:** Через LLM-as-Judge: задаём вопрос «насколько этот ответ релевантен вопросу» с scoring 1-5. Более строгий вариант: обратный тест — из ответа генерируем вопрос, сравниваем embedding-близость с оригинальным вопросом. Высокая similarity = высокая relevancy. Это стандартная RAGAS-методика. Реализация: в eval-сервисе есть заглушки под эти метрики, заполнение eval-датасетом — первоочерёдная задача после защиты.

### Product Reviewer

**Вопрос 1:** Onboarding — новый пользователь откроет приложение и увидит пустой экран без документов. Что происходит?

**Ответ:** Есть два состояния onboarding. Первое — пустой индекс: приложение показывает quickstart-подсказку «Загрузите документы → нажмите Индексировать → задайте первый вопрос». На экране администрирования — явный CTA «Загрузить первый документ». Второе — демо-режим: можно запустить с demo data (несколько публичных документов) для демонстрации без своих материалов. Это снижает, но не устраняет «холодный старт» UX-проблему.

---

**Вопрос 2:** Как убедить пользователя, который уже использует ChatGPT, что стоит потратить 15 минут на установку?

**Ответ:** Конкретное value proposition, которое ChatGPT не даёт: ваши данные остаются на вашей машине (приватность), ответы привязаны к вашим документам (verifiable), карточки создаются автоматически из ответов (нет ручного труда), система помнит ваш прогресс через месяцы (не сбрасывается). Для пользователя, которому достаточно ChatGPT — инструмент не нужен. Для пользователя с конфиденциальными материалами, систематическим обучением и желанием не забывать — 15 минут окупятся.

---

**Вопрос 3:** Killer feature — одна функция, которая удерживает пользователя. Что это у вас?

**Ответ:** Автоматические карточки из ответа. Пользователь задаёт вопрос, получает ответ, нажимает «создать квиз» — через 2 минуты у него 3 карточки по теме в SRS. Раньше это занимало 30 минут вручную. Это единственный момент «вау», который меняет привычку: не нужно отдельно открывать Anki и думать, что туда записать. Остальные функции важны, но эта — крюк, который делает инструмент незаменимым.

### Systems Architect

**Вопрос 1:** SQLite не поддерживает concurrent writes. Что будет, если два запроса придут одновременно?

**Ответ:** SQLite использует file-level locking. При concurrent writes — один ждёт, другой выполняется. Для single-user приложения (один человек, один браузер) это не проблема. При Telegram-боте + UI одновременно — возможна очередь на writes, задержка несколько миллисекунд. При multi-user сценарии — SQLite не подходит, нужен PostgreSQL с connection pooling. Это честное ограничение, которое принято осознанно для local-first single-user архитектуры.

---

**Вопрос 2:** ChromaDB PersistentClient — что происходит при повреждении индекса?

**Ответ:** ChromaDB хранит данные в `chroma_db/` директории. При повреждении — нужна переиндексация (исходные документы не теряются, только векторный индекс). Время переиндексации: зависит от размера корпуса и скорости embedding-модели — от минут до часов. Для защиты индекса: в Docker-деплое `chroma_db/` — отдельный volume с возможностью backup. Автоматический backup или replication нет — это single-machine local storage.

---

**Вопрос 3:** Если проект вырастет до 1000 пользователей — что нужно изменить в архитектуре?

**Ответ:** Три категории изменений. Хранилище: SQLite → PostgreSQL (row-level security, connection pooling), ChromaDB → managed vector store с tenancy (Qdrant Cloud, Pinecone), файлы → object storage (S3). Auth: session cookie → OAuth2/JWT, user isolation в индексе. Инфраструктура: stateless FastAPI instances + load balancer, Celery для async задач (индексация в background), Redis для caching embeddings и сессий. Provider.py и бизнес-логика — не меняются. Streamlit → React/Next.js для масштабируемого frontend с SSR.
