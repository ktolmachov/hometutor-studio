> Цель: подготовить ответы на всё, что может спросить комиссия по теме «разработка полнофункционального веб-приложения с использованием AI-агентов».

## Вопросы руководителя — Соответствие требованиям

**1. У вас Streamlit, а требования говорят React / Vue / Next.js. Почему это допустимо?**

**Ответ:** Streamlit — современный Python-фреймворк для веб-приложений, который генерирует полноценный HTML/CSS/JS frontend. Он поддерживает responsive layout, интерактивные компоненты, формы, state management и реальный HTTP. Для data/AI-приложений это стандартный production-инструмент (используется в Hugging Face Spaces, корпоративных ML-платформах). Выбор обоснован: избегаем дублирования логики между Python-backend и JS-frontend, что критично при быстром прототипировании учебного инструмента. При необходимости замены — API-слой (FastAPI) остаётся без изменений.

---

**2. Требования говорят PostgreSQL. У вас SQLite. Это техническое упрощение или осознанное решение?**

**Ответ:** Осознанное архитектурное решение, вытекающее из принципа local-first. SQLite — серверный движок нулевой конфигурации, он хранит базу одним файлом рядом с приложением. Это принципиально важно для offline-сценария и для пользователя, который устанавливает инструмент на свою машину без инфраструктуры. Схема спроектирована с теми же нормализованными связями: 6 таблиц с foreign keys. Миграция на PostgreSQL через SQLAlchemy — это замена строки подключения, бизнес-логика не меняется. Для multi-user cloud-деплоя PostgreSQL был бы правильным выбором.

---

**3. Покажите три связанные таблицы и объясните связи.**

**Ответ:** Три центральных таблицы:

- `flashcards` — карточки (id, concept_id FK, front, back, created_at)
- `concepts` — концепты из граф-индексации (id, name, source_doc, mastery_score, level)
- `reviews` — история повторений (id, flashcard_id FK, ease_factor, interval_days, next_review, rating)

Связь: одна карточка принадлежит одному концепту (`flashcards.concept_id → concepts.id`). У одной карточки много записей в истории повторений (`reviews.flashcard_id → flashcards.id`). Mastery score концепта агрегируется из результатов reviews через SM-2 алгоритм. Это позволяет строить запросы вида «все карточки по слабым концептам» или «история прогресса по теме за 30 дней».

---

**4. Где аутентификация? Требования её явно требуют.**

**Ответ:** Реализована на двух уровнях. Для Streamlit UI — персистентный `session_id`, который хранится в таблице `sessions` (session_store.py) и связывает диалог с историей; это не user-auth в смысле логин/пароль, это single-user state isolation для local-first приложения. Для защищённых REST endpoints — API-key header: клиент передаёт `X-API-Key`, а ключ задаётся в `.env` как `HOME_RAG_API_KEY` (app/api_auth.py); `/health` и bootstrap — публичные. Для Telegram-бота — `chat_id` используется как `session_id`; whitelist разрешённых chat_id — в roadmap. Полноценный OAuth2 с регистрацией/логином — следующий уровень для multi-user сценария.

**5. Как реализована валидация данных? Покажите конкретно.**

**Ответ:** На двух уровнях. Первый — Pydantic v2 модели для всех API запросов и ответов: `QueryRequest` проверяет минимальную длину вопроса (3 символа), тип запроса (enum), диапазон параметров. При нарушении — автоматический 422 Unprocessable Entity с описанием. Второй — `app/guardrails.py`: доменная валидация поверх Pydantic — проверка на prompt injection паттерны, максимальный размер загружаемого файла, санитизация имён файлов, ограничение частоты запросов. Все точки входа (API, UI, Telegram) проходят через эти проверки.

**6. Минимум 2 дополнительные функции из списка — что выбрали и почему?**

**Ответ:** Три функции. Первая — интеграция с внешними API: LLM-провайдеры (Ollama, OpenRouter, OpenAI) через единый `provider.py`. Это не просто HTTP-вызов — это абстракция, позволяющая переключать провайдеров через `.env` без изменения бизнес-логики. Вторая — файловое хранилище: загрузка PDF, DOCX, MD, TXT через UI и API endpoint `/admin/upload`, с последующей индексацией в ChromaDB. Третья — поиск и фильтрация: гибридный RAG-поиск (векторный + ключевые слова) с фильтрацией по курсу, источнику и типу запроса.

## Вопросы руководителя — Техническая глубина

**9. Что такое RAG и почему он лучше простого «дать PDF в ChatGPT»?**

**Ответ:** RAG — Retrieval-Augmented Generation. Вместо того чтобы положить весь PDF в контекст (ограничен 128k токенов, дорого), система сначала находит релевантные фрагменты через векторный поиск, затем передаёт только их в LLM. Три преимущества: масштабируемость (работает на сотнях документов, не только на одном), воспроизводимость (ответ привязан к конкретным источникам), экономия (оплачиваем только нужные фрагменты). Плюс: ответ верифицируем — пользователь видит, из каких фрагментов он собран.

---

**10. Почему пять ступеней в pipeline, а не одна — просто поиск + генерация?**

**Ответ:** Каждая ступень решает конкретную проблему. Классификация — разные типы вопросов требуют разных стратегий (точечный ответ vs обзорный vs синтез). Переформулировка — нечёткий вопрос даёт плохой поиск, enrichment улучшает recall. Гибридный поиск — векторный ловит смысл, ключевой ловит точные термины. Переранжирование — top-k по cosine similarity не равно top-k по релевантности; cross-encoder точнее. Self-correction — если контекст слабый, лучше честно сказать «данных мало», чем галлюцинировать. Каждая ступень измеримо улучшает результат — это не украшение.

---

**11. Как вы защищаете от галлюцинаций?**

**Ответ:** Многослойная защита. Первый слой: retrieval с источниками — LLM получает только найденные фрагменты, промпт явно запрещает выходить за их пределы. Второй слой: self-correction — если confidence retrieval ниже порога, система делает retry с другим запросом; если снова слабо — возвращает «источников недостаточно». Третий слой: output guardrails — проверка формата ответа, наличия ссылок на источники. Четвёртый слой: прозрачность — пользователь видит источники и может проверить.

---

**12. Что такое SM-2 и почему выбран он, а не более современные алгоритмы?**

**Ответ:** SM-2 — алгоритм интервального повторения 1987 года, основа Anki. Оптимизирует расписание повторений на основе оценки сложности карточки. Выбран как проверенная baseline с десятилетиями валидации на реальных пользователях. Более современные алгоритмы (FSRS-4.5, SM-18) требуют больше данных для персонализации. Для MVP SM-2 даёт 80% ценности при 20% сложности. В дорожной карте — ML-модель кривой забывания поверх SM-2 данных.

---

**13. Что такое mastery tracking и как он связан с RAG?**

**Ответ:** Mastery tracking — система отслеживания освоения концептов по трём уровням: recognition (узнаю), recall (воспроизвожу), transfer (применяю). Связь с RAG двойная. Входящая: при вопросе система смотрит, какие концепты слабые, и приоритизирует их в поиске. Исходящая: после квиза и SM-2 оценки mastery score концепта обновляется, что влияет на следующие рекомендации умного маршрутизатора. Цикл замкнут: RAG-ответ → mastery score → следующий RAG-запрос.

---

**14. Что является вашим личным инженерным вкладом, а не просто интеграцией библиотек?**

**Ответ:** Интеграция всего цикла — это инженерная задача, которую ни одна библиотека не решает. Конкретно: проектирование typed QueryContext pipeline (каждый шаг принимает и возвращает контекст без побочных эффектов), связка RAG-ответа с автоматическим созданием квиза и карточки (нет разрыва контекста), алгоритм умного маршрутизатора с 7 приоритизированными состояниями, partial reindex логика (только изменённые файлы), архитектурный инвариант provider.py (единая точка LLM без утечек в бизнес-логику), автоматизированный workflow с DoD-защитой (14 итераций).

## Вопросы студентов-наблюдателей

**1. Как отличается ваш проект от NotebookLM Google?**

**Ответ:** NotebookLM — отличный инструмент для Q&A по документам. Но: облачный (данные у Google), нет SRS/карточек, нет mastery tracking, нет tutor-режима, нет offline. Мой инструмент добавляет полный учебный цикл: ответ → тьютор → квиз → карточки → повторения → mastery. И работает полностью локально. Это не конкурент, это другое: NotebookLM для поиска по документам, мой инструмент — для обучения по ним.

---

**2. Что делать, если у меня нет GPU — система работает?**

**Ответ:** Да. Ollama работает на CPU, просто медленнее: 7B-модель на CPU генерирует ~5–15 токенов/сек. Для повседневного использования это приемлемо — ответ приходит за 30–60 секунд. Embeddings на CPU — норм для работы, медленнее при индексации. Если скорость критична — cloud LLM через API-ключ, там серверные GPU.

---

**3. Как бы вы проверили, что система реально помогает учиться, а не создаёт иллюзию знания?**

**Ответ:** Через измеримые метрики. Retention rate: процент карточек, вспомненных правильно через 7 и 30 дней. Quiz accuracy: динамика результатов квизов по одной теме. Mastery progression: время от recognition до transfer. Recall without context: тест без подсказок системы. Сейчас инструмент собирает эти данные — дашборд их отображает. Лонгитюдное исследование с контрольной группой — это следующий научный шаг в дорожной карте.

---

**4. Что будет, если LLM-провайдер изменит API или станет платным?**

**Ответ:** Этот риск закрыт архитектурно. Все LLM-вызовы инкапсулированы в `app/provider.py`. Смена провайдера — это изменение `.env` (URL, ключ, имя модели) и, возможно, нескольких строк в provider-адаптере. Бизнес-логика не трогается. Ollama всегда доступен как free fallback с локальными моделями.

---

**5. Почему в дорожной карте 15 недель на AI-маршрутизатор? Это не слишком долго?**

**Ответ:** Это честная оценка при параллельной разработке. Последовательно было бы 22 недели. 5 уровней: ML-модель кривой забывания, LLM-объяснения, недельный планировщик, граф-маршрутизатор, feedback loop. Каждый требует: data collection, обучение модели, eval-контракт с метриками, интеграционные тесты, UI. Можно срезать углы, но тогда нет измеримых метрик и нет честных DoD. 15 недель — это профессиональная оценка с запасом.

---

**6. Вы говорите local-first, но при cloud LLM данные уходят в интернет. Это не противоречие?**

**Ответ:** Нет, если быть точным в определениях. Local-first означает: все пользовательские данные (прогресс, карточки, mastery, история) хранятся локально. Индекс — локально. State — локально. Inference (генерация ответа) может быть local (Ollama) или cloud. При cloud inference фрагменты документа передаются провайдеру — это известный trade-off, который явно описан в UI и документации. Для конфиденциальных данных — только Ollama. Это не противоречие, это два режима с разными гарантиями приватности.

---

**7. Почему 200+ тестов, но проект всё равно может выдавать неверные ответы?**

**Ответ:** Потому что тесты покрывают разные слои. Unit-тесты проверяют: SM-2 правильно считает интервалы, API возвращает 422 на невалидный запрос, guardrails блокируют инъекции. Они не тестируют качество ответа LLM — это недетерминированная система. Для качества ответов нужен отдельный eval-набор с ground truth: вопрос → ожидаемый ответ → метрики faithfulness и relevancy. Это следующий уровень quality assurance, который запланирован.

---

**8. Что самое сложное было в разработке?**

**Ответ:** Три вещи. Первая — поддержание архитектурных границ при росте функциональности: каждый раз, когда хотелось «быстро» добавить LLM-вызов прямо в сервис, нужно было остановиться и пойти через provider.py. Вторая — связка учебного цикла без разрыва контекста: RAG-ответ должен передавать контекст в тьютор, тьютор — в квиз, квиз — в карточку, не теряя исходный вопрос. Третья — честная оценка качества: отказаться от красивых чисел без воспроизводимого eval-run.

## Острые вопросы комиссии

**«Это не веб-приложение, это скрипт с интерфейсом»**

**Ответ:** Нет. Это многоуровневое приложение: FastAPI backend (порт 8000) с 90+ REST endpoints в 16 router-файлах, Streamlit frontend (порт 8501) с 5 функциональными экранами, SQLite база данных с 10+ таблицами, ChromaDB векторное хранилище (in-process PersistentClient), Telegram-бот как третий клиент, 1824 pytest теста. Всё взаимодействует через HTTP, контейнеризировано в Docker. По всем критериям — полнофункциональное веб-приложение.

---

**«AI-агенты использованы только для написания кода. Где они в самом продукте?»**

**Ответ:** AI-агенты — это ядро продукта. LLM-агент отвечает на вопросы по документам (RAG). LLM-агент проводит сократический диалог в тьюторе. LLM-агент генерирует квизы из контекста. ML-модель (baseline SM-2, следующий шаг — sklearn/XGBoost) управляет расписанием повторений. LLM-агент генерирует объяснения «почему следующий шаг — именно это». Умный маршрутизатор — детерминированный агент с состоянием. В разработке тоже использованы AI-агенты — Cursor, Claude — как показано на слайде 8. Это двойное использование: AI в продукте и AI в процессе разработки.

---

**«Где измеримые результаты? Что улучшилось на сколько процентов?»**

**Ответ:** Измерено сегодня: retrieval recall@3 на demo_data — 87 % для hybrid режима, 80 % для vector-only (run_id=`eval-2026-05-20`, N=15 вопросов). Это retrieval-quality, не faithfulness. Что ещё измеримо прямо сейчас: время создания карточки (автоматически ~30 сек vs вручную ~30 мин), confidence score как прокси качества retrieval. Для утверждений о том, что «система не галлюцинирует» нужен faithfulness eval с LLM-as-judge — методика есть, запуск запланирован. Лонгитюдное retention-исследование с контрольной группой — это следующий научный шаг.
