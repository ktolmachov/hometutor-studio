# Гайд: Эволюционный разбор («сильный ход») — переиспользуемый формат

> **Что это.** Промпт-шаблон и метод для стратегического продуктового разбора:
> не список правок, а разрешение противоречий («простота ↔ глубина», «мощь ↔ фокус»)
> через инвентаризацию реального кода и одну проверяемую боль. На выходе — тезис о сути,
> карта всего реализованного с вердиктами и implementation-ready план волн.
>
> **С чего начинать.** Скопируй шаблон промпта (раздел 2), заполни плейсхолдеры под
> свою область, прогони отдельным запросом — не смешивай с обычной правкой кода в той
> же сессии.
>
> **Пример-эталон.** hometutor essence-разбор 2026-07-10: полный анализ Mission
> Control/UX/петли обучения → [`../next/learning_loop_simplicity_plan.md`](../next/learning_loop_simplicity_plan.md)
> (implementation-ready продолжение с candidate-пакетами). Формат подачи самого
> разбора — самодостаточный HTML-лонгрид, см. [`html_deck_guide.md`](html_deck_guide.md)
> для принципа «без внешних зависимостей» (здесь — не слайды, а notebook-style страница).
>
> **Версия гайда:** 1.0 (2026-07-10).

---

## 0. TL;DR — разбор за 5 шагов

1. **Заполни 8 плейсхолдеров** шаблона (раздел 2) под свою область — ничего не пропускай,
   особенно пункты 1 (СУТЬ) и 4 (БОЛЬ-ЯКОРЬ).
2. **Прогони отдельным запросом.** Это исследовательская задача, не патч кода.
3. **Требуй вердиктов, не обзоров** — см. запрет «просто перечислить варианты» (раздел 3).
4. **Оформи результат в два слоя:** самодостаточный HTML для чтения/презентации +
   implementation-ready `.md`-план в `doc/next/` для дальнейшей реализации волнами.
5. **Проверь по чек-листу** (раздел 5) прежде чем считать разбор готовым.

---

## 1. Когда это работает, а когда — нет

**Работает для:**
- продуктовый/UX редизайн запутанного накопившегося интерфейса (много хороших фич без фокуса);
- приоритизация roadmap, когда кандидатов много, а критерия выбора нет;
- поиск product differentiation («в чём наша уникальность и как её доказать»);
- разбор «что мы упускаем» в зрелой системе с историей.

**Не работает для:**
- точечного баг-фикса — там нужен фикс, а не тезис о сути;
- вопроса с единственным правильным ответом (тогда «противоречия» и «вердикты» —
  лишний театр);
- случая, когда нет реального кода/данных для инвентаризации — пункт 2 шаблона
  («РЕАЛЬНОСТЬ») тогда выродится в фантазии без опоры.

## 2. Шаблон промпта (копировать целиком, заполнить `<плейсхолдеры>`)

```markdown
Нужен сильный ход.

1. СУТЬ. Чего по-настоящему хочет <человек/роль>, когда <делает X>?
   В чём суть <X>? (первые принципы, не фичи)

2. РЕАЛЬНОСТЬ. Посмотри с высоты на всё, что уже реализовано <в коде / данных / процессе>.
   Собери, классифицируй, сгруппируй. Для каждого элемента: зачем он нужен, плюсы,
   минусы, и как связать с остальным, чтобы вместе давало новый эффект.

3. ПРОТИВОРЕЧИЯ. Разреши напряжения (не выбирай сторону — найди синтез):
   - <простота> ↔ <глубина>
   - <мощь> ↔ <фокус>
   - <специфичное для этой области напряжение>

4. БОЛЬ-ЯКОРЬ. Вот конкретная проверяемая боль: <факт из жизни/использования>.
   Найди её точную причину в реальности (код/данные), а не в теории.

5. ВЕРДИКТЫ. Для каждого элемента карты — решение: оставить / спрятать глубже /
   объединить / починить / убрать. Не обзор опций — позиция с обоснованием.

6. ЦЕННОСТЬ. В чём уникальность и как её доказать измеримо (не декларативно).

7. ПУТЬ. Один золотой путь пользователя + лестница уровней, которые легко достигать.

8. ПЛАН. Приоритеты P0–P2 по соотношению «эффект / усилие». P0 — не больше двух ходов.

Планка: шедевр. Гениальность — в простоте. Ничего не выдумывать: каждое утверждение
опирается на реальное и проверяемое.
```

## 3. Почему это работает — 7 ингредиентов

| # | Ингредиент | Зачем | Как применять |
|---|---|---|---|
| 1 | Вопросы о сути от первых принципов | Уводит от списка фич к тезису, из которого всё остальное выводится | Формулируй пункт 1 как «чего хочет человек», не «что должен делать продукт» |
| 2 | Инвентаризация реального (не абстракции) | Заземляет разбор — каждое утверждение проверяемо | Читай реальный код/данные, а не полагайся на память/документацию «как должно быть» |
| 3 | Противоречия для синтеза | Настоящие решения разрешают напряжение, а не выбирают из списка | Формулируй именно как пару полюсов (`X ↔ Y`), не как открытый вопрос |
| 4 | Одна конкретная боль-якорь | Даёт разбору измеримую точку опоры и проверяет его на прочность | Ищи факт, который можно подтвердить одним grep/командой за минуту |
| 5 | Вердикты, а не перечисление | Запрещает отделаться списком вариантов | Каждый элемент карты получает явное решение: оставить/спрятать/объединить/починить/убрать |
| 6 | Планка «шедевр» | Критерий отбраковки: что не проходит — не попадает в ответ | Держи её явно в тексте промпта, не подразумевай |
| 7 | Свобода решений при жёсткости вопросов | Промпт с готовым решением получает исполнение; промпт с вопросами — мышление | Никогда не предписывай ответ в самом шаблоне |

Пункты **1** и **4** обязательны всегда — без пункта 1 разбор превращается в аудит
фич, без пункта 4 — в философию без проверки на реальности.

## 4. Как оформлять результат

Разбор всегда даёт **два слоя**, не один:

1. **Читаемый слой** — самодостаточный HTML-лонгрид (нет внешних зависимостей, инлайн-SVG
   для диаграмм, тема light/dark). Это не слайд-дек — принцип из
   [`html_deck_guide.md`](html_deck_guide.md) («факты из источника → токены → компоненты»)
   применяется к одностраничному документу с секциями, а не к раскадровке 16:9.
2. **Actionable-слой** — implementation-ready `.md`-план в `doc/next/` со сквозной
   структурой на каждый пункт: `Problem / Evidence (файл:строка) / Proposed / Files /
   DoD / North star / Kill switch / Effort / Priority / Dependencies`. Этого достаточно,
   чтобы владелец мог напрямую промоутить пункты в `backlog_registry.yaml` волнами/пакетами
   без повторного анализа.

**Правило границы:** разбор **никогда** не редактирует `backlog_registry.yaml` напрямую —
только предлагает кандидатов. Образец такого разделения — существующий
[`roadmap_recommendations_2026-06-11.md`](../next/roadmap_recommendations_2026-06-11.md)
(рекомендации с ID, статусом и «Привязка к SSoT», сам registry редактируется отдельно
через lint-инструментарий).

## 5. Чек-лист перед сдачей разбора

- [ ] Пункт 1 (СУТЬ) сформулирован как один тезис, а не список наблюдений
- [ ] Пункт 2 (РЕАЛЬНОСТЬ) проверен по реальному коду/данным, не по памяти или предположению
- [ ] Есть минимум одна БОЛЬ-ЯКОРЬ, подтверждённая конкретным `файл:строка`
- [ ] Каждый элемент карты получил вердикт — не голое перечисление возможностей
- [ ] П0 не длиннее двух ходов
- [ ] Результат оформлен в двух слоях: HTML-разбор + (если нужна реализация) detail-plan `.md`
- [ ] Ничего не отредактировано в `backlog_registry.yaml` напрямую — только предложено

## 6. Частые ошибки

| Симптом | Причина | Фикс |
|---|---|---|
| Разбор читается как красивое эссе без опоры на реальность | Пропущен пункт 2 (инвентаризация) или сделан по памяти | Перечитать реальный код/скриншоты перед выводами, процитировать `файл:строка` |
| Рекомендации звучат как «можно A, можно B, можно C» | Пропущен пункт 5 (вердикты) | Для каждого элемента — явное решение и одна причина |
| План невозможно передать в реализацию | Actionable-слой не написан отдельно от читаемого HTML | Добавить detail-plan `.md` с полями Problem/Evidence/Files/DoD |
| P0 разросся на 5+ пунктов | Не применена планка «P0 ≤ 2 хода» | Оставить только те два хода, что меняют ощущение продукта сильнее всего остального списка |
| Боль-якорь осталась недоказанной | Не проверена в коде, взята со слов | Найти конкретный `файл:строка`, который её объясняет, прежде чем фиксировать вывод |
