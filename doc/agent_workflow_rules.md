## Token Budget & Retry Safety (v1)

Применяется ко **всем** LLM-вызовам агентов в этом проекте — planning, execution, verify, architecture review.

### 1. Жёсткий бюджет входных токенов

| Зона | Лимит | Действие |
|---|---|---|
| Целевой | ≤ 12k токенов | Без ограничений |
| Soft-limit | 12k–20k | Обязательное сжатие history и read-set до отправки |
| Hard-limit | > 20k | Вызов **запрещён** — остановка и отчёт о блокере |

Перед каждым LLM-вызовом агент оценивает размер входного контекста. Превышение hard-limit = немедленная остановка.

**Примечание:** старый бюджет (20k/30k) заменён на 12k/20k. Подробнее: [token_safety.md](token_safety.md).

**Синхронизация оценок размеров:** после крупных изменений в «тяжёлых» файлах выполнить `python scripts/measure_token_registry.py --write` и при необходимости подправить числа в таблицах [token_safety.md](token_safety.md) (раздел «Поддержка документации после крупных правок»).

### 2. Правило Read-set (Критический Лимит)

- **Читать ТОЛЬКО 3–5 ключевых файлов, явно перечисленных в промте.**
- Запрещено читать дополнительные файлы без явного указания в контракте.
- Запрещено читать крупные документы целиком, если задача требует только секцию.
- Для roadmap/doc-файлов использовать только релевантный фрагмент, а не весь файл.
- Избыточные файлы (tasklist.md, rules.mdc, полные closed_iterations.md): исключены, используйте only конкретные epoch-файлы.
- `Owner files` / `Write-set` задают ответственность за изменения, но **не являются read-set**. Читать полностью можно только файлы, явно перечисленные в `Read ONLY`; для остальных сначала signatures/grep/секция.

**Эффект:** снижение входных токенов на 40–50% (18k → 10k) без потери качества.

### 3. Ограничение history (Исключение Накопления)

- **Скопи из conversation истории только последние 2–3 шага**, всё остальное выкидывается.
- Запрещено переносить в каждый вызов полный лог предыдущих tool-действий.
- При росте контекста заменять старую историю коротким summary (1–2 предложения).
- **Scope: "Ignore prior responses/tools. Fresh context only."** — добавлять явно в начало каждого нового LLM-вызова.

**Эффект:** -15–20% входа в цепочках вызовов (дельты истории → 0).

### 4. Retry policy при ERR (с Fallback для Больших Контекстов)

- Запрещён повторный вызов с тем же payload после ERR.
- Ретрай допускается **только** после одного из:
  - сокращение history (удалить старую историю),
  - сокращение read-set (убрать лишние файлы),
  - summary вместо сырых вставок (заменить большие тексты на краткие описания).
- **Fallback при >20k входных токенов:** если контекст превышает hard-limit, остановить вызов и отчитаться о блокере (не делать ретрай).
- Максимум **1 безопасный ретрай** после сжатия; далее — остановка и отчёт о блокере.

**Эффект:** предотвратить 3+ ретрая, -10% задержки/риска.

### 5. Model routing по стоимости (Guard против glm-5.1)

**Таблица стоимости (рубли/1M входных токенов на 2026-04-19):**

| Модель | Цена/1M | Цена типичный вызов (18k) | % от grok | Использование |
|---|---|---|---|---|
| grok-4.1-fast-thinking | ~85 руб | ~1.5 | 100% | ✅ DEFAULT для всех |
| claude-sonnet-4.6 | ~120 руб | ~2.2 | 150% | ⚠️ только на запрос owner |
| openai/gpt-5.3-codex | ~300 руб | ~5.4 | 360% | 🚫 ЗАПРЕЩЕНА (см. row #15-17 в логе) |
| z-ai/glm-5.1 | ~400 руб | ~7.1 | 475% | 🚫 ЗАПРЕЩЕНА (см. row #14 в логе) |

**Правила:**
- **Default:** `grok-4.1-fast-thinking` для planning, execute, verify.
- **Запрещены:** glm-5.1, openai/gpt-5.3-codex (доказано дорого без выигрыша).
- **Исключение:** Только для критических arch-review Phase 1 (conventions audit) — если контракт явно одобрил дорогую модель И остальные фазы выполнены на grok.
- **Guard-механизм:** перед вызовом проверить `model in BLOCKED_MODELS` → остановить и отчитаться блокером (не делать fallback).

**Эффект:** row #14 (glm-5.1, 7.12 руб) → row #1 (grok, 1.89 руб) = **73% экономия на один вызов**; row #15-17 (140k потраченных токенов) → 0.

### 6. Отчётность

В финале каждого пакета агент кратко фиксирует:
- примерный размер входного контекста,
- применённые меры сжатия,
- факт и причину ретрая (если был).
- **модель и стоимость** вызова (в форме: `model | input_tokens | output_tokens | cost_rub`).

### 7. Cost Tracking & Alerts

**Цель:** предотвратить дорогие ошибки (row #14-17 из логе = 146.5 руб потрачено).

| Сценарий | Триггер | Действие |
|---|---|---|
| Вызов glm-5.1 или openai/gpt-5.3 | model ∈ BLOCKED_MODELS | **STOP** — не отправлять, отчитаться блокером |
| ERR с теми же параметрами (row #15-17) | вызов → FAIL → повторный вызов с тем же payload | **STOP** — запрещён ретрай без сжатия, fallback |
| Input > 20k токенов | вызов pending | **STOP** — hard-limit нарушен, отчитаться о блокере |
| Cost за пакет > 10 руб (unusual) | cost monitoring | 🔔 **LOG** с пометкой `EXPENSIVE PACKAGE` + контекст (модель, tokens) |
| Кумулятивная стоимость эпохи > 100 руб | tracking per-epoch | 📊 **Report в doc/cost_tracking.md** (еженедельный дайджест) |

**Примеры из логе (2026-04-19):**
- ✅ Нормально (1–2 руб): rows #1-13 (grok with typical 18-20k input)
- 🔴 Anomaly (7.1 руб): row #14 (glm-5.1, должна была заблокирована)
- 🔴 Error-loop (0 полезного output × 3): rows #15-17 (повторный вызов БЕЗ сжатия, потрачено 140k)

**Действие:** добавить guard в LLM-caller, логировать cost per call, еженедельный отчёт.

> **Arch review:** полный review (Phase 1–5) превышает 20k — запускать **отдельными фазами** (один вызов = одна фаза). См. [`agent_workflow_arch_review.md`](agent_workflow_arch_review.md).
