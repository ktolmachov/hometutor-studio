# Запуск полного умного конспекта лекции

Прочитай и выполни приложенный файл
`smart_lecture_konspekt_universal_cloud_llm.md`.

Он является главным обязательным контрактом. Эта инструкция уточняет режим
исполнения, выбор источника и критерии завершения. При конфликте применяй
более строгое требование полноты, достоверности и отсутствия выдумок.

## Режим исполнения: только полный максимальный вариант

Сразу создавай финальную полную версию уровня учебной главы. Пользователь уже
запросил максимальный формат. Не спрашивай, нужна ли полная версия.

Запрещено:

- сначала выдавать сокращённую версию;
- создавать «расширенный каркас», outline или документ с placeholders;
- объявлять работу готовой, если обязательные разделы ещё не заполнены;
- предлагать «собрать максимальную версию следующим сообщением»;
- завершать ответ предложением продолжить, расширить или перегенерировать;
- заменять содержательные разделы обещанием добавить их позже;
- создавать промежуточный DOCX и называть его полным конспектом;
- несколько раз просить заново приложить уже найденные в текущей сессии
  файлы только потому, что поисковая выдача показала фрагмент.

Целевой объём полного конспекта: ориентировочно 8 000–15 000 слов, если
материал лекции это поддерживает. Не раздувай текст повторами ради объёма.

## Обязательные входные файлы

Найди среди приложенных файлов комплект одной лекции.

Обязательное ядро:

1. `smart_lecture_konspekt_universal_cloud_llm.md`;
2. один главный транскрипт лекции в формате `.txt` или `.ts.md`.

Дополнительные источники, если приложены:

3. презентация этой же лекции в формате `.pdf` или `.pptx`;
4. вспомогательный HTML-конспект этой же лекции в формате `.html`;
5. другие явно обозначенные пользователем материалы этой же лекции.

Определи принадлежность файлов к одной лекции по совокупности признаков:

- номер урока или занятия;
- название и тема лекции;
- общий базовый фрагмент имени файла;
- заголовки и metadata внутри файлов;
- явное указание пользователя.

Не связывай файлы только по совпадению расширения. Сначала установи единую
идентичность лекции, затем выбери относящийся к ней комплект.

Приоритет источников:

1. главный транскрипт — источник смысла, объяснений, примеров и вопросов;
2. презентация — источник структуры, формулировок и визуальных материалов;
3. HTML — только вспомогательная карта тем, оформления и возможных акцентов;
4. внешние источники — только для проверки и явно маркированных дополнений.

Не используй HTML как замену транскрипту и не копируй из него готовый
конспект. Если HTML противоречит транскрипту, приоритет у транскрипта. Если
презентация противоречит транскрипту, основное изложение строй по
транскрипту, а расхождение отметь в проверке точности.

Если приложены несколько транскриптов, презентаций или HTML-файлов разных
лекций, выбери согласованный комплект по номеру/теме. При реальной
неоднозначности перечисли только кандидатов и остановись до создания
документа. Не объединяй разные уроки и не выбирай случайно.

Если для одной лекции приложены несколько версий транскрипта, выбирай главный
источник в таком порядке:

1. файл, который пользователь явно назвал основным;
2. `.ts.md` с корректными маркерами `# Чистый текст` и `# С таймкодами`;
3. полный `.txt`-транскрипт;
4. другой файл, только если пользователь явно обозначил его транскриптом.

Остальные версии не используй как независимые источники и не увеличивай из-за
повторов значимость отдельных тем.

Если презентация или HTML отсутствуют, это не блокирует генерацию по полному
транскрипту. Не создавай и не оценивай неприменимые презентационные разделы
как будто презентация была доступна; честно укажи ограничение в рубрике.

Не запрашивай дополнительные эталонные конспекты, HTML-эталоны или другие
необязательные материалы: правила стиля уже содержатся в universal prompt.
Отсутствие необязательного источника не является блокером, если главный
транскрипт доступен. HTML используй только как дополнительный источник.

## Полное чтение файлов

Поисковый snippet или первый фрагмент файла не считается полным чтением.

Если интерфейс возвращает большие файлы частями:

1. Продолжай чтение самостоятельно без дополнительной команды пользователя.
2. Используй поиск по заголовкам, переход к следующим диапазонам и повторное
   открытие с другой позиции.
3. Читай последовательными частями без пропусков; небольшой overlap допустим
   для проверки непрерывности.
4. Веди внутренний журнал:
   `файл → прочитанные диапазоны/разделы → следующий диапазон → конец`.
5. Не начинай конспект по первому фрагменту и не проси разрешить продолжение.

Universal prompt прочитай до раздела `# Финальный ответ` и физического конца
файла.

Файл считается технически недоступным только если после нескольких попыток
открытия/поиска интерфейс действительно не предоставляет следующий диапазон.
Фраза «поиск показал только часть» сама по себе не доказывает недоступность.

## Главный источник транскрипта

Если главный транскрипт имеет формат `.txt`, используй файл от первой до
последней строки и читай его последовательными частями до физического конца.

Если главный транскрипт имеет формат `.ts.md` и содержит оба заголовка
`# Чистый текст` и `# С таймкодами`, используй только диапазон между ними:

- начало: строка после `# Чистый текст`;
- конец: строка перед `# С таймкодами`;
- сами граничные заголовки не включай.

Всё до `# Чистый текст` и секцию `# С таймкодами` полностью игнорируй.

Если `.ts.md` не содержит этих двух маркеров, не угадывай нужный диапазон:
используй другой явно обозначенный транскрипт или сообщи конкретный блокер.

Не считай первый поисковый фрагмент полным транскриптом.

Не используй другие версии транскрипта, таймкодовые копии, VTT/SRT или ранее
сгенерированные summaries как дополнительное подтверждение фактов.

Сохраняй определения, живые объяснения, вопросы, примеры, оговорки и
причинно-следственные связи. Удаляй приветствия, технические паузы, повторы и
разговорный шум без учебной ценности. Не исправляй сомнительное распознавание
по догадке.

## Scope gate: не подменяй лекцию внешним учебником

Основной конспект должен отражать именно приложенную лекцию.

Запрещено без прямого запроса пользователя:

- добавлять отдельную главу о проекте пользователя, компании или персональном
  стеке, если это прямо не запрошено;
- приписывать конкретную внутреннюю архитектуру современным продуктам и
  сервисам без надёжного актуального источника;
- объявлять неподтверждённые классификации вроде `система X = ReAct`;
- расширять шкалы, определения и классификации так, будто они были на слайде;
- переносить внешние сведения в основное изложение без маркировки.

Современные продукты можно упомянуть только в разделе внешнего эксперта или
дополнительных материалов, если утверждение проверено по актуальному
официальному источнику и явно обозначено как внешнее дополнение.

## Презентация и Visual QA

Если презентация приложена, просмотри её полностью до генерации результата.
Проводи аудит партиями по 10–20 слайдов и веди внутреннюю карту покрытия.

Обязательно включи:

- карту содержательных слайдов и диапазонов;
- восстановленные таблицы, формулы, чек-листы и decision criteria;
- номера слайдов у схем и таблиц;
- раздел `## 🖼 Визуальная выжимка: слайды, которые нужно помнить`;
- 8–12 ключевых визуальных опор для длинной презентации;
- реальные изображения, если доступны asset-файлы, иначе подробные visual
  brief: композиция, элементы, стрелки, подписи и главный вывод;
- честный список нечитаемых или непросмотренных диапазонов.

Не заявляй покрытие слайдов, которые фактически не видел. Битый OCR не
восстанавливай по догадке.

## Обязательное внешнее веб-исследование

Веб-исследование входит в текущую задачу и уже разрешено пользователем.
Отдельная команда или согласие не требуются.

Если доступны web/search/browser-инструменты:

1. Выдели 4–6 центральных тем лекции.
2. Найди официальную документацию, стандарты, первоисточники, инженерные
   статьи авторов технологий и репозитории проектов.
3. Открой каждый выбранный URL: snippet не считается проверкой.
4. Проверь заголовок, автора/организацию, содержание и актуальность.
5. Добавь только прямые канонические URL конкретных материалов.
6. Подготовь 5–8 сильных материалов и 1–3 проверенных видео, если подходящие
   видео найдены. В раздел `### Видео` клади только реальные видео
   (YouTube, вебинар, видеокурс с плеером); страницы проектов и лендинги —
   в статьи/документацию. Если видео не нашлось, явно напиши причину.

Для каждого материала укажи язык, автора/организацию, что изучить и какую
часть лекции он углубляет. Добавь 2–4 экспертных дополнения и явно отдели их
от слов лектора.

Не отвечай, что «отдельное веб-исследование не запрашивалось».

Только если web-инструменты технически отсутствуют, разрешено не добавлять
ссылки. Тогда назови именно техническую причину, не подставляй URL по памяти
и поставь `н/д` или не выше 2/5 за дополнительные материалы.

## Minimum Definition of Done

Финальный документ не считается готовым, пока не выполнены применимые
требования universal prompt и минимум ниже:

- полное оглавление и карта лекции;
- главная мысль и 5–9 полноценных ключевых тем;
- покрытие презентации и визуальная выжимка, если презентация приложена;
- самостоятельный Agentic Loop с success/failure/stop/no-progress, если тема
  присутствует в лекции;
- Mermaid-схемы процессов и архитектур, если они полезны;
- сравнительные таблицы с критериями и trade-offs;
- примеры лектора с контекстом и практическим выводом;
- объяснения внешнего эксперта, отделённые от содержания лекции;
- ошибки, риски, антипаттерны и проверка точности;
- практические выводы и терминология;
- контрольные вопросы и практическое задание;
- итоги, мини-шпаргалка и каркас знаний;
- 15–25 flashcards;
- 12–20 quiz-вопросов с ответами и объяснением `почему`;
- spaced repetition plan;
- чек-лист внедрения;
- мини-проект с критериями готовности;
- честная рубрика качества — строго по таблице критериев universal prompt:
  те же названия критериев, тот же порядок, включая `Использование
  презентации` и `Сохранение графической информации`; собственные наборы
  критериев и переименования запрещены;
- проверенные дополнительные материалы при доступном web-поиске.

Не создавай пустой раздел только для галочки. Если специальный раздел
неприменим к содержанию лекции, кратко объясни это или корректно исключи его.

## Измеримые пороги полноты

Для длинного используемого диапазона транскрипта размером более 50 000
символов итоговый Markdown
должен пройти следующие нижние границы:

- не менее 6 000 содержательных слов; целевой диапазон 8 000–15 000;
- не менее 15 заголовков второго/третьего уровня с наполненным содержанием;
- не менее 4 содержательных Markdown-таблиц;
- не менее 3 Mermaid-схем, если лекция содержит архитектуры, циклы или
  процессы;
- 8–12 визуальных опор в Visual QA, если приложена длинная презентация;
- 15–25 полноценных flashcards;
- 12–20 quiz-вопросов с ответами и объяснениями;
- минимум одно практическое задание и один мини-проект;
- заполненная рубрика качества по всем критериям.

Порог считается выполненным по реальному содержанию, а не по числу
заголовков. Заголовок с одной общей фразой, placeholder или обещанием
дописать позже не засчитывается.

Если материал объективно не поддерживает один из структурных порогов,
объясни исключение в рубрике качества. Ограничение не разрешает снижать общий
объём до краткого summary или каркаса.

## Completion gate перед сохранением

Перед созданием или сохранением итогового файла проведи внутреннюю проверку:

- universal prompt прочитан полностью;
- главный `.txt` прочитан от первой строки до физического конца или главный
  `.ts.md` прочитан от `# Чистый текст` до `# С таймкодами`;
- HTML, если приложен, использован только как вспомогательная карта, а не как
  главный источник;
- презентация, если приложена, просмотрена целиком либо пробелы честно
  перечислены;
- все применимые пункты Minimum Definition of Done заполнены содержанием;
- измеримые пороги полноты посчитаны и выполнены;
- нет placeholders, TODO, обещаний доделать позже и сокращённых секций;
- нет выдуманных цитат, URL, номеров слайдов и изображений;
- рубрика не завышена и использует ровно критерии universal prompt, без
  самодельных критериев и без текстовой «итоговой самооценки» после таблицы;
- итоговый документ является полной версией, а не каркасом.

Если проверка не пройдена, продолжай работу автономно до её прохождения. Не
публикуй промежуточный документ как готовый.

Если полный доступ к обязательному источнику технически невозможен, остановись
**до создания любых итоговых файлов** и один раз сообщи конкретный блокер:
роль и имя файла, какой диапазон/маркер недоступен и какие попытки чтения
выполнены.
Не создавай в этом случае сокращённую замену.

## Формат результата

Создай один полный итоговый Markdown-документ.

Сформируй имя динамически по metadata текущей лекции:

`Урок_<номер>_<краткая_тема>_умный_конспект_cloud_llm.md`

Если номер урока неизвестен:

`<краткая_тема>_умный_конспект_cloud_llm.md`

Для имени файла:

- бери номер и тему только из приложенных материалов;
- не подставляй номер другого урока;
- заменяй пробелы на `_`;
- убирай символы, недопустимые в имени файла;
- не используй общие имена вроде `Full_Konspekt.md`, если тема известна.

Верни или сохрани только Markdown. Не создавай DOCX, PDF, HTML или другие
производные файлы в этом запуске. Не вызывай инструменты генерации документов
и не заменяй Markdown ссылкой только на DOCX.

Если пользователь позже отдельно запросит DOCX, он должен быть создан из уже
готового полного Markdown без сокращения и пройти отдельную проверку.

Не добавляй после результата предложения сделать более полную версию: текущий
результат уже обязан быть максимальной полной версией.

Не выводи перед результатом планы, оценки будущего документа, списки того,
что «можно было бы добавить», или сообщения `Готово`, если за ними следует
признание, что документ сокращён. Первый публикуемый артефакт должен быть
финальным полным Markdown.
