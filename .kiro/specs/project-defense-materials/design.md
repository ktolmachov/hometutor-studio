# Design Document — project-defense-materials

## Overview

Фича создаёт два самодостаточных Markdown-документа для академической защиты проектной работы по проекту **home-rag_v2**:

1. **`doc/defense_presenter_script.md`** — структурированный скрипт выступления на 3 минуты с таймкодами, ключевыми фразами и таблицей страховочных ответов.
2. **`doc/defense_presentation.md`** — презентация в Markdown со структурой слайдов (8+ слайдов), охватывающая все аспекты проекта: продукт, архитектуру, демо, RAG-пайплайн, сравнение LLM-режимов, процесс разработки и идеи развития.

Оба документа опираются на реальные артефакты репозитория: скриншоты из `doc/screenshots/final/`, документацию из `doc/`, архитектурные решения из `doc/architecture.md` и `doc/adr.md`.

Ключевое отличие от существующего `doc/presenter_script.md` (конкурсный питч): академическая защита требует акцента на постановке задачи, обосновании архитектурных решений, описании процесса разработки и измеримых результатах — а не на «hook» и «killer feature» для жюри конкурса.

---

## Architecture

Фича не изменяет код приложения. Это задача генерации документов — создание двух Markdown-файлов в директории `doc/`.

```
.kiro/specs/project-defense-materials/
├── requirements.md          ← одобрено
├── design.md                ← этот документ
└── tasks.md                 ← будет создан

doc/
├── defense_presenter_script.md   ← СОЗДАТЬ (скрипт выступления)
├── defense_presentation.md       ← СОЗДАТЬ (презентация)
├── presenter_script.md           ← существующий конкурсный скрипт (только для сравнения стиля)
├── screenshots/final/            ← реальные скриншоты (ссылки в презентации)
│   ├── scenario_01/              ← быстрый ответ с источниками
│   ├── scenario_03/              ← tutor
│   ├── scenario_04/              ← quiz
│   ├── scenario_07/              ← прогресс и слабые места
│   ├── scenario_09/              ← персональный план
│   └── scenario_13/              ← course workspace
└── team_workflow/
    └── workflow_router.md        ← источник для слайда «Идеи развития»
```

Зависимости при создании документов:
- Скрипт выступления не зависит от презентации и может быть создан независимо.
- Презентация ссылается на реальные файлы скриншотов — пути должны быть проверены перед записью.
- Оба документа ссылаются на существующие `doc/*.md` — все ссылки должны указывать на реально существующие файлы.

## Components and Interfaces

### Компонент 1: Defense_Script (`doc/defense_presenter_script.md`)

**Структура документа:**

```
# Скрипт выступления на академической защите — home-rag_v2

> Метаданные: формат, аудитория, инструкция

## 0:00–0:30 · Постановка задачи
## 0:30–1:15 · Архитектурное решение
## 1:15–2:15 · Демонстрация ключевых функций
## 2:15–2:45 · Процесс разработки
## 2:45–3:00 · Итоги и дальнейшее развитие

## Страховочные ответы на вопросы комиссии
(таблица: вопрос | ответ | артефакт)

## Технические страховки
```

**Интерфейс:** Markdown-файл, читается напрямую. Нет API, нет зависимостей от кода приложения.

**Ключевые фразы** (🎯 **Ключевая фраза**) — по одной на каждый раздел, выделены жирным.

**Таблица страховочных ответов** — минимум 5 строк, каждая со ссылкой на реальный артефакт репозитория.

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system — essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

Для данной фичи большинство acceptance criteria — это структурные и контентные проверки статических документов (EXAMPLE/SMOKE). Единственная универсальная property, применимая ко всем ссылкам в обоих документах — **referential integrity**: каждая ссылка на файл должна указывать на реально существующий путь в репозитории.

Это свойство применимо к property-based testing: мы можем извлечь все ссылки из документов и для каждой проверить существование файла — поведение одинаково для любой ссылки, и 100 итераций (по одной на каждую ссылку) действительно находят больше ошибок, чем 2–3 примера.

### Property 1: Referential Integrity

*For any* file or document reference contained in `doc/defense_presenter_script.md` or `doc/defense_presentation.md`, the referenced path SHALL exist as a real file in the repository.

**Validates: Requirements 1.10, 4.2, 9.2**

## Error Handling

Поскольку фича создаёт статические документы, а не код с runtime-поведением, «ошибки» — это нарушения структурных инвариантов при создании документов.

| Ситуация | Обработка |
|----------|-----------|
| Ссылка на несуществующий скриншот | Заменить на существующий файл из `doc/screenshots/final/` или убрать ссылку |
| Ссылка на несуществующий `doc/*.md` | Проверить реальное имя файла через `listDirectory` перед записью |
| Таймкоды не покрывают 3 минуты | Скорректировать границы разделов до суммы 0:00–3:00 |
| Менее 5 строк в таблице страховочных ответов | Добавить строки до минимума |
| Менее 8 слайдов в презентации | Добавить недостающие слайды |
| Схема не в блоке кода | Обернуть в ` ```mermaid ``` ` или ` ```text ``` ` |

**Принцип:** перед записью любой ссылки на файл — проверить существование через `listDirectory` или `fileSearch`. Не ссылаться на файлы «по памяти».

## Testing Strategy

### Оценка применимости PBT

Фича создаёт статические Markdown-документы. Большинство acceptance criteria — это проверки наличия контента, структуры и форматирования конкретного документа. Это **не** подходит для property-based testing в классическом смысле (нет функции с варьируемыми входными данными).

Исключение — **referential integrity** (Property 1): можно параметризовать по всем ссылкам в документах и проверить каждую. Это единственная property, где 100 итераций (по числу ссылок) добавляют ценность.

### Подход к тестированию

**Структурные проверки (EXAMPLE-тесты):**

Для каждого документа — набор проверок после создания:

```python
# defense_presenter_script.md
assert Path("doc/defense_presenter_script.md").exists()
assert "0:00" in content and "3:00" in content  # таймкоды
assert content.count("🎯 **Ключевая фраза") >= 5  # по одной на раздел
assert "## 0:00" in content  # раздел «Постановка задачи»
assert "## 0:30" in content  # раздел «Архитектурное решение»
assert "## 1:15" in content  # раздел «Демонстрация»
assert "## 2:15" in content  # раздел «Процесс разработки»
assert "## 2:45" in content  # раздел «Итоги»
# таблица страховочных ответов: минимум 5 строк (|...|...|...|)
table_rows = [l for l in content.splitlines() if l.startswith("|") and "---" not in l]
assert len(table_rows) >= 6  # заголовок + 5 строк

# defense_presentation.md
assert Path("doc/defense_presentation.md").exists()
slide_headings = [l for l in content.splitlines() if l.startswith("## Слайд")]
assert len(slide_headings) >= 8
assert "## Оглавление" in content
assert "Академическая защита" in content  # метаданные
```

**Property-тест (referential integrity):**

```python
# Feature: project-defense-materials, Property 1: Referential Integrity
import re
from pathlib import Path
import pytest

@pytest.mark.parametrize("doc_path", [
    "doc/defense_presenter_script.md",
    "doc/defense_presentation.md",
])
def test_referential_integrity(doc_path):
    """
    For any file reference in the defense documents,
    the referenced path must exist in the repository.
    Feature: project-defense-materials, Property 1: Referential Integrity
    """
    content = Path(doc_path).read_text(encoding="utf-8")
    # Извлечь все ссылки вида [text](path) где path не начинается с http
    refs = re.findall(r'\[.*?\]\(([^)]+)\)', content)
    local_refs = [r for r in refs if not r.startswith("http")]
    for ref in local_refs:
        # Нормализовать путь относительно корня репозитория
        resolved = Path(ref)
        assert resolved.exists(), f"Broken reference in {doc_path}: {ref}"
```

Этот тест параметризован по всем локальным ссылкам в документах — каждая ссылка проверяется независимо. Количество итераций = количество ссылок в документах (ожидается 15–30).

**Smoke-проверки (ручные):**

- Язык документов — русский (визуальная проверка).
- Defense_Script отличается от `doc/presenter_script.md` по структуре (академические акценты vs конкурсный питч).
- Схемы корректно рендерятся в Markdown-просмотрщике.

### Инструменты

- **pytest** — для структурных и property-тестов (уже используется в проекте).
- Нет новых зависимостей — только стандартная библиотека Python (`re`, `pathlib`).
- Тесты размещаются в `tests/test_defense_materials.py`.
